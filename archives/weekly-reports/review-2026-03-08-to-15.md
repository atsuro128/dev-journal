## 3. 意思決定の質

**良かった点**:
- ADR による意思決定記録が定着しつつある（001〜004）
- Rust → Go の判断は早期・低コスト・明確な根拠
- 「人間の注意力に頼らず仕組みで防ぐ」方針は一貫している（Issue #022 のセッションログ時刻検証など）
- レビュー指摘の Issue 昇格判断（#024→#021）は適切

**気になった点**:
- ファイアウォール方式の変更が場当たり的（iptables → Squid → SNI → revert を2サイクル）。事前調査が不足していた可能性
- Auto Memory 導入→廃止の流れが早い（Dev Container の揮発問題を事前に想定できたはず）
- プロセス改善の issue を一度に13件（ops-023〜035）起票したが、これ自体がプロダクト開発のブロッカーにならないか注意が必要

## 4. 気になるパターン

### パターン A: プロセス整備がプロダクト開発を押しのけている

8日間で expense-saas にプロダクトコードが1行も追加されていない。整備された基盤は将来の生産性に寄与するが、「1ヶ月目標」との整合が取れているか要確認。

- 3/8: リポジトリ分離
- 3/9: Hook 整備
- 3/10: Step 2 + Skills 移行
- 3/11: Dev Container + Rust→Go
- 3/12: Git 衛生 + 公開準備
- 3/13: ファイアウォール + レビュー改善
- 3/14: Egress Proxy + Step 2 完了
- 3/15: Work Breakdown + ops issue 起票

プロダクト設計に使えた日は実質 3/10 と 3/14 の一部のみ。

### パターン B: ツーリングの深追い

Hook の修正5回、ファイアウォールの方式変更3回、Squid proxy の試行と revert 2回。1つの問題に深入りして時間を消費する傾向がある。「80%で先に進む」判断ができるとプロダクト開発に時間が回る。

### パターン C: 「仕組み化」の再帰

ミスを防ぐために hook を作り、hook の問題を防ぐためにカウンターを作り、カウンターの問題を防ぐために temp ディレクトリに移動する——という再帰的な仕組み化が見られる。根本解決に見えるが、プロダクト開発の文脈では「そもそもそこまでの仕組みが今必要か」を問うべき場面もある。
