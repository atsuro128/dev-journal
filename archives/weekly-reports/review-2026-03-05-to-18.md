### パターン1: 「基盤先行型」の時間配分
設計 Step に着手しても、実際の設計作業より環境・プロセス整備に多くの時間を使う傾向がある。

- **示唆**: Step 4+5 では「設計作業に集中する日」を意識的に作らないと、同じパターンが繰り返される可能性

### パターン2: 「判断の振れ幅」
新しい仕組みを導入 → 問題発見 → 大幅変更、というサイクルが短期間で複数回発生。
- 例: branching（3回）、egress proxy（3回）、Auto Memory（導入→廃止）
- **示唆**: 導入前に小さく検証（PoC）するステップを挟むと振れ幅を抑えられるかもしれない

### パターン3: 「セッション密度のばらつき」
1日のセッション数が1〜7と大きく変動。セッション数が多い日（03-09: 4, 03-10: 5, 03-11: 5, 03-13: 7）は細かい修正の繰り返しが多い。
- **示唆**: セッション数が多い日は「試行錯誤中」のシグナル。振り返り時の注意ポイント

### パターン4: 「日報の繰り越しタスク」
「Step 2 着手」が03-05〜03-09の4日間、「Step 3 着手」が03-11〜03-15の5日間にわたり繰り越されていた。
- **示唆**: 繰り越しが続く場合は、ブロッカーの明示化や「今日はプロセス改善日」と宣言する運用があると実態と乖離しない

## 6. セッションログの分析可能性と改善提案

### 現状のセッションログで分析できること
- **何をやったか**: `作業:` 行で各セッションの実施内容を追跡可能
- **なぜそう決めたか**: `判断:` 行で意思決定とその理由を追跡可能
- **時系列**: `## HH:MM セッション` で時間軸を追跡可能
- **セッション数の変動**: 日ごとのセッション数で作業密度を把握可能

### 現状のセッションログでは分析できないこと

#### (a) 時間の使い方（所要時間）
セッション開始時刻はあるが終了時刻がない。複数セッションの開始時刻から推測は可能だが、最終セッションの長さは不明。
- **提案**: セッション終了時に `- 終了: HH:MM` を記録する、または次セッション開始時に前セッションの終了を追記する
- **分析への効果**: 「DevContainer に何時間使ったか」「設計作業の実時間」が正確に測れる

#### (b) 作業の分類（カテゴリ）
`作業:` 行の内容からカテゴリを推測はできるが、明示されていない。
- **提案**: `作業:` 行にカテゴリタグを付ける — `[設計]`, `[基盤]`, `[プロセス]`, `[レビュー]`, `[修正]` など
- **分析への効果**: カテゴリ別の時間配分・コミット数が即座に集計可能。「基盤先行型パターン」の定量化

#### (c) 計画と実績の差分
「このセッションで何をする予定だったか」が記録されていない。
- **提案**: セッション開始時に `- 予定: [今回やること]` を記録し、終了時に `- 実績: [実際にやったこと]` と突合
- **分析への効果**: 計画からの逸脱パターン（割り込み、想定外の問題）を可視化

#### (d) ブロッカー・割り込みの記録
作業中に発生した問題やブロッカーが `作業:` の中に埋もれている。
- **提案**: `- 問題: [発生した問題]（解決: [解決方法] / 未解決）` を追加
- **分析への効果**: 再発する問題パターンの検出、問題解決にかかった時間の可視化

#### (e) セッション間の引き継ぎ
前セッションからの継続作業か新規作業かが不明確。
- **提案**: 継続作業の場合 `- 継続: [前セッションからの引き継ぎ内容]` を記録
- **分析への効果**: 1つの作業が何セッションにまたがったかを追跡可能

### /session-log スキルへの具体的な改善案

現在のスキル定義の「記載内容」セクション:
```
1. 作業要約 — そのセッションで何をやったか
2. 判断事項（該当がある場合のみ） — 設計方針の決定・運用ルールの変更...
```

**提案する拡張**:
```
1. 作業要約 — そのセッションで何をやったか（カテゴリタグ付き）
2. 判断事項（該当がある場合のみ）
3. 問題・ブロッカー（該当がある場合のみ） — 発生した問題と解決状況
4. 確認事項（該当がある場合のみ） — ユーザーへの確認待ち、次セッションへの申し送り
```

**形式の拡張案**:
```markdown
## HH:MM セッション
- 作業: [基盤] DevContainer の Squid 設定を修正
- 作業: [設計] ドメインモデルの状態遷移図を作成
- 判断: RLS バイパス経路を認証エンドポイントに限定（理由: ...）
- 問題: Node.js が http_proxy を無視する（解決: SNI 透過プロキシに変更）
- 確認: branching 戦略を PR ベースに変更してよいか → ユーザー承認済み
```

### 優先度の提案
記録コストと分析価値のバランスから:
1. **カテゴリタグ**（低コスト・高価値）— `[設計]` `[基盤]` 等を `作業:` に付けるだけ
2. **問題行**（中コスト・高価値）— ブロッカーの可視化は振り返りに直結
3. **終了時刻**（低コスト・中価値）— 時間計測の精度向上
4. **予定/実績**（高コスト・高価値）— 有用だが記録の手間が大きい。週次で振り返る運用でカバー可能
