### 学び・気づき

#### MVP 対応方針の一貫性確認

- ユーザーから「ここまで起票した issue 全部 MVP 内で対応した方が良いか」相談 → 過去 #137〜#150 の方針と整合させ「SMK 検証中の UI バグは MVP 対応」が標準と確認
- 全件まとめて MVP 対応の判断は一貫性の観点で妥当
- **教訓**: スコープ判断は過去判断との一貫性を必ず確認する

#### ユーザー報告の精度確認（思い込みで起票しない）

- SMK-101 の当初報告「画面全体が横スクロール」→ DevTools 未使用時のブラウザリサイズ挙動と判明
- DevTools iPhone SE での再検証で「横スクロールなし・列幅省略」が真の問題と判明
- ユーザー自身が「嘘です」と訂正してくれた
- **教訓**: ユーザー報告を受けたら起票前に「再現環境の正確性」を確認する。特にレスポンシブ系は実機 vs DevTools vs ブラウザリサイズで挙動が異なる

#### 設計乖離の根拠は設計書直接引用で固める

- SMK-096 / SMK-101 / SMK-102 の FAIL 判定で、設計書のどの行が期待結果かを明示しながら issue 起票
- 「これは設計乖離である」を主張する際、設計書のファイルパス + 行番号 + 引用文を必ず含める
- **教訓**: FAIL 判定の根拠を設計書直接引用で示すことで、対応者が「修正方針 = 設計書通りに合わせる」と即座に判断できる

#### 推測原因と確定原因の区別

- SMK-153（承認待ちカード見た目幅狭い）の原因として「Badge wrapper の display: inline-flex」を推測したが、ユーザーから「確定ですか?」と質問
- 「コード読みからの推測。確定には DOM 検証が必要」と正直に答え、issue 内に「推定原因」「対応時に DOM 検証で確定」と明記
- **教訓**: 推測を確定として書かない。issue 内に推測の根拠と確定方法を明記することで、対応者が「まず DOM 検証する」のステップを踏める

#### コミット粒度の選択（案A vs 案B）

- 1 SMK ごとにコミット（案A）/ Phase 全消化後にまとめてコミット（案B）の選択を都度確認
- ユーザーは「節目ごとに確認できる粒度」を好む → 案A 寄りの粒度で進めた
- **教訓**: コミット粒度はユーザーの好みに合わせる。ユーザーが選択肢を提示すれば判断を委ねるが、自走中は「節目ごと小まめ」がデフォルト

#### 設計書 vs 実装の乖離の整理パターン

- パスワードリセットメール: 設計書「メール送信する」 / 実装「ログ出力で代替」
- TenantInfoCard: 設計書「Card で表示」 / 実装「div + dt/dd で平文」
- 却下バリデーション: 設計書「エラー文言表示」 / 実装「ボタン disabled のみ」
- いずれも実装側の「自然な選択」だが設計書との乖離を生んでいる
- **教訓**: 実装着手時に設計書の「期待される視覚 / 体験」を確認するチェックポイントが必要。実装は通っても SMK で発覚 → 修正の流れを Step 10 内で防げる
