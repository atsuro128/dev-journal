### 学び・気づき
- **Docker compose の profile 除外サービスは `up --build` で再ビルドされない** — `seed` が `profiles: [seed]` だったため、reset 後も古い seed イメージを使っていて検証が成立しなかった。fix は `compose run --build` フラグで対応。同じ罠は他の profile サービスにも該当する可能性
- **PowerShell スクリプトは ASCII 英語で書く** — Windows PowerShell 5.1 は BOM なし UTF-8 を Shift-JIS として読む。日本語メッセージは化けるので避ける（または BOM 付きで書く）
- **smoke_check.md の URL 表記が実装に追従していない** — 前セッションで発覚した乖離（SMK-025）が SMK 側で未修正のまま放置されていた。発見時の即時 issue 起票が機能していなかった
- **issue 化のラインを引きやすい** — SMK 検証中に発見した課題は基本「機能してるかどうか」と切り分けて、機能していれば issue 化、機能していなければ FAIL + issue。今セッションでは合計 10 件起票したが、すべてアドホック修正に走らず一貫してこのルールで進められた
- **「対応前の合意確認」セクションは UX 系 issue で有効** — UI/UX 系は実装方針が複数あり、勝手に決め打ちすると手戻りが発生する。issue 内に論点を明示しておけば、対応セッションで合意形成が機械的に進められる。ユーザーから明示的にこのパターンの導入指示があった（issue 100）
