### 学び・気づき

#### テスト基盤の構造的脆弱性
- `db-test` 非永続 + handler テストの build tag 不揃い + migrate 経路が testutil integration に間接依存、の 3 点が重なり「タイミング次第で PASS/FAIL する」状態が Step 8-7 以降潜在していた
- 「以前は通っていた」ことを根拠にスルーするのは危険。PASS が**構造的に保証されているか**を疑う癖が必要

#### handler テストの build tag 不揃いは別の潜在バグ
- `attachment/auth/category/dashboard/item/report` の 6 ファイルが tag なし（unit phase で走る）
- `tenant/workflow` の 2 ファイルだけ `//go:build integration`
- 今回は migrate-test で対症療法。根本的には unit/integration 切り分けを統一すべき（今後の issue 候補）

#### 軽量化プロセスの妥当性
- 30 行程度の基盤穴埋めに対し `/issue` 起票 → `/implement` → reviewer → codex は過剰
- ユーザー判断で指揮役直編集 → PR → 動作確認 → マージで完結（20 分程度）
- ただし expense-saas 側は master 直 commit 禁止のため feature branch + PR は必須（workflow.md §21）

### 意思決定ログ

#### 方針 A（migrate-test サービス独立）採用理由
- B（tag 統一）は 6 ファイルの test 分類変更で影響範囲確認が必要、本来は別 issue で扱うべき
- C（volume 追加）はテスト独立性を損なう一時しのぎ
- A はテスト基盤の欠落を補うだけで、既存の設計判断に干渉しない

#### `migrate-test` サービス設計
- `profile: [test]` で通常起動から除外（必要時のみ `docker compose --profile test run --rm migrate-test`）
- `depends_on: db-test (service_healthy)` で db-test 起動完了を待つ
- `restart: on-failure` で瞬間的な接続失敗をリカバー

#### tasks.json `dependsOrder: "sequence"` 指定
- `dependsOn` だけだと parallel 実行される恐れがあり、migrate 完了前にテストが走ると競合
- `sequence` 指定で必ず migrate 完了後にテスト実行

#### `--delete-branch` 不使用の継続
- 前セッションからの継続方針。PR #83 も `gh pr merge 83 --squash`（`--delete-branch` 付与なし）
