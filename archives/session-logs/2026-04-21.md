### 意思決定ログ

#### 方針 A（migrate-test サービス独立）採用理由
- B（tag 統一）は 6 ファイルの test 分類変更で影響範囲確認が必要、本来は別 issue で扱うべき
- C（volume 追加）はテスト独立性を損なう一時しのぎ
- A はテスト基盤の欠落を補うだけで、既存の設計判断に干渉しない

#### `migrate-test` サービス設計
- `profile: [test]` で通常起動から除外（必要時のみ `docker compose --profile test run --rm migrate-test`）
- `depends_on: db-test (service_healthy)` で db-test 起動完了を待つ
- `restart: on-failure` で瞬間的な接続失敗をリカバー

#### tasks.json `dependsOrder: "sequence"` 指定
- `dependsOn` だけだと parallel 実行される恐れがあり、migrate 完了前にテストが走ると競合
- `sequence` 指定で必ず migrate 完了後にテスト実行

#### `--delete-branch` 不使用の継続
- 前セッションからの継続方針。PR #83 も `gh pr merge 83 --squash`（`--delete-branch` 付与なし）

### 学び・気づき

#### PR スタック運用の罠（rebase 時の最新化）

force push された下流 PR を rebase する際、古い worktree の local ref で rebase すると **origin の最新 commit が失われる**。再発防止:
- 必ず `git fetch origin && git reset --hard origin/<branch>` で最新化してから rebase

#### サブエージェントへの確認取りすぎ（自走モードの運用）

ユーザーから「マージ以外は自走」と明示されたのに `/test` 実行の確認を取り反省。

#### /test スキルの手順違反（tail パイプで進捗見えず）

`npm test 2>&1 | tail -40` で起動したため 18 分間進捗が完全にブラックボックスに。
- **再発防止**: 長時間コマンドは `Bash(run_in_background: true)` で起動する

#### 削除エラーの表示経路バグ（#135 として分離）

追加したテストが既存バグを検出して失敗した場合、テスト削除ではなくバグ発見として新 issue 起票が正しい対応。

#### codex の価値（設計意図と実装の乖離検出）

reviewer + codex の二段構えは相互補完。codex は設計意図と実装の乖離を実行ベースで検出する。
