### 学び・気づき

#### スコープ creep の典型パターンを実体験

- 元の課題（issue #156 / #159 = 却下ダイアログ限定の問題）→ codex F1 指摘 → 私が「FormAlert パターンと整合」と過度に一般化 → ユーザーに「全使用箇所拡張？」と提示 → 承諾 → 5 ダイアログ拡張 → loading 未連動が露呈 → さらに修正 → codex が「提出/削除も統一を」要求 → 際限なく広がりかけた
- ユーザーの「**なんでエラートーストを廃止するんですか？**」「**元々の問題は何だった？趣旨がずれてないか？**」で気づき、案 C で巻き戻し
- **教訓**: codex 等の「設計書と実装の乖離」指摘は、**設計書側の記述が拡大解釈されていないか** をまず疑う。原規定（D4 = 却下ダイアログ form validation）のスコープを意識的に守る

#### 過剰適用 vs 正しい設計の境目

- バッジパターンの正しい用途は「情報が隠れている対象」（アイコン / メニュー項目）。件数が見えるカードでは冗長
- apiError + FormAlert パターンは「form-context error」（required input validation）に有効。単純確認ダイアログでは旧トーストで十分
- **教訓**: 「整合性のため統一しよう」が必ずしも正解ではない。**目的別のパターン使い分け** が UX としても正しい

#### codex の指摘を批判的に評価する重要性

- memory `feedback_critical_review_of_codex` は「形式的な指摘には押し返す」と教えていたが、本セッションでは codex F1 指摘を「形式的でない」と判断して全使用箇所拡張に進んでしまった
- 「設計書と実装の乖離」自体は事実だが、**設計書側の記述が原規定（D4）を逸脱していたら設計書を訂正する選択肢もある**
- **教訓**: 「設計書 vs 実装の乖離」指摘を受けたら、双方が原規定（issue / requirements）に整合しているかを確認する。設計書側が拡大解釈なら設計書を訂正する

#### ユーザーの「素朴な質問」が方針修正のトリガーになる

- 「なんでエラートーストを廃止するんですか？」のような素朴な質問は、私が暗黙に置いていた前提を可視化する
- **教訓**: ユーザーの「？」には方針反省の機会が含まれる。即座に正当化せず、まず立ち止まって元の課題に戻る

#### 並列処理の費用対効果

- 4 ブランチ並列で実装 → CI → レビュー → マージまで完遂できた（worktree 隔離 + run_in_background 効果）
- ただし G2 だけは 5 イテレーション（初回 → W1 W2 → 大幅改修 → 巻き戻し）を要した
- **教訓**: 単純な改修は並列が圧倒的に効率的。設計判断を伴う改修は逐次の方が安全（先に方針確定 → 並列実装の流れがベスト）

#### #152 Badge dot 削除の根拠（#085 整合ではなく独立判断）

- ユーザーから「却下のバッジを消してしまっていたため、消えているから合わせるという考えは捨てて考えてください」と指示
- 改めて「バッジが価値を出すのは情報が隠れている対象」原則から考察 → カード本体に件数が表示されているため Badge dot は冗長
- 結果的に #085 と同じ結論だが、根拠は独立した設計原則（情報冗長性）

#### apiError パターン適用範囲（案 C 巻き戻し）

- 全 7 ダイアログ統一 → 却下のみに限定
- 設計書 D4 規定の本来のスコープ（required input validation 422 = MissingRejectionReason）に整合
- アプリ全体のトーストパターンは維持（ReportEditPage / ItemForm の FormAlert は form 専用、確認ダイアログとは責務が異なる）

#### loading prop 連動は全ダイアログで keep

- apiError パターン採用ダイアログ（却下）では多重送信防止 + キャンセル不可で必須
- 旧トーストパターンダイアログでも多重押下防止として有用
- → 共通基盤として全 7 ダイアログで `loading={mutation.isPending}` 連動を維持

#### usePrevious 全表示 props 拡張は keep

- 案 C で apiError 適用範囲を狭めても、usePrevious による表示値保持は全表示 props で必要
- 却下ダイアログでフェードアウト中に「却下する」ボタンが「支払完了にする」に変わる等の他 props ちらつきは依然として #156 完了条件「閉じる際にちらつかない」を脅かすため
