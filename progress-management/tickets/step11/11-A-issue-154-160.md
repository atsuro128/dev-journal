## 責務

### 含めること

#### #154 対応（minHeight 条件付き適用）

- `AppDataGrid.tsx` の DataGrid sx の `minHeight` を **0 件時のみ条件付き適用** に変更する
  - 0 件時: `300`（EmptyState のアクションボタン込みで見切れない値）
  - 1 件以上: `undefined`（DataGrid 自然高さに任せる）
- 実装パターン: `minHeight: rows.length === 0 ? 300 : undefined`
- 既存テスト ADG-005 を改修し、0 件時 / 1 件以上の両ケースを検証
- 設計書 §AppDataGrid の minHeight 規定を「件数 0 時のみ条件付き適用（300px）」に更新

#### #160 対応（横スクロール再対応）

- 再検証 FAIL の真因を実装側で特定する。**現時点の grep 結果では `overflowX: 'auto'` も 4 画面の minWidth 絶対値（100-200px）も既に設定済み**であるため、以下のいずれかが推定原因
  - 推定 a: `overflowX: 'auto'` が **Box ルート** に付いているが、内部 DataGrid が `width: '100%'` で親 Box 幅にフィットしてしまい、列幅合計（合計 630-700px 程度）が画面幅（375px）を超えても DataGrid 自体が縮もうとして overflow が発生しない
  - 推定 b: DataGrid 標準の `flex: 1` と `minWidth` の組み合わせで、`flex` が支配的になり minWidth が効いていない（MUI X DataGrid の `flex` は分数指定であって絶対幅ではない）
  - 推定 c: 親要素（PageContainer 等）側で `overflow: hidden` 等が効いており、AppDataGrid の `overflowX: 'auto'` が伝播していない
- worktree 内で DevTools 相当の検証 + コード読みで原因を確定し、ピンポイントで修正する。修正候補は以下
  - 候補 1: DataGrid 自身の sx にも `overflowX: 'auto'` を追加（Box ルートと併用）
  - 候補 2: 4 画面の `flex: N` 指定をやめて `width` を絶対値固定にする（`flex` を外す）
  - 候補 3: AppDataGrid の Box ルートの `width: '100%'` を維持しつつ、内部 DataGrid のラップ要素（`MuiDataGrid-root` 等）に `minWidth` を渡し、列幅合計を強制する
- columnDef の minWidth 絶対値（100-200px）は現状維持。`flex` 併用の可否は原因確定後に判断
- 設計書 §AppDataGrid の overflowX / minWidth 規定を、原因に応じた具体策（推奨は候補 1 の Box + DataGrid 両方に overflowX）で明文化する

### AppDataGrid.tsx 改修パターン（#160 真因確定後）

worktree 内で以下の順で原因を絞り込む:

1. 現状実装の `overflowX: 'auto'` が Box ルートに付いていることを再確認（既に L34-35 で確認済み）
2. 4 画面の columnDef に `flex: N` と `minWidth: M` が併用されていることを確認（grep 結果から確定済み: 全 4 画面で `flex` + `minWidth` 併用）
3. **最有力仮説**: MUI X DataGrid の挙動で、`flex` が指定された列はコンテナ幅を `flex` 比率で分配し、コンテナ自体が画面幅にフィット（`width: '100%'`）するため、`minWidth` が効くより前にコンテナが縮む。結果、各列が `minWidth` 値より小さくなり ellipsis 省略になる
4. 修正候補（推奨優先順）:
   - **候補 A（推奨）**: 4 画面の `flex` 指定を全て削除し、`width` または `minWidth` のみで列幅を決める。これにより列幅合計（630-700px 程度）が固定となり、375px のスマホ幅では合計幅が超過するため Box ルートの `overflowX: 'auto'` が発火する
   - **候補 B**: `flex` を維持しつつ、AppDataGrid の DataGrid 自身に内部最小幅を強制する（`& .MuiDataGrid-main { minWidth: 700px }` 等）。DataGrid の幅が外側の Box 幅より大きくなり、Box ルートの `overflowX: 'auto'` が発火する
   - **候補 C**: ルート Box に加えて DataGrid 自身の sx に `overflowX: 'auto'` を追加し、内部 virtualScroller の overflow を許可する

worktree で実装エージェントが上記候補を順に試し、SMK-101 期待結果（横スクロール可能 + 列内容が読める）が満たされる最小修正を選択する。**推奨は候補 A**（最も MUI X 仕様に素直）。

## リスク・注意点

1. **#160 真因の不確実性**
   - 本チケット起票時点で「現状実装には `overflowX: 'auto'` も `minWidth` 絶対値も既に設定済み」が grep で確認できている。再検証 FAIL の真因は実機 DevTools での挙動観察が必須
   - frontend-developer は Phase 1 の真因確定を**先に完了**してから実装に入ること。原因不明のまま「とりあえず候補 A を試す」のは案件として不誠実（feedback_accountability）
   - 候補 A / B / C のいずれを選択するかで設計書改訂内容が変わるため、Phase 1 → Phase 3 設計書改訂までを 1 PR で完結させる
2. **`flex` 削除の副作用（候補 A 採用時）**
   - 4 画面の DataGrid 領域が画面幅にフィットしなくなり、デスクトップ幅で右側に空白が空く可能性
   - 対策: 最後尾の列のみ `flex: 1` を残す or DataGrid 領域に `width: '100%'` + 親 Box の `overflowX: 'auto'` で対応
   - PC / タブレット / スマホ各幅で**必ず動作確認**する
3. **モック拡張による既存テスト破壊**
   - ADG-005 改修のため `data-has-min-height` を `data-min-height`（実値）に変える場合、ADG-005a/b 以外で `data-has-min-height` を参照している箇所がないか念のため grep で確認
4. **設計書改訂の最終文言**
   - Phase 2 PR の実装内容を Phase 3 で設計書に反映する。Phase 2 と Phase 3 の文言齟齬がないよう、指揮役が dev-journal commit 前に PR 差分と最終照合する
5. **MVP スコープ厳守**
   - 「minHeight 条件付き適用 + #160 真因ピンポイント修正」のみ。AppDataGrid のアーキテクチャ刷新（NoRowsOverlay の外部化、columnVisibilityModel 導入等）は含めない
6. **既存 PR #107 の上書き**
   - 本チケットは PR #107 で導入された `minHeight: 200` 一律適用を**部分的に上書き**する。コミットメッセージ・PR 本文で「PR #107 の修正方針を再オープン後 案A に変更」を明記する
