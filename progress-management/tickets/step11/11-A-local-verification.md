## 不具合発見時の運用

SMK 実施中に不具合・UI 崩れ・設計ギャップ等を発見した場合は **その場で即座に issue 起票** する。`11-A-smoke-results.md` に「UI 観察バッファ」「発見した不具合」等の滞留セクションを作らない。

### 理由

- 粒度を問わず即時起票することで、後からまとめて分類する手間と漏れを防ぐ
- 軽微な UI 観察レベルの項目も独立 issue として記録されることで、修正可否の判断履歴がトレース可能になる
- 優先度判断・対応順序は issue メタデータ（影響度・ブロッカー判定）で行い、バッファ段階での仕分けを廃止する

### 起票時のルール

- issue ID は連番（`dev-journal/issues/open/` の既存最大番号 + 1）
- ファイル名: `{ID}-{kebab-case-title}.md`
- 既存テンプレート（例: `issues/resolved/083-auth-token-persistence-on-reload.md`）に準拠
- 起票後、本チケットの「発行 issue」テーブルに ID・タイトル・**起票 SMK**・状態を追記
  - 起票 SMK 列は再検証時の逆引きに使う（各 Phase 末尾の再検証で「この Phase 内で resolved になった issue」を抽出するため）
  - SMK と直接紐付かない場合（post-MVP / 一般的バグ / ITM-007 相当の仕様漏れ等）は `-` を入れる
- FAIL 判定の SMK は結果テーブルの備考欄を「→ issue {ID}」に簡約化

### SMK 実施後の必須更新（本チケットの運用ルール）

SMK を 1 件実施するたびに、**次の SMK に進む前**に本チケットの以下 3 箇所を必ず更新する。
本チケットは「次の SMK を決める起点ファイル」であり、更新漏れがあると次セッションで誤った前提で再開してしまう。

1. **結果テーブル（§4.x 配下）**: 該当 SMK 行の `状態` / `確認日` / `備考` を更新
   - PASS: `状態=PASS`、副次発見があれば備考に「→ issue {ID}」
   - FAIL: `状態=FAIL`、備考に「→ issue {ID}」
2. **発行 issue テーブル**: セッション中に起票した issue を追記
   - 起票 SMK 列には「SMK-XXX」または「SMK-XXX 副次」（SMK 実施中の副次発見の場合）と記載
3. **進捗サマリ**: `完了: N / 62` と PASS / FAIL / SKIP / 未実施 の件数を再計算。次に実施する SMK の ID を明示する

### 設計原則

1. **Member を最優先で潰す** — ボリューム最大（約 37 項目）の役割を先に終わらせ、進捗を稼ぐ
2. **データ依存の順序を守る** — Approver SMK-011（承認）や Accounting SMK-092（支払）はフィクスチャ状態を変更する。`reportSubmitted` / `reportApproved` を参照する Member 項目はこれより**前**に実施する
3. **未ログイン項目は最後** — 全ロール完了後にログアウト状態でまとめて実施
4. **各 Phase 末尾で再検証** — 後述「各 Phase 末尾の再検証」参照

### 各 Phase 末尾の再検証

各 Phase の通常項目を通して実施した後、**その Phase 内で発生した FAIL のうち対応 issue が resolved になったもの**を再検証ブロックとして同じロール・同じセッション内で実施する。

#### 対象判定
- その Phase 内で FAIL 判定した SMK のうち、対応 issue が `dev-journal/issues/resolved/` に移動済みのもの
- 副次発見で関連 issue が resolved になった SMK（PASS 判定だった項目も影響範囲によっては再確認対象）

#### 実施方針
- ロールログイン済み・フィクスチャ状態が新鮮なうちに連続実施する（ロール切替コスト最小）
- 再実施の結果を結果テーブルの「状態」列で FAIL → PASS に更新、備考欄に「再検証 YYYY-MM-DD（PR #NN）」を追記
- 対応 issue が未 resolved の SMK は FAIL のまま次セッション以降へ持ち越す（Phase を跨ぐ保留扱い）

#### 完了条件への反映
- 全 Phase 完了 + 全 resolved 分の再検証 PASS で 11-A クローズ
- 未 resolved の FAIL が残存する場合は、対応 issue の解決後に改めて再検証を行うまで 11-A は「未完了」扱い
