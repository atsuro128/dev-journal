### 学び・気づき

- **codex の pushback は慎重に判断する** — 1回目の codex 指摘で「§3.2 はアプリケーション設定の概念記述」「monitoring.md/ADR はスコープ外」と pushback したが、ユーザーから「指摘自体は妥当では？」と確認され、再評価で実害ありと判定。codex の指摘は形式的に見えても、本文を読み直すと「設定する」等の動詞があれば値の指示になっている。**pushback の判断基準は「概念名か / 設定値の指示か」を文脈で判定する**こと。形式的な見た目だけで pushback してはいけない
- **issue 079 Phase 1 の方針判断は issue 078 と同パターン** — 「実装が動いているなら設計書を実装に寄せる」が MVP の正攻法。実装変更は PR フロー・テスト・レビューが必要なオーバーヘッドが大きく、設計書修正は dev-journal の 1 コミットで済む。同じ問題は同じ方針で処理する
- **codex レビューが反復するときの集約点を把握する** — LOG_LEVEL の小文字化は 4 ラウンド（env_config.md §4.1/§7.1 → §3.2 → monitoring.md コード例 → monitoring.md §3.1 本文 → ADR-0005）かかった。最初に「LOG_LEVEL を全ドキュメントで小文字統一」というスコープで grep していれば 1 ラウンドで終わった。**「キーワード修正は文書横断の grep を最初に実行」**を徹底する。`feedback_search_before_delete.md` を「修正前の grep」にも適用すべき
- **設計書 §4.x 棚卸しの精度** — Phase 0/1/2 を通じて、設計書の正確性が「実装と乖離しないか」だけでなく「セクション内で自己整合しているか」も問われる。1ドキュメント内の §3.2 と §4.1 が矛盾するパターンは codex の指摘で初めて気づいた
- **設計成果物フローのレビューラウンド数の見積もり** — issue 079 は内部レビュー 2 ラウンド + codex レビュー 4 ラウンド = 計 6 ラウンドを要した。波及範囲が広い設計書修正では codex が新しい乖離を芋づる式に発掘するため、1 回で終わると見積もるのは楽観的すぎる
- **owner pool の MaxConns=5 ハードコード認識漏れ** — 当初「DB プールは pgxpool デフォルト使用」と注記したが、owner pool だけ `MaxConns = 5` を明示設定していた。設計書注記は実装の細部まで反映するべきで、「デフォルト使用」のような断定は実装を全面確認してから書く

- **issue 079 Phase 1 方針 A 採用**: issue 078 と同じ「設計書を実装に寄せる」。実装の `JWT_PRIVATE_KEY_PATH` / `JWT_PUBLIC_KEY_PATH` をそのままに、env_config.md と security.md を `_PATH` 方式に統一。ECS では Secrets Manager から PEM 値を `JWT_*_PEM` として注入し、エントリーポイントスクリプトで一時ファイルに書き出す方式を §5.3 に明記
- **issue 079 Phase 2 全削除**: 14 変数すべて §4.x から削除。要件固定値（SEC-002/003、security.md §2.1）の 5 件は issue 077 と同パターン。要件なし固定値 6 件・運用可変候補 3 件も「実装ハードコード」「全環境同値」「MVP では運用変更の実益薄」の理由で削除。「実装に環境変数参照を追加する」選択肢は MVP スコープ外
- **codex 指摘への pushback 撤回判断**: 1回目で §3.2 ログレベル「概念名」、monitoring.md/ADR「スコープ外」と pushback したが、ユーザー確認で再評価。「設定する」と明記されている箇所は値の指示なので、概念名としての pushback は誤り。同一文書内 (§3.2 vs §4.1) の矛盾は同一 issue で解消すべきというユーザー判断
