# 認可エラーの 403/404 仕様が文書間で不整合

## 発見日
2026-03-05

## 関連ステップ
Step 1（要件定義）

## カテゴリ
security

## 影響度
中

## 問題
`requirements.md` では「権限不足時は 403 Forbidden（RBC-004）」のみ定義されている。  
一方で `rbac.md` と `rules/security-policy.md` では「他テナントリソースへのアクセスは 404 Not Found（存在漏洩防止）」を定義している。

同じ認可失敗でも、ケース別のHTTPステータスが文書ごとに揺れている。

## 影響
- 実装時にハンドラ/ミドルウェアで返却コードが分岐し、統一できない
- テストケースの期待値（403か404か）が環境や実装者でぶれる
- 他テナント存在漏洩の防止方針が曖昧になる

## 提案
- `requirements.md` にケース分岐を明記する
  - 権限不足（同テナント内のロール不足）: 403
  - テナント境界越えアクセス: 404
- RBACルールIDを分離して表現を明確化する（例: `RBC-004` と `RBC-005`）
- `rbac.md` / `security-policy.md` と同一文言に統一する

## 参照
- `review-findings/015-authz-error-code-403-404-inconsistency.md`

---

## 解決内容

対応不要と判断。理由:
- requirements.md の RBC-004（403）はロールベース認可のケースを定義
- rbac.md 6節・security-policy.md 2節はテナント境界越えのケース（404）を定義
- 両者は層別に分離された正しい構造であり、不整合ではない
- requirements.md は「詳細な権限マトリクスは rbac.md を参照」と明示的に委譲しており、ケース分岐は rbac.md 側で完結している
