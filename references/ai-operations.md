# AI 運用方針

## 概要

本プロジェクトは「AIを主軸に置いたアプリ開発」を実践するポートフォリオである。
単にAIを補助ツールとして使うのではなく、設計・実装・レビューの各フェーズにAIエージェントを組み込み、
人間はレビューと意思決定に集中する運用を目指している。

---

## AIエージェント構成

| エージェント | 種別 | 主な役割 |
|---|---|---|
| Claude (Claude Code) | 対話型 | 設計・実装・ドキュメント作成・判断 |
| Codex | 非同期型 | 成果物レビュー・issue 起票 |

### 役割分担の思想

- **Claude**: 人間との対話の中でタスクを進める。判断が必要な場面では人間に確認を求める。
- **Codex**: 完成した成果物を非同期でレビューし、指摘を issue として起票する。人間の作業フローを止めない。

---

## コンテキスト設計

### 基本思想：コンテキスト節約

AIの初回ロード時に読み込まれる情報を最小限に抑え、必要な詳細はタスクに応じて遅延ロードする。

### 実装方針

- **エントリーポイントは索引に徹する**: CLAUDE.md / AGENTS.md には概要と参照先のみを記載し、手順の全文は書かない。
- **タスク条件付きロード**: 作業種別に応じて読み込むファイルを切り替える。詳細手順は `agents/` に格納し、AGENTS.md から条件分岐で参照させる。
- **常時適用ルールのみインライン**: 毎回必要な制約（禁止事項・コミットルール等）だけをエントリーポイントに残す。

### ファイル構成

```
CLAUDE.md          ← Claude 向けエントリーポイント（常時適用ルール + 参照索引）
AGENTS.md          ← Codex 向けエントリーポイント（タスク条件付きロード指示）
agents/            ← Codex が作業時に読み込む詳細手順
rules/             ← Claude・人間が共有するプロジェクトルール（詳細）
```

## ワークフロー設計

### issue ライフサイクル(TODO)→issue記載はcodexに全委任するの？

```
issues/           → 対応中・未対応
pending-review/   → 対応済み・レビュー待ち
resolved/         → レビュー完了・クローズ済み
```

Codex が起票した issue を Claude が対応し、`pending-review/` へ移動。
Codex が再レビューして問題なければ `resolved/` へ移動する。

### review-findings ライフサイクル

```
review-findings/
├── open/             # 未対応の指摘詳細
├── pending-review/   # 対応済み・再レビュー待ち
└── resolved/         # 再レビュー完了済み
```

issue と 1:1 で対応する指摘詳細ファイルを管理し、issue の状態と連動して移動する。

### 人間の関与ポイント

- issue の `pending-review/` 移動前に人間が対応内容を確認する（自動移動禁止）
- 設計上の重要な判断は `logs/YYYY-MM-DD/decisions.md` に記録する

---

## 運用から得られた知見

- エントリーポイントの肥大化はAIの初回コンテキストを圧迫するため、定期的に削減を検討する
- Claude 向けルール（`rules/`）と Codex 向け手順（`agents/`）は明確に分離する
- 「AIが自律的に動ける範囲」と「人間が判断する範囲」の境界を明示することが重要
