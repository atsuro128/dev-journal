# ADR-005: リグレッションテストのスコープ — ポートフォリオ MVP 向け簡略化

## ステータス
承認済

## 日付
2026-04-20

## 背景（ペイン）

Step 11-A（ローカル動作確認）実施中に多数の issue が起票 → 修正 → マージされている。本来のシステムテストの原則では、修正 PR のたびに全項目を再実行して **リグレッション（デグレ）** がないことを確認すべきだが、本プロジェクトの性格を踏まえると現実的ではない。

### 検討のトリガー

- Step 11-A Phase 2 実施中に 20 件超の issue を起票・並行対応
- 修正 PR が共通層（api/client.ts 等）に手を入れるケースが複数あり、波及影響を持ちうる
- 起票 SMK と紐付かない `-` issue（例: 117 / 127）の修正が別 SMK に影響する可能性がある
- SMK は 62 項目・手動実施。フル再実行は Phase 毎に数時間規模

### プロジェクトの性格

- **ポートフォリオ用途の MVP**。商用リリースではない
- 開発者は単独（ユーザー1名）で、テスト担当者を別に立てない
- 自動テスト（11-B 横断 Go / 11-C Playwright E2E）と UAT（11-F）で多層防御を構成する設計

---

## 検討した選択肢

### A. 現行案（Phase 末尾再検証）— 最小コスト
- 各 Phase の通常項目完了後、その Phase 内で resolved になった issue の SMK のみ再検証
- デグレは 11-B / 11-C / 11-F で拾う前提
- 工数: 小

### B. 11-B/C PASS 後に 11-A を軽量再実行 — 中間案
- 自動テスト通過後、主要 SMK のみ再実施（例: 各 Phase から代表項目を 3 件）
- 工数: 中

### C. 11-A 全項目やり直し — 最も安全
- 全 Phase を通しで再実行
- 工数: 大（Phase 毎に数時間 × 6 Phase）

---

## 決定

**案 A（Phase 末尾再検証）を採用する**。

### 理由

1. **多層防御で担保する設計**
   - 11-B 横断 Go テスト: テナント分離・RBAC・非機能のデグレを自動で検出
   - 11-C Playwright E2E: 主要フロー（申請→承認→支払/却下→再申請）のデグレを自動で検出
   - 11-F UAT: ユーザー視点の実操作で最終関門を担う
2. **コスト対効果**
   - MVP スコープで全 SMK 手動フル再実行は過剰
   - 自動テストが回っていれば代表的なデグレは拾える
3. **目的との整合**
   - 本プロジェクトの目的はポートフォリオ提示（開発プロセスと成果物の実証）
   - 商用品質のリグレッション保証は目的外

### 運用ルール

- 各 Phase 通常項目完了 → その Phase 内で FAIL → resolved になった issue の SMK を再検証 → PASS 判定（11-A チケット「各 Phase 末尾の再検証」に準拠）
- 起票 SMK が `-` の issue（一般的バグ / 副次発見）の修正は、PR 内容から影響 SMK を判断して必要なら再検証に含める（強制ではない）
- 全 Phase 完了後、11-B / 11-C のテストがデグレ検出を担う

---

## 残リスク

- 11-B / 11-C でカバーされない UI 細部（例: トースト表示位置、レスポンシブのブレイクポイント）のデグレを見逃す可能性
- 11-F UAT での人手確認が最後の砦になる。UAT で発見したデグレは post-MVP issue として別扱い

### 緩和策

- 11-C E2E で主要業務フロー網羅を保証（test_cases/cross-cutting.md §3）
- 11-F UAT の uat_check.md チェックリストで UI 細部も人手確認
- デグレが UAT 後に発覚した場合は issue 起票 → 次イテレーション対応（商用運用ではないので許容）

---

## 適用範囲

- Step 11-A（ローカル動作確認）におけるリグレッションテスト運用
- 本決定は **MVP スコープの範囲内のみ**有効。商用リリースを視野に入れる場合は再検討が必要
