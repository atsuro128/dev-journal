# ADR-004: 作業分解と並列実行による開発プロセス改善

## ステータス
承認済

## 日付
2026-03-15

## 背景（ペイン）

`project_steps.md` を全体地図として運用してきたが、Step 2 完了時点で以下の問題が顕在化した。

### 問題1: 毎セッションの参照コストが高い

`project_steps.md` は全 Step（0〜7）を網羅する約340行の文書。セッション開始時に毎回全文を読み込んでおり、作業中の Step に無関係な情報がノイズになっている。Claude Code のコンテキストウィンドウを圧迫し、作業精度にも悪影響を与える。

### 問題2: 下流 Step ほど粒度が粗い

Step 0〜3（要件定義・ドメイン設計・アーキテクチャ設計）は成果物が明確で1セッションで完結可能だった。しかし Step 5（詳細設計）以降は「DB スキーマ + OpenAPI + 認可 + セキュリティ + 添付ファイル」と複数の専門領域にまたがり、1ステップを1セッションで完了させるのは非現実的。作業指示としての具体性も不足している。

### 問題3: 並列実行の仕組みがない

Claude Code を複数セッション並列で走らせることで開発速度を上げられるが、現行の Step 単位の管理では「どのセッションが何を担当するか」「依存関係はどうなっているか」を判断する基盤がない。

### 問題4: Step 4（基本設計）と Step 5（詳細設計）の分離が非効率

現行は「全画面を設計（Step 4）→ 全 API を設計（Step 5）」という横割り構成。画面設計と API 設計は密結合（画面の入出力 ≒ API の入出力）であり、横割りにすると画面設計の手戻りが API 設計に波及する。機能単位で縦割りにする方が自然。

### 従来の運用フロー
```
セッション開始
  → project_steps.md 全文読み込み（約340行）
  → progress.md で現在位置確認
  → Step 単位で作業開始（1 Step = 1セッション前提）
  → 完了 or 中断
```

## 検討した選択肢

### 案A: project_steps.md をStep別ファイルに分割

- **概要**: project_steps.md を step0.md, step1.md ... に分割し、セッション開始時に該当 Step だけ読む
- **利点**: 参照コスト削減。既存の構造をほぼそのまま維持
- **欠点**: Step 内の粒度の粗さは解決しない。並列実行の基盤にならない。Step 4+5 の非効率な横割りもそのまま

### 案B: 作業分解ファイル（Work Breakdown）の導入

- **概要**: project_steps.md は全体地図として維持。各 Step 着手前に「作業分解ファイル」を作成し、タスク単位で依存関係・入出力・並列可否を定義する。Step 4+5 は機能単位の縦割りに再構成する
- **利点**: セッションはタスク ID だけで作業開始可能。依存グラフに基づく並列実行が可能。機能単位の縦割りで手戻りが閉じる
- **欠点**: 作業分解ファイルの作成・維持コストが発生する。Step 着手前に分解作業が必要

### 案C: プロジェクト管理ツール（GitHub Projects 等）への移行

- **概要**: タスク管理を外部ツールに移し、Claude Code は API 経由でタスクを取得する
- **利点**: タスク管理の標準的な機能（担当者割当、フィルタ、カンバン）が使える
- **欠点**: Claude Code のコンテキストに自然に入らない。ファイルベースの運用との二重管理になる。セットアップコストが高い

## 決定

**案B: 作業分解ファイルの導入**を採用する。

### 変更内容

#### 1. 作業分解ファイルの配置

```
dev-journal/guide/
  project_steps.md              ← 全体地図（変更なし）
  work-breakdown/
    step3-architecture.md       ← Step 3 の作業分解
    step4-5-design.md           ← Step 4+5 統合・機能単位の縦割り
    step6-test-design.md        ← Step 6（Step 4+5 確定後に作成）
    step7-phaseN-xxx.md         ← Step 7（Phase 単位でさらに分解）
```

#### 2. 作業分解ファイルの構成

各ファイルは以下を含む:
- **前提**: 上流成果物（入力）のパス一覧、完了条件
- **タスク一覧**: ID、入力、出力、依存関係、並列可否、状態
- **依存グラフ**: タスク間の依存を図示
- **並列実行パターン**: Wave（同時実行可能なタスク群）の定義
- **タスク詳細**: 各タスクの具体的な作業内容と完了条件

#### 3. Step 4+5 の機能単位統合

従来の横割り（全画面 → 全 API）を、以下の機能単位の縦割りに変更:

| タスク | 内容 | 並列 |
|--------|------|------|
| 4+5-A | DB スキーマ設計（全体） | Wave 1 |
| 4+5-B | セキュリティ設計（横断） | Wave 1 |
| 4+5-C | 認証・ユーザー管理（画面 + API） | Wave 2 |
| 4+5-D | 経費レポート CRUD（画面 + API） | Wave 2 |
| 4+5-E | 添付ファイル（画面 + API + S3） | Wave 3 |
| 4+5-F | 承認フロー（画面 + API） | Wave 3 |
| 4+5-G | 認可設計 + 画面遷移統合 | Wave 4 |

各機能タスクは「画面仕様 + API 定義」をセットで出力し、手戻りが機能内に閉じる設計。

#### 4. セッション運用の変更

```
【従来】
セッション開始 → project_steps.md 全文 → progress.md → Step 単位で作業

【改善後】
セッション開始 → work-breakdown の担当タスク ID 確認
→ タスクの「入力」に書かれた成果物だけ読む → 作業 → タスク状態更新
```

## 理由

- **参照コストの削減**: セッションが必要とする情報がタスク詳細に集約されており、project_steps.md の全文読み込みが不要
- **並列実行の基盤**: 依存グラフと Wave 定義により、並列実行可能なタスクを機械的に判断できる
- **手戻りの局所化**: 機能単位の縦割りにより、ある機能の設計変更が他の機能タスクに波及しにくい
- **段階的な適用**: Step 3 で小規模に検証し、Step 4+5 で本格運用する計画のため、リスクが低い

## 影響・結果

- **project_steps.md は変更しない**: 全体地図としての役割を維持する。ただし、セッション開始時の必須参照先ではなくなる
- **ワークフロー規約の更新が必要**: `.claude/rules/workflow.md` のセッション開始手順を work-breakdown ベースに更新する必要がある
- **progress.md との関係を整理する必要がある**: Step 単位の進捗と タスク単位の進捗の二重管理を避ける設計が必要
- **レビュー粒度の再定義が必要**: タスク単位レビュー vs Wave 単位レビューの判断が未決定
- **並列実行時のファイル競合リスク**: 複数タスクが同一ファイル（openapi.yaml 等）に書き込む場合の運用ルールが必要
