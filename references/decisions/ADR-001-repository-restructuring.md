# ADR-001: リポジトリ構成の分離

## ステータス
承認済

## 日付
2026-03-08

## 背景（ペイン）

現在 `root-project` に以下の4つの関心事が混在しており、管理が困難になっている。

1. **AI運用設定**: CLAUDE.md, .claude/, rules/, prompts/, agents/, scripts/
2. **設計成果物**: deliverables/, references/, templates/
3. **プロジェクト管理**: progress-management/, daily-reports/, logs/, guide/
4. **プロダクトコード**: project/

### 具体的なペイン
- **ポートフォリオの訴求点が埋もれる**: このプロジェクトの最大の売りは「AI駆動開発の設計力」だが、AI運用設定が設計資料・日報・ログと同列に散在しており、見せたいものが見えない
- **関心の分離ができていない**: AI運用フレームワークは本来プロダクト非依存で再利用可能なはずだが、経費精算SaaS固有のファイルと同じリポジトリにあるため、汎用性が伝わらない
- **プロダクトコードが入れ子構造**: `root-project/project/` というネスト構造は不自然で、実務のリポジトリ構成とかけ離れている
- **Git履歴が混線する**: AI運用の改善コミットとプロダクト開発のコミットが同じ履歴に混在し、変更の意図が追いづらい

## 検討した選択肢

### 案A: 現状維持（モノレポのまま整理）
- **概要**: ディレクトリ構成だけ整理し、リポジトリは分けない
- **利点**: 移行作業が不要、ファイル間の参照が楽
- **欠点**: 根本的な関心の混在が解消しない、ポートフォリオとしての見せ方が改善しない

### 案B: 2リポジトリ（AI運用+プロダクト / 開発記録）
- **概要**: AI運用設定をプロダクトと一体化し、開発記録だけ分離
- **利点**: 分離の手間が少ない
- **欠点**: AI運用フレームワークの汎用性が示せない、プロダクトリポジトリが雑多なまま

### 案C: 3リポジトリ分離
- **概要**: AI運用フレームワーク / プロダクト本体 / 開発プロセス記録に分離
- **利点**: 各リポジトリの責務が明確、AI運用が独立した成果物として訴求できる、プロダクトリポジトリが実務に近い構成になる
- **欠点**: 移行作業が最も大きい、リポジトリ間の参照関係の設計が必要

## 決定

**案C: 3リポジトリ分離**を採用する。

### リポジトリ構成

| リポジトリ | 責務 | 公開 |
|---|---|---|
| `ai-dev-framework` | AI駆動開発フレームワーク（ポートフォリオの主役） | public |
| `expense-saas` | プロダクト本体（経費精算SaaS） | public |
| `dev-journal` | 開発プロセスの記録（日報・ログ・進捗） | private or public |

### 各リポジトリの内容

**ai-dev-framework/**
- CLAUDE.md のテンプレート・設計思想ドキュメント
- Hooks定義、カスタムコマンド（.claude/commands/）
- ルール体系（rules/）
- オーケストレーション設計、サブエージェントパターン（agents/）
- 自動化スクリプト（scripts/）
- ADR（decisions/）— AI運用に関する意思決定記録
- テンプレート（templates/）

**expense-saas/**
- アプリケーションコード（現 project/）— 実装コードのみ

**dev-journal/**
- progress-management/（進捗管理）
- daily-reports/（作業日報）
- logs/（セッションログ）
- review-findings/（レビュー記録）
- deliverables/（設計成果物）
- references/（参照資料）
- guide/（プロジェクト進行ガイド）

## 理由

- **ポートフォリオの訴求構造が明確になる**: 面接官に「まず `ai-dev-framework` を見てください」と言える。AI運用が独立した設計成果物として提示できる
- **フレームワークの汎用性を示せる**: 経費精算SaaSに依存しない形で切り出すことで「他のプロジェクトにも適用できる」という説得力が生まれる
- **プロダクトリポジトリが実務的になる**: 入れ子構造が解消され、一般的なWebアプリケーションリポジトリの構成に近づく
- **Git履歴の意図が明確になる**: 各リポジトリのコミットがそのリポジトリの関心事に限定される

## 影響・結果

- **移行作業が発生する**: ファイルの移動、参照パスの更新、CLAUDE.mdの分割・再設計が必要
- **CLAUDE.mdはroot-projectに一元管理**: Claude Codeの起動がroot-projectのため、実行基盤（CLAUDE.md, .claude/）は統括側に置く
- **セッションログ・日報の運用変更**: コミット前のセッションログ記録先が dev-journal に移るため、運用フローの調整が必要
- **既存のルール・参照パスの大幅な書き換え**: CLAUDE.md、rules/内の相互参照、directory-structure.md 等の更新が必要
