# ADR-002: AI運用設計の抜本改善

## ステータス
承認済

## 日付
2026-03-08

## 背景（ペイン）

ポートフォリオの最大の訴求ポイントは「AI駆動開発の設計力」だが、現在のAI運用設計は静的な構成（CLAUDE.md → ルール参照 → 指示された作業を実行）に留まっており、上級者レベルのAIオーケストレーションとは程遠い。

### 現状の問題点

1. **タスク駆動が人間依存**: AIが自律的に次タスクを判断しない。毎回ユーザーが「次は何をやるか」を指示している
2. **品質ゲートが「ルール参照」止まり**: ルールは書いてあるが、自動チェック→自動修正のループがない。ルール違反は人間が気づくまで放置される
3. **Hooks未使用**: Claude Codeの最も強力な自動化機能（PreToolUse / PostToolUse / Stop）が一切活用されていない
4. **サブエージェント並列処理なし**: レビューや検証が全て直列実行。複数観点の同時チェックができない
5. **自己改善ループなし**: 過去の判断ミス・手戻りから学習する仕組みがない。同じ失敗を繰り返すリスクがある
6. **完了条件の自動判定なし**: progress.mdの状態遷移が手動。ステップの完了判定をAIに任せられない

### 現状のアーキテクチャ（改善前）
```
人間がタスク指示 → AI が CLAUDE.md 読み込み → rules/ 参照
→ 指示された作業を実行 → 人間が結果確認 → 人間が次を指示
```

## 検討した選択肢

### 案A: 現状維持（ルール追加のみ）
- **概要**: rules/ にルールを追加し、CLAUDE.mdの参照指示を充実させる
- **利点**: 実装コストゼロ
- **欠点**: 根本的に「静的な構成」から脱却できない。ポートフォリオとしての訴求力向上はない

### 案B: Hooks + オーケストレーションコマンドの段階的導入
- **概要**: まずHooksでイベント駆動の自動化を実現し、次にオーケストレーションコマンドで自律的タスク管理を導入する
- **利点**: 段階的に導入でき、各段階で効果を確認できる。Hooksは低コスト高インパクト
- **欠点**: 全体の完成まで時間がかかる

### 案C: フルオーケストレーション一括導入
- **概要**: Hooks、サブエージェント、自己改善ループ、完了条件自動判定を一度に導入
- **利点**: 完成形が一度に手に入る
- **欠点**: 実装コストが非常に高い。未検証の仕組みを同時に入れるリスク

## 決定

**案B: 段階的導入**を採用する。以下の優先度順で導入する。

### 導入ロードマップ

#### Phase 1: Hooks（最優先・低コスト高インパクト）
- **PreToolUse**: ファイル変更時のスコープ逸脱チェック（MVP外の機能実装を防止）
- **PostToolUse**: Bash実行後の自動検証（テスト実行結果の自動解析など）
- **Stop**: セッション終了時のprogress.md更新確認・セッションログ記録漏れ防止

#### Phase 2: オーケストレーションコマンド
- 完了条件を自動検証 → 未達タスクの分解 → 優先度順に提案 → ブロッカーは自動起票
- セッション開始時の現状分析の自動化（progress.md + issues/ + 直近ログの横断読み込み）

#### Phase 3: サブエージェント並列パターン
- レビュー時にアーキテクチャ / セキュリティ / コーディング規約 / テスト方針を並列チェック
- 実装時の影響範囲分析とテスト対象の自動特定を並列実行

#### Phase 4: 自己改善ループ
- セッション終了時に判断ミス・手戻りを自動抽出 → memory/patterns.md に記録 → 次回セッションで自動参照
- 繰り返し発生するパターンの検出と対策の蓄積

#### Phase 5: 完了条件の機械可読化
- 各ステップの完了条件をYAML等で定義 → AIが自動判定 → ステップ遷移を半自動化

### 目指すアーキテクチャ（改善後）
```
人間がゴール設定 → AI が状態分析・タスク分解・優先度判断
→ サブエージェント並列委任 → Hooks で自動品質ゲート
→ 完了条件自動判定 → 次タスク自律提案 → 人間は承認 or 方向修正のみ
```

## 理由

- **段階的に効果を確認できる**: 各Phaseが独立して価値を持つため、途中でも成果が見える
- **Hooksは投資対効果が最も高い**: 設定ファイルとシェルスクリプトだけで実現でき、即座にイベント駆動の自動化が手に入る
- **リスクが低い**: 各Phase単独で導入・検証・ロールバックが可能
- **ポートフォリオとして段階的な改善の過程を見せられる**: 「最初からフル装備」ではなく「課題を発見→段階的に改善」という実務的なアプローチを示せる

## 影響・結果

- **ai-dev-frameworkリポジトリ（ADR-001）にHooks定義・スクリプト・オーケストレーション設計が配置される**: リポジトリ分離と並行して進める
- **既存のルール体系は維持しつつ拡張**: rules/ の内容は変えず、Hooksによる自動適用レイヤーを上に乗せる
- **セッションの流れが変わる**: 開始時の自動分析、作業中の自動チェック、終了時の自動記録が入ることで、人間の介入ポイントが減る
- **Phase 2以降は設計の具体化が必要**: オーケストレーションコマンドやサブエージェントパターンの詳細設計は、Phase 1の運用実績を踏まえて行う
