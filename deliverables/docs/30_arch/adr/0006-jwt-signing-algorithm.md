# ADR-0006: JWT 署名アルゴリズム選定

## 背景

本システムはマルチテナント経費精算 SaaS であり、API 認証に JWT を使用する。アクセストークン（有効期限 15分）とリフレッシュトークン（同 7日）を発行し、全 API リクエストで署名検証を行う。

JWT の署名アルゴリズムは、トークンの改ざん防止と認証基盤の信頼性を担う。選定にあたり、以下の観点で判断する必要がある:

- セキュリティ強度（暗号標準の動向を含む）
- Go エコシステムでの実装成熟度
- 署名サイズ（毎リクエストの通信量への影響）
- プロジェクトの性質と想定ライフサイクル

なお、NIST は耐量子移行ガイドライン（IR 8547、2024年11月ドラフト）にて、RSA 2048 を 2030年に非推奨、2035年に使用禁止とする方針を示している。本 ADR ではこの動向も考慮した上で判断する。

## 検討した選択肢

### 案A: EdDSA (Ed25519)

- 署名サイズが小さい（64 bytes、RS256 の 256 bytes に対して 1/4）
- 署名生成・検証が高速
- 決定的署名（乱数不要）でパディング不要のため、実装ミスの余地が少ない
- golang-jwt で標準サポート済み
- **不採用理由**: 量子耐性がない点は RS256 と同じ。NIST IR 8547 の 2030年非推奨対象は 112bit セキュリティ強度の古典暗号全般であり、EdDSA（128bit）も同様に非推奨となる見込み。「2030年非推奨」を理由に RS256 から移行するのであれば、EdDSA では論理的に一貫しない。署名サイズや速度の改善だけでは、本プロジェクトの規模で移行コストに見合わない

### 案B: ML-DSA (CRYSTALS-Dilithium, FIPS 204)

- NIST 標準化済みの耐量子署名アルゴリズム
- 量子コンピュータに対する長期的な耐性がある
- **不採用理由**: Go エコシステムでの実装が未成熟で、golang-jwt へのカスタム SigningMethod 登録が必要。署名サイズが RS256 の 4〜6倍（ML-DSA-44 で約 2.4KB）で毎リクエストの通信量が増加する。未成熟な技術を採用するリスクが高い

### 案C: RS256（現行維持）

- golang-jwt で標準サポート。Go の JWT ライブラリとしてデファクト
- 設計・実装の変更が不要

## 決定

**RS256 を維持する。** 設計・コードの変更は行わない。

## 理由

1. **NIST 非推奨は予防的ガイドラインである**: NIST IR 8547 の 2030年非推奨は、量子コンピュータの将来的な脅威に備えた予防的措置。現在の量子コンピュータは約 1,000 量子ビット台であり、RSA 2048 を破るには数百万量子ビットが必要。少なくとも今後 5〜10年で RSA 2048 が実際に破られるリスクは低い
2. **本プロジェクトの想定ライフサイクルで実リスクがない**: ポートフォリオ兼デモ用途のプロジェクトであり、長期運用を前提としていない
3. **実装リスクが最も低い**: golang-jwt の RS256 は広く使われており、現行コードは alg 固定検証（alg 混乱攻撃の防止）も実装済み
4. **代替案に移行する合理的根拠がない**: EdDSA は量子耐性の観点で RS256 と同等の問題を抱え、ML-DSA はエコシステムが未成熟

## 影響・結果

- security.md §2.1（RS256 鍵管理）、`jwt.go` の RS256 固定実装は変更不要
- テストコード（Step 9）は RS256 前提で実装してよい
- 将来の変更容易性: 署名アルゴリズムの実装は `internal/pkg/jwt/jwt.go` に集約されており、鍵の種類・SigningMethod・alg 検証の3箇所を変更すれば切り替え可能。golang-jwt は EdDSA, ECDSA 等を標準サポートしているため、将来の移行時に大規模な改修は不要
- ML-DSA の Go エコシステムが成熟し、かつプロジェクトが長期運用に移行する場合は再検討する
