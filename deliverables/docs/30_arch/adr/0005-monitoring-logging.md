# ADR-0005: 監視・ログ戦略

## ステータス
承認済

## 日付
2026-03-16

## 背景（ペイン）
- 本番環境で問題発生時に原因を特定できる仕組みが必要
- 非機能要件: API p95 500ms以下（requirements.md §4.1）、稼働率 99.5%（§4.3）、ヘルスチェックエンドポイント（§4.3）
- ADR-0004 で AWS（ECS Fargate + RDS）を採用しており、監視基盤はAWSサービスとの親和性が重要
- MVP フェーズのため、最小限の監視で「問題に気づける」ことを優先し、高度な可観測性は段階的に構築する

## 検討した選択肢

### ログフレームワーク

#### 案A: log/slog（Go 標準ライブラリ）
- **概要**: Go 1.21 で導入された構造化ログの標準パッケージ
- **利点**: 外部依存ゼロ、Go の標準として長期サポートが保証、JSONHandler で構造化出力が可能
- **欠点**: zerolog 等と比較してパフォーマンスがやや劣る（MVP 規模では誤差）、カスタマイズ性が限定的

#### 案B: rs/zerolog
- **概要**: ゼロアロケーションの高性能構造化ログライブラリ
- **利点**: 最高レベルのパフォーマンス、チェーンAPI が使いやすい
- **欠点**: 外部依存の追加、標準化が不十分（独自API）

#### 案C: uber-go/zap
- **概要**: Uber 開発の高性能構造化ログライブラリ
- **利点**: 高性能、豊富な実績
- **欠点**: APIが複雑（Sugar/Logger の二重構造）、外部依存

### メトリクス収集

#### 案A: Amazon CloudWatch（ECS 統合）
- **概要**: ECS Fargate のメトリクスを CloudWatch で自動収集 + カスタムメトリクス
- **利点**: ECS との統合がネイティブ（CPU/メモリ/タスク数は自動収集）、追加インフラ不要、アラーム機能組み込み
- **欠点**: カスタムメトリクスの送信にはEMF（Embedded Metric Format）か PutMetricData API が必要、Grafana と比較してダッシュボードの柔軟性が低い

#### 案B: Prometheus + Grafana
- **概要**: Prometheus でメトリクス収集、Grafana で可視化
- **利点**: 高い柔軟性、リッチなダッシュボード
- **欠点**: Prometheus サーバーの運用が必要（Amazon Managed Prometheus は高コスト）、MVP フェーズではインフラ管理が過剰

## 決定

### ログ

| 項目 | 選定 |
|------|------|
| フレームワーク | log/slog（Go 標準） |
| 出力形式 | JSON（構造化ログ） |
| 出力先 | stdout → CloudWatch Logs（ECS が自動転送） |
| ログレベル | DEBUG / INFO / WARN / ERROR |
| ログレベル設定 | 環境変数 `LOG_LEVEL` で制御（本番: INFO、開発: DEBUG） |

### ログに含める標準フィールド

| フィールド | 説明 | 用途 |
|-----------|------|------|
| timestamp | ISO 8601 (UTC) | 時系列分析 |
| level | ログレベル | フィルタリング |
| msg | メッセージ | 内容把握 |
| request_id | リクエスト固有ID（UUID） | リクエストトレース |
| tenant_id | テナントID | テナント別の問題特定 |
| user_id | ユーザーID | ユーザー別の問題特定 |
| method | HTTP メソッド | アクセスログ |
| path | リクエストパス | アクセスログ |
| status | HTTP ステータスコード | エラー分析 |
| duration_ms | レスポンスタイム（ms） | パフォーマンス分析 |
| error | エラー詳細（エラー時のみ） | デバッグ |

### ログレベル方針

| レベル | 用途 | 例 |
|--------|------|-----|
| DEBUG | 開発時の詳細情報 | SQLクエリ、リクエストボディ |
| INFO | 通常の業務イベント | リクエスト完了、状態遷移実行、ファイルアップロード完了 |
| WARN | 異常だが処理続行可能 | レート制限接近、リトライ発生 |
| ERROR | 処理失敗 | DB接続エラー、外部サービスエラー、予期せぬパニック |

### セキュリティ上の注意

ログに以下を **出力しない**:
- パスワード（平文・ハッシュ含む）
- JWT トークン（アクセス・リフレッシュ）
- パスワードリセットトークン
- リクエストボディの個人情報（DEBUG レベルでもマスク処理を行う）

### メトリクス

| 項目 | 選定 |
|------|------|
| 基盤 | Amazon CloudWatch |
| 自動収集 | ECS タスクの CPU/メモリ使用率（CloudWatch Container Insights） |
| カスタムメトリクス（ダッシュボード用） | CloudWatch Logs Insights でアドホック集計 |
| カスタムメトリクス（アラート用） | CloudWatch メトリクスフィルタで時系列メトリクスを生成 |

### メトリクス生成方式の使い分け

| 用途 | 方式 | 説明 |
|------|------|------|
| ダッシュボード | Logs Insights | アドホッククエリで柔軟に集計。アラームのソースにはならない |
| CloudWatch Alarm | メトリクスフィルタ | 構造化ログからパターンマッチで時系列メトリクスを自動生成。Alarm のソースとして使用 |

**メトリクスフィルタの定義例:**
- `5xx エラー`: `{ $.status >= 500 }` → メトリクス `API/5xxCount`
- `リクエスト数`: `{ $.status >= 200 }` → メトリクス `API/RequestCount`
- `レスポンスタイム`: `{ $.duration_ms >= 0 }` → メトリクス `API/Duration`（統計値で p95/p99 を算出）

### MVP で計測すべき指標

| カテゴリ | 指標 | 取得方法 | アラート閾値 |
|----------|------|----------|-------------|
| パフォーマンス | API レスポンスタイム p95 | メトリクスフィルタ → CloudWatch Alarm（p95 統計） | > 500ms |
| パフォーマンス | API レスポンスタイム p99 | メトリクスフィルタ → CloudWatch Alarm（p99 統計） | > 1000ms（参考値） |
| エラー | 5xx エラーレート | メトリクスフィルタ → CloudWatch Alarm（Math: 5xxCount/RequestCount） | > 1% |
| エラー | 4xx エラーレート | 構造化ログの status >= 400 を集計 | > 10%（参考値） |
| 可用性 | ヘルスチェック失敗 | ALB のヘルスチェック | 連続2回失敗 |
| リソース | ECS タスク CPU 使用率 | Container Insights 自動収集 | > 80% |
| リソース | ECS タスク メモリ使用率 | Container Insights 自動収集 | > 80% |
| リソース | RDS CPU 使用率 | RDS 標準メトリクス | > 80% |
| リソース | RDS 空きストレージ | RDS 標準メトリクス | < 20% |
| リソース | RDS 接続数 | RDS 標準メトリクス | > 80% of max_connections |

### ヘルスチェック

```
GET /health

成功時 (200):
{
  "status": "ok",
  "checks": {
    "database": "ok"
  }
}

失敗時 (503):
{
  "status": "degraded",
  "checks": {
    "database": "error"
  }
}
```

- ALB のヘルスチェックターゲットとして `/health` を使用
- DB 接続確認を含む（`SELECT 1` の実行）
- ヘルスチェック自体の応答タイムアウト: 5秒

### アラート通知

| 重要度 | 通知先 | 対象 |
|--------|--------|------|
| Critical | SNS → メール | ヘルスチェック連続失敗、5xx > 5% |
| Warning | SNS → メール | p95 > 500ms、CPU > 80%、ディスク残量 < 20% |

MVP ではメール通知。将来的に Slack/PagerDuty 連携を検討。

## 理由
1. **log/slog**: 外部依存ゼロで保守コスト最小。Go 標準としての長期安定性。構造化ログ（JSON）を stdout に出力し、ECS → CloudWatch Logs の自動転送で追加設定不要
2. **CloudWatch**: ECS/RDS のメトリクスがネイティブに収集され、追加のインフラ管理が不要。Prometheus は MVP フェーズでは運用過剰。Logs Insights でカスタムメトリクスをクエリで取得でき、ダッシュボードも作成可能
3. **構造化ログベースのメトリクス**: 専用のメトリクス SDK を導入せず、構造化ログに含まれる duration_ms, status 等を活用する。ダッシュボード用途は Logs Insights のアドホッククエリ、アラート用途は CloudWatch メトリクスフィルタで時系列メトリクスを生成し CloudWatch Alarm に接続する。ログとメトリクスの一元管理が可能

## 影響・結果
- 全 HTTP ハンドラにログミドルウェアを適用し、request_id, tenant_id, duration_ms 等を自動記録する
- request_id はミドルウェアで生成し、レスポンスヘッダー（`X-Request-ID`）にも含める（クライアントからの問い合わせ時に追跡可能）
- ログの保持期間は CloudWatch Logs の設定で管理（MVP: 30日間）
- 将来的に OpenTelemetry への移行を阻害しない設計にする（slog のハンドラを差し替え可能）
- CloudWatch ダッシュボードの構築は Step 7 Phase 2 で実施
