## 背景（ペイン）
- 経費精算SaaS はマルチテナント（複数企業が同一システムを利用）で提供する
- テナント間のデータ分離は最重要の不変条件（TNT-005: テナント間のデータ参照は一切不可）
- ドメインモデルでは Tenant を起点にすべての業務エンティティ（ExpenseReport, ExpenseItem, Attachment）が tenant_id を保持する構造になっている
- テナント分離方式の選定は、コスト・運用性・セキュリティ・スケーラビリティのトレードオフ判断が必要

## 検討した選択肢

### 案A: DB per tenant（テナントごとにデータベースを分離）
- **概要**: テナントごとに独立した PostgreSQL データベースを作成
- **利点**: 物理的な完全分離により最強のセキュリティ、テナント単位のバックアップ・リストアが容易、テナント単位のスケーリングが可能
- **欠点**: テナント数に比例してDB接続数・管理コストが増大（RDS インスタンス or DB 数の爆発）、マイグレーションを全DBに適用する運用が複雑、MVP の100ユーザー規模では過剰

### 案B: Schema per tenant（テナントごとにスキーマを分離）
- **概要**: 同一 PostgreSQL インスタンス内でテナントごとに別スキーマを作成
- **利点**: DB は1つで管理集約しつつ論理的な分離を実現、テナント単位のデータ操作がスキーマ切替で可能
- **欠点**: テナント追加時にスキーマ作成 + マイグレーション適用が必要（動的DDLの運用負荷）、`search_path` の切替ミスがテナント越えに直結、接続プーリングとの相性が悪い

### 案C: Shared DB + tenant_id（共有DB + 行レベル分離）
- **概要**: 全テナントが同一データベース・同一テーブルを共有し、各行の `tenant_id` カラムで分離
- **利点**: 管理するDBは1つ（RDS 1インスタンス）でコスト最小、マイグレーションは1回で全テナントに適用、PostgreSQL RLS で行レベルの強制分離が可能、接続プーリングが素直に動作
- **欠点**: クエリに `WHERE tenant_id = ?` の漏れがテナント越えに直結（→ RLS で緩和）、テナント単位のバックアップ・リストアが複雑、将来的な大規模テナントのデータ量偏りへの対処が必要

## 決定
**案C: Shared DB + tenant_id** を採用する。

アプリケーション層の `WHERE tenant_id` と PostgreSQL RLS の二重保証でセキュリティを担保する。

## 理由
1. **MVP 規模との整合**: 同時接続100ユーザー（requirements.md §4.1）の規模では、DB per tenant / Schema per tenant の運用コストが見合わない
2. **二重保証の実現**: Shared DB でも RLS を組み合わせることで、アプリケーション層のバグ（WHERE句漏れ）があってもDB層でテナント越えを防止できる（TNT-004）
3. **運用のシンプルさ**: マイグレーション・バックアップ・監視が1DBで完結する。MVP フェーズでは運用工数を最小化したい
4. **ドメインモデルとの整合**: domain_model.md で定義された tenant_id の冗長保持（ExpenseItem, Attachment）は、Shared DB + RLS を前提とした設計

### リスクと緩和策

| リスク | 緩和策 |
|--------|--------|
| WHERE tenant_id 漏れ | RLS で二重保証（ADR-0003 で詳細化）。リポジトリ層で tenant_id フィルタを強制（TNT-003） |
| 大規模テナントのデータ偏り | tenant_id を含む複合インデックスで対処。MVP 規模では問題にならない |
| テナント単位のバックアップ不可 | MVP では RDS 自動バックアップ（全体単位）で対応（requirements.md §4.3） |
| ノイジーネイバー問題 | MVP 規模では発生しない。将来的にはコネクションプールのテナント別制御で対処可能 |

## 影響・結果
- 全業務テーブルに `tenant_id` カラムが必須（TNT-001）。User テーブルは TenantMembership を介して関連するため例外
- リポジトリ層は全クエリに `WHERE tenant_id = ?` を含む設計が必須（TNT-002）
- RLS ポリシーの詳細設計は ADR-0003 で定義する
- テナント境界越えアクセスは 404 Not Found を返却し、リソースの存在を漏洩しない（TNT-006）
