## 背景（ペイン）
- マルチテナント型経費精算SaaS の MVP を構築するにあたり、バックエンド・フロントエンド・データベース・主要ライブラリの技術選定が必要
- 選定にあたっては、非機能要件（API p95 500ms以下、同時接続100ユーザー）、セキュリティ要件（JWT RS256、Argon2id、RLS）、およびチームの技術的親和性を考慮する必要がある

## 検討した選択肢

### バックエンド

#### 案A: Go
- **概要**: 静的型付け、コンパイル言語。標準ライブラリが充実
- **利点**: 高い並行処理性能、シンプルな言語仕様、バイナリデプロイで依存少、コンテナイメージが軽量（ECS Fargate のコスト最適化に有利）
- **欠点**: ジェネリクスの成熟度がやや低い、ORM エコシステムが他言語比で小規模

#### 案B: Node.js (TypeScript)
- **概要**: フロントエンドと言語統一が可能
- **利点**: フロント/バックで TypeScript 統一、npm エコシステムが豊富
- **欠点**: シングルスレッドモデルで CPU バウンド処理に弱い（Argon2id 等）、ランタイム依存

#### 案C: Rust
- **概要**: 高性能・メモリ安全
- **利点**: 最高レベルの実行性能とメモリ安全性
- **欠点**: 学習コストが高い、コンパイル時間が長い、MVP の開発速度を阻害するリスク

### フロントエンド

#### 案A: React (TypeScript) + Vite
- **概要**: コンポーネントベースの UI ライブラリ + 高速ビルドツール
- **利点**: 豊富なエコシステム、TypeScript との親和性、Vite による高速な開発体験
- **欠点**: ライブラリ選定の自由度が高い分、判断コストがある

#### 案B: Next.js
- **概要**: React ベースのフルスタックフレームワーク
- **利点**: SSR/SSG 対応、ルーティング組み込み
- **欠点**: バックエンドを Go で構築するため SSR の恩恵が薄い、フレームワーク自体の複雑度が高い

#### 案C: Vue.js
- **概要**: 漸進的フレームワーク
- **利点**: 学習コストが低い、テンプレート構文が直感的
- **欠点**: React と比較してエコシステムが小さい、TypeScript サポートの成熟度がやや低い

### データベース

#### 案A: PostgreSQL
- **概要**: オープンソースの RDBMS。RLS（Row Level Security）をネイティブサポート
- **利点**: RLS でテナント分離をDB層で二重保証可能、JSONB・UUID 型のネイティブサポート、AWS RDS で運用実績豊富
- **欠点**: NoSQL と比較して水平スケーリングが複雑（MVP 規模では問題なし）

#### 案B: MySQL
- **概要**: 広く使われている RDBMS
- **利点**: 導入実績が豊富、学習リソースが多い
- **欠点**: RLS が未対応（テナント分離の二重保証にアプリケーション層の追加実装が必要）

### 主要ライブラリ

| 領域 | 選定 | 理由 |
|------|------|------|
| HTTP ルーター | Chi | 標準 net/http 互換、ミドルウェアチェーンがシンプル |
| DB アクセス | sqlc | SQL ファーストで型安全なコード生成、RLS との相性が良い |
| マイグレーション | golang-migrate | SQL ベースでバージョン管理、RLS ポリシー定義もマイグレーションで管理可能 |
| JWT | golang-jwt/jwt | RS256 対応、Go の JWT ライブラリとしてデファクト |
| パスワードハッシュ | alexedwards/argon2id | Argon2id のラッパー、OWASP 推奨パラメータ対応 |
| バリデーション | go-playground/validator | 構造体タグベースのバリデーション |
| AWS SDK | aws-sdk-go-v2 | S3 署名付きURL発行、公式 SDK |
| テスト | 標準 testing + testify | アサーションヘルパーで可読性向上 |
| フロントエンド状態管理 | TanStack Query | サーバー状態管理に特化、キャッシュ・再取得の自動化 |
| フロントエンドルーティング | React Router | React エコシステムのデファクト |
| HTTP クライアント（フロント） | ky / fetch ラッパー | 軽量、インターセプターでJWT自動付与 |
| UI コンポーネント | MUI (Material UI) | 業務アプリ向けの豊富なコンポーネント群、テーマによるデザイン一貫性、DataGrid 等の高機能コンポーネント |

## 決定
- **バックエンド**: Go
- **フロントエンド**: React (TypeScript) + Vite
- **データベース**: PostgreSQL
- **主要ライブラリ**: 上記表の通り

## 理由
- **Go**: p95 500ms 以下のパフォーマンス目標を余裕で達成可能。Argon2id のようなCPUバウンド処理もゴルーチンで効率的に並行処理できる。コンテナイメージの軽量さは ECS Fargate のコスト最適化に直結する
- **React + Vite**: MVP で必要な画面（ログイン・ダッシュボード・経費管理・承認）をコンポーネント指向で効率的に構築できる。Vite の HMR で開発速度を確保
- **PostgreSQL**: テナント分離の二重保証（TNT-004）を実現する唯一の選択肢。RLS はアーキテクチャ要件の根幹であり、これを欠くと追加のセキュリティ層をアプリケーションで構築する必要がある
- **sqlc**: SQL を直接書いて型安全なGoコードを生成する方式は、RLS の `SET app.current_tenant` やカスタムクエリとの相性が良い。ORM の暗黙的なクエリ生成による RLS バイパスリスクを排除

## 影響・結果
- Go + PostgreSQL の組み合わせにより、バックエンドはシングルバイナリ + RDS の構成でシンプルに運用可能
- sqlc の採用により、SQL マイグレーションとクエリ定義が一元的に管理される。スキーマ変更時のコンパイルエラーで不整合を早期検出できる
- フロントエンドとバックエンドの言語が異なるため、API スキーマ（OpenAPI）を契約として明確に定義する必要がある（Step 4+5 で対応）
- MUI の採用により、テーマ設定でデザインの一貫性を担保できる。DataGrid・Dialog・AppBar 等の業務アプリ向けコンポーネントにより、UI 実装の工数を削減
