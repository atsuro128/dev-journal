# ADR-0004: インフラ選定

## この文書の役割

| 項目 | 内容 |
|------|------|
| 目的 | コンピュート・DB・ストレージのインフラ構成の選定根拠を記録する |
| 正本情報 | インフラ構成の判断対象、選択肢、決定、理由、トレードオフ |
| 扱わない内容 | 監視・ログ方式（ADR-0005 に委譲）、CI/CD パイプライン詳細 |
| 主な参照元 | `../../10_requirements/requirements.md` |
| 主な参照先 | `../architecture.md`, `../diagrams.md`, `../../70_operations/*` |

## 背景
- 経費精算SaaS の本番環境インフラを選定する必要がある
- 非機能要件: 稼働率 99.5%（requirements.md §4.3）、同時接続100ユーザー（§4.1）、API p95 500ms以下（§4.1）
- 添付ファイル（領収書）のオブジェクトストレージが必要（ATT-014: S3パスにテナントIDを含む）
- MVP フェーズのため、運用コストと管理工数を最小化したい

## 検討した選択肢

### コンピュート

#### 案A: AWS ECS Fargate
- **概要**: サーバーレスコンテナ実行環境。EC2 インスタンスの管理不要
- **利点**: サーバー管理不要、タスク単位のスケーリング、Go バイナリの軽量コンテナで起動が速い、ALB 連携でヘルスチェック・ロードバランシング対応
- **欠点**: EC2 比でコスト単価が高い（MVP 規模では誤差）、コールドスタートが発生しうる（Go の起動は高速なため影響軽微）

#### 案B: AWS EC2
- **概要**: 仮想マシンを直接管理
- **利点**: コスト単価が安い、完全な制御が可能
- **欠点**: OS パッチ・セキュリティ更新の運用が発生、スケーリング設定が複雑、MVP フェーズに不相応な管理工数

#### 案C: AWS Lambda
- **概要**: イベント駆動のサーバーレスコンピュート
- **利点**: リクエスト課金で低トラフィック時のコスト最小、スケーリングが自動
- **欠点**: Go の HTTP サーバーをLambda用にアダプターが必要、コールドスタートの影響、RDS Proxy が必要（接続数管理）、長時間リクエスト（ファイルアップロード）に制約

### データベース

#### 案A: Amazon RDS for PostgreSQL
- **概要**: マネージド PostgreSQL。自動バックアップ・パッチ適用・フェイルオーバー対応
- **利点**: RLS サポート、自動バックアップ（7日間保持で §4.3 充足）、Multi-AZ で可用性確保、Performance Insights で監視
- **欠点**: Aurora 比でスケーリング上限が低い（MVP 規模では問題なし）

#### 案B: Amazon Aurora PostgreSQL
- **概要**: AWS 独自の PostgreSQL 互換エンジン
- **利点**: RDS より高いスループットと可用性、ストレージの自動拡張
- **欠点**: RDS 比で約2倍のコスト、MVP 規模では性能過剰

### オブジェクトストレージ

**Amazon S3** 一択。領収書ファイル（JPEG/PNG/PDF、最大5MB）の保存先として、署名付きURLによるアクセス制御（ATT-010〜012）と整合する。他の選択肢（EFS 等）は用途に対して過剰。

### IaC（Infrastructure as Code）

#### 案A: Terraform
- **概要**: 宣言的なインフラ定義ツール。HCL で記述
- **利点**: マルチクラウド対応、豊富なプロバイダー、状態管理が明確
- **欠点**: HCL の学習コスト（ただし広く普及しており情報は豊富）

#### 案B: AWS CDK
- **概要**: TypeScript/Python 等のプログラミング言語でインフラを定義
- **利点**: TypeScript でフロントエンドと言語統一が可能、プログラマブルな抽象化
- **欠点**: AWS ロックイン、CloudFormation への変換レイヤーが複雑さを生む

## 決定

| 領域 | 選定 | 構成 |
|------|------|------|
| コンピュート | ECS Fargate | ALB → Fargate タスク（Go バイナリ + SPA 静的ファイル embed） |
| データベース | RDS for PostgreSQL | db.t3.micro（MVP）、Multi-AZ は初期は無効（コスト優先）、自動バックアップ7日間 |
| オブジェクトストレージ | S3 | 領収書保存、署名付きURLでアクセス制御 |
| IaC | Terraform | AWS プロバイダーで全リソース管理 |
| CI/CD | GitHub Actions | lint → test → build → deploy のパイプライン |

### SPA 配信方式

| 方式 | 採用 | 理由 |
|------|------|------|
| Go embed で同一コンテナから配信 | **採用** | コンテナ1つで完結、デプロイが最もシンプル、MVP に最適 |
| S3 + CloudFront | 不採用 | CDN 設定・CORS・キャッシュ制御の追加コストが発生する。将来スケール時に移行 |
| フロントエンド専用コンテナ | 不採用 | コンテナ管理コストが2倍になり、MVP では過剰 |

### 環境構成

| 環境 | 用途 | 構成 |
|------|------|------|
| ローカル | 開発 | Docker Compose（Go + PostgreSQL + MinIO） |
| ステージング | 検証 | ECS Fargate + RDS（本番同等構成、小スペック） |
| 本番 | 運用 | ECS Fargate + RDS + S3 |

## 理由
1. **ECS Fargate**: サーバー管理不要で MVP フェーズの運用工数を最小化。Go のバイナリは軽量コンテナ（Alpine ベース）で起動が速く、Fargate の弱点（コールドスタート）を緩和。Vite build の静的ファイルを `go:embed` でバイナリに同梱し、API と SPA を単一コンテナで配信する（MVP のシンプルさを優先。将来のスケール時に S3+CloudFront へ移行可能）
2. **RDS for PostgreSQL**: RLS 対応のマネージド PostgreSQL として最適。Aurora は MVP 規模でコスト過剰。自動バックアップ7日間保持で §4.3 のバックアップ要件を充足
3. **S3**: 署名付きURL（15分有効、ATT-012）によるアクセス制御がネイティブ対応。パスに tenant_id を含めることで（ATT-014）、バケットポリシーでの追加の分離も将来的に可能
4. **Terraform**: インフラをコードで管理し、環境の再現性を保証。GitHub Actions との連携でインフラの変更もCIで管理可能
5. **GitHub Actions**: リポジトリと同じプラットフォームでCI/CDを完結。Go のビルド・テスト・Docker イメージの構築・ECS デプロイまでのパイプラインを構築

### コスト概算（MVP 月額）

| リソース | スペック | 概算（USD/月） |
|----------|---------|--------------|
| ECS Fargate | 0.25 vCPU / 0.5GB × 2タスク | ~$15 |
| RDS PostgreSQL | db.t3.micro、Single-AZ | ~$15 |
| ALB | 1 ALB | ~$20 |
| S3 | 10GB 以下 | ~$1 |
| その他（NAT Gateway 等） | - | ~$35 |
| **合計** | | **~$86** |

## ポートフォリオ対応

本 ADR の構成は実務を想定した設計である。本プロジェクトはポートフォリオ用途のため、実際のデプロイでは AWS 無料枠内に収める。

| 領域 | 実務想定（本 ADR） | ポートフォリオ実装 | 理由 |
|------|-------------------|-------------------|------|
| コンピュート | ECS Fargate | EC2 t3.micro | 無料枠（12ヶ月 750時間/月） |
| DB | RDS db.t3.micro | RDS db.t3.micro | 無料枠（12ヶ月 750時間/月）。変更なし |
| ストレージ | S3 | S3 | 無料枠（5GB）。変更なし |
| ロードバランサー | ALB | ALB | 無料枠（12ヶ月 750時間/月）。変更なし |

- アプリケーションコードへの影響はない（同一 Docker イメージを EC2 上で実行する）
- 変更が必要なのはデプロイ先（Fargate タスク定義 → EC2 の docker run）と CI/CD の最終ステップのみ
- Terraform の定義で切り替え可能

## 影響・結果
- ローカル開発は Docker Compose で本番と同等の構成を再現する。S3 の代替として MinIO を使用
- デプロイパイプラインの実装は Step 7 Phase 1 で行う。本 ADR ではパイプラインの方針のみ定義
- Terraform の state 管理には S3 + DynamoDB（ロック）を使用する
- **可用性に関するリスク受容（MVP）**: 要件 §4.3 では 99.5% の稼働率を目標としている。AWS RDS SLA では Single-DB Instance は 99.5%、Multi-AZ は 99.95% である。Single-AZ では SLA と要件目標が同値でマージンがなく、AZ 障害時にはフェイルオーバーに数分〜十数分を要する。MVP フェーズではこのリスクを受容し、コスト（Multi-AZ は約2倍）を優先する。テナント数の増加や SLA マージンが必要になった時点で Multi-AZ を有効化する
- ALB ヘルスチェック（`/health`）は API タスクの異常検知・自動置換を担い、ECS 層の可用性を向上させる。DB 層の可用性とは別の役割であり、両者を混同しない
- 監視・ログ戦略は ADR-0005 で定義する
