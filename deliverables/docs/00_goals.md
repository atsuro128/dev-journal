# プロジェクトのゴール・評価ポイント・スコープ方針

## 概要

マルチテナント型 経費精算SaaS。
設計判断の明文化・セキュリティ・テナント分離・テスト戦略を通じて、実務レベルのエンジニアリング力を示すことを目的とする。

## 狙い

単なる「動くアプリ」ではなく、以下を通じてエンジニアリング力を示す。

1. **設計判断の言語化** — ADR（Architecture Decision Records）で技術選定の根拠を明文化
2. **業務フローの正確な実装** — 経費申請〜承認〜支払の状態遷移を厳密に制御
3. **セキュリティ・テナント分離** — マルチテナントのデータ分離をアプリ層 + RLS の二重保証で実現
4. **テスト戦略** — テナント分離・RBAC・状態遷移の重点テストで品質保証の意識を示す
5. **運用を見据えた構成** — CI/CD、構造化ログ、ヘルスチェック、監視方針の整備

## 技術的な訴求ポイント

| 評価ポイント | 本プロジェクトでの対応 |
|---|---|
| マルチテナント設計の理解 | Tenant 単位のデータ分離、全クエリでの `tenant_id` 必須化 |
| RBAC（権限設計） | Admin / Approver / Member / Accounting の4ロールと API 制御 |
| 実務的な業務フロー実装 | 経費申請〜承認〜支払の状態遷移、却下フロー |
| クラウド運用経験 | AWS デプロイ、CI/CD、運用・監視方針の明文化 |
| 英語ドキュメント対応 | 英語 README、API ドキュメント（英語） |
| テスト戦略 | テナント分離テスト、RBAC テスト、状態遷移テストの網羅 |
| 技術選定の合理性 | 各技術の採用理由を明文化（ADR形式） |

## スコープ方針

- **MVP優先**: コアの業務フロー（申請→承認→差戻し→再申請→承認→支払完了）を最優先で実装
- **段階的拡張**: 付随機能（通知・監査ログ・CSVエクスポート・招待）は Phase 3 で追加
- **品質 > 機能数**: 機能を増やすより、テナント分離・権限制御・テストの品質を重視
- **設計ドキュメント先行**: 実装前に要件・ドメイン・アーキテクチャの設計を固める
