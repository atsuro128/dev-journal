# ユースケース一覧

## 1. 概要

本書では、経費精算SaaS の全ユースケースを**アクター別**に整理する。
各ユースケースは正常系・代替系・例外系を含み、要件の網羅性を担保する。

### ユースケースID体系

| プレフィックス | アクター |
|--------------|---------|
| UC-M | Member（申請者） |
| UC-A | Approver（承認者） |
| UC-AC | Accounting（経理） |
| UC-AD | Admin（管理者） |
| UC-SYS | システム共通 |

---

## 2. Member（申請者）のユースケース

### UC-M01: 経費レポートを作成する

| 項目 | 内容 |
|------|------|
| **アクター** | Member（Approver/Admin も自分の経費を精算する場合は同様に実行可能） |
| **目的** | 業務で立て替えた経費をまとめて精算するためのレポートを作成する |
| **事前条件** | ログイン済み |
| **事後条件** | draft 状態のレポートが作成される |

**正常系**
1. Member がレポート作成画面を開く
2. タイトル（例:「2026年2月 営業経費」）を入力する
3. 対象期間（開始日・終了日）を指定する
4. 作成ボタンを押す
5. システムが draft 状態のレポートを作成する
6. レポート詳細画面に遷移し、明細追加が可能になる

**代替系**
- 2a. 過去のレポートを参考にしてタイトルを入力する
- 3a. 対象期間を月単位で簡易選択する（UIの利便性）

**例外系**
- 3e. 開始日が終了日より後 → バリデーションエラー「開始日は終了日以前を指定してください」
- 5e. システムエラー → エラーメッセージ表示、入力内容を保持

---

### UC-M02: 経費明細を追加する

| 項目 | 内容 |
|------|------|
| **アクター** | Member |
| **目的** | レポートに個別の経費明細（1件の支出）を登録する |
| **事前条件** | draft 状態のレポートが存在する |
| **事後条件** | レポートに明細が追加され、合計金額が更新される |

**正常系**
1. Member がレポート詳細画面で「明細追加」を選択する
2. 日付を入力する
3. 金額（円）を入力する
4. カテゴリ（交通費/宿泊費/飲食費/消耗品費/通信費/その他）を選択する
5. 摘要（例:「XX社訪問 タクシー代」）を入力する
6. 保存する
7. 明細がレポートに追加され、合計金額が自動計算される

**代替系**
- 1a. 複数の明細を続けて追加する（追加後に「続けて追加」を選択）

**例外系**
- 3e. 金額が0以下 → バリデーションエラー「正の金額を入力してください」
- 3e. 金額が整数でない → バリデーションエラー「円単位の整数で入力してください」
- 5e. 摘要が空 → バリデーションエラー「摘要を入力してください」
- 6e. レポートが draft 以外の状態 → エラー「提出済みのレポートには明細を追加できません」

---

### UC-M03: 領収書を添付する

| 項目 | 内容 |
|------|------|
| **アクター** | Member |
| **目的** | 経費明細の証跡として領収書画像を添付する |
| **事前条件** | draft 状態のレポートに明細が存在する |
| **事後条件** | 明細に添付ファイルが紐づく |

**正常系**
1. Member が明細の「領収書添付」を選択する
2. ファイルを選択する（JPEG / PNG / PDF）
3. システムがファイルをS3にアップロードする
4. アップロード完了を表示する

**代替系**
- 2a. カメラで撮影した画像を直接アップロードする（モバイル時）
- 2b. 複数ファイルを一度に選択してアップロードする

**例外系**
- 2e. 非対応ファイル形式（例: .doc） → エラー「JPEG, PNG, PDF のみアップロード可能です」
- 2e. ファイルサイズ超過（5MB超） → エラー「ファイルサイズは5MB以下にしてください」
- 3e. アップロード失敗（ネットワーク） → エラー表示、リトライ可能

---

### UC-M03a: 添付ファイルを削除する

| 項目 | 内容 |
|------|------|
| **アクター** | Member（Approver/Admin も自分のレポートで実行可能） |
| **目的** | 誤ってアップロードした添付ファイルを削除する |
| **事前条件** | draft 状態のレポートの明細に添付ファイルが存在する |
| **事後条件** | 添付ファイルが削除される（S3上のファイルも削除） |

**正常系**
1. Member が明細詳細画面で添付ファイル一覧を確認する
2. 削除対象のファイルを選択し「削除」を押す
3. 確認ダイアログが表示される
4. 削除を確定する
5. 添付ファイルが削除される

**例外系**
- 2e. レポートが draft 以外の状態 → エラー「提出済みのレポートの添付ファイルは削除できません」

---

### UC-M04: レポートを編集する

| 項目 | 内容 |
|------|------|
| **アクター** | Member |
| **目的** | 作成済みのレポートのタイトルや対象期間を修正する |
| **事前条件** | 自分が作成した draft 状態のレポートが存在する |
| **事後条件** | レポート情報が更新される |

**正常系**
1. Member が下書きレポート一覧から対象レポートを選択する
2. タイトルまたは対象期間を修正する
3. 保存する

**例外系**
- 1e. 他人のレポートを選択 → 403 Forbidden
- 1e. 他テナントのレポートIDを指定 → 404 Not Found（存在漏洩防止）
- 2e. submitted 以降のレポートを編集しようとする → エラー「提出済みのレポートは編集できません」

---

### UC-M05: 明細を編集・削除する

| 項目 | 内容 |
|------|------|
| **アクター** | Member |
| **目的** | 入力ミスの修正や不要な明細の削除 |
| **事前条件** | draft 状態のレポート内に明細が存在する |
| **事後条件** | 明細が更新/削除され、合計金額が再計算される |

**正常系（編集）**
1. Member が明細の編集を選択する
2. 日付・金額・カテゴリ・摘要を修正する
3. 保存する
4. 合計金額が再計算される

**正常系（削除）**
1. Member が明細の削除を選択する
2. 確認ダイアログが表示される
3. 削除を確定する
4. 明細が削除され、合計金額が再計算される
5. 紐づく添付ファイルも削除される

---

### UC-M06: レポートを提出する

| 項目 | 内容 |
|------|------|
| **アクター** | Member（Approver/Admin も自分のレポートで実行可能） |
| **目的** | 完成したレポートを承認者に提出する |
| **事前条件** | draft 状態のレポートに明細が1件以上存在する |
| **事後条件** | レポートが submitted 状態になり、Approver が承認可能になる |

**正常系**
1. Member がレポート詳細画面で「提出」を選択する
2. 確認ダイアログ「提出後は編集できなくなります。提出しますか？」が表示される
3. Member が提出を確定する
4. レポートの状態が draft → submitted に遷移する
5. 提出完了メッセージが表示される

**例外系**
- 1e. 明細が0件 → エラー「明細を1件以上追加してから提出してください」
- 1e. 同テナント内に Approver が0人 → エラー「承認者が登録されていないため提出できません」
- 4e. 状態遷移に失敗 → エラーメッセージ表示

---

### UC-M07: レポートを削除する

| 項目 | 内容 |
|------|------|
| **アクター** | Member（Approver/Admin も自分のレポートで実行可能） |
| **目的** | 不要になった下書きレポートを削除する |
| **事前条件** | 自分が作成した draft 状態のレポートが存在する |
| **事後条件** | レポートが論理削除される |

**正常系**
1. Member が下書きレポートの「削除」を選択する
2. 確認ダイアログが表示される
3. 削除を確定する
4. レポートが論理削除される（明細・添付も連動）

**例外系**
- 1e. submitted 以降のレポートの削除 → エラー「提出済みのレポートは削除できません」

---

### UC-M08: レポートの状況を確認する

| 項目 | 内容 |
|------|------|
| **アクター** | Member |
| **目的** | 提出したレポートの承認状況・支払状況を確認する |
| **事前条件** | ログイン済み |
| **事後条件** | なし（参照のみ） |

**正常系**
1. Member がレポート一覧画面を開く
2. 自分のレポート一覧がステータス付きで表示される
3. ステータス・対象期間でフィルタリングできる
4. レポートを選択して詳細を確認できる

---

### UC-M09: 却下理由を確認し再申請する

| 項目 | 内容 |
|------|------|
| **アクター** | Member |
| **目的** | 却下されたレポートの理由を確認し、修正して再申請する |
| **事前条件** | rejected 状態のレポートが存在する |
| **事後条件** | 元レポートへの参照を持つ新規レポートが作成される |

**正常系**
1. Member がレポート一覧で rejected 状態のレポートを選択する
2. 却下理由を確認する
3. 「再申請」ボタンを選択する
4. 元レポートの内容をコピーした新規レポートが draft で作成される
   - コピー対象: タイトル・対象期間・明細
   - コピー対象外: 添付ファイル（必要に応じて再アップロード）
5. 新規レポートには元レポートへの参照（reference_report_id）が設定される
6. Member が内容を修正する（添付ファイルの再アップロードを含む）
7. 修正完了後に提出する（UC-M06 に合流）

**代替系**
- 3a. 元レポートを参考にせず、完全に新規でレポートを作成する

## 3. Approver（承認者）のユースケース

### UC-A01: 承認待ちレポートを確認する

| 項目 | 内容 |
|------|------|
| **アクター** | Approver |
| **目的** | 提出されたレポートの一覧を確認し、承認作業に着手する |
| **事前条件** | ログイン済み、同テナントに submitted レポートが存在する |
| **事後条件** | なし（参照のみ） |

**正常系**
1. Approver が承認待ち一覧画面を開く
2. 同テナントの submitted レポートが一覧表示される（申請者名・タイトル・合計金額・提出日）
3. レポートを選択して詳細（明細・領収書）を確認する

---

### UC-A02: レポートを承認する

| 項目 | 内容 |
|------|------|
| **アクター** | Approver |
| **目的** | 内容に問題がないレポートを承認する |
| **事前条件** | submitted 状態のレポートが存在する |
| **事後条件** | レポートが approved 状態になる |

**正常系**
1. Approver がレポート詳細画面で内容・明細・領収書を確認する
2. 「承認」ボタンを選択する
3. 確認ダイアログが表示される
4. 承認を確定する
5. レポートの状態が submitted → approved に遷移する

**代替系**
- 4a. 承認時にコメント（メモ）を入力する（任意。例:「次回から領収書を鮮明に添付してください」）

**例外系**
- 2e. 自分が提出したレポートを承認しようとする → エラー「自分のレポートは承認できません」
- 2e. 他テナントのレポートIDを指定 → 404 Not Found（存在漏洩防止）

---

### UC-A03: レポートを却下する

| 項目 | 内容 |
|------|------|
| **アクター** | Approver |
| **目的** | 不備のあるレポートを却下する |
| **事前条件** | submitted 状態のレポートが存在する |
| **事後条件** | レポートが rejected 状態になり、却下理由が記録される |

**正常系**
1. Approver がレポート詳細画面で不備を確認する
2. 「却下」ボタンを選択する
3. 却下理由入力フォームが表示される
4. 具体的な却下理由を入力する（例:「カテゴリが不正。3件目の飲食費は交際費として申請してください」）
5. 却下を確定する
6. レポートの状態が submitted → rejected に遷移する

**例外系**
- 4e. 却下理由が空 → バリデーションエラー「却下理由を入力してください」
- 2e. 他テナントのレポートIDを指定 → 404 Not Found（存在漏洩防止）

## 6. システム共通のユースケース

### UC-SYS01: サインアップする

| 項目 | 内容 |
|------|------|
| **アクター** | 未認証ユーザー |
| **目的** | 新規アカウントとテナントを作成し、サービスの利用を開始する |
| **事前条件** | アカウントを持っていない |
| **事後条件** | ユーザーアカウントとテナントが作成され、ログイン状態になる |

**正常系**
1. ユーザーがサインアップ画面を開く
2. メールアドレス、パスワード、ユーザー名、会社名を入力する
3. サインアップボタンを押す
4. システムがテナント（企業）を新規作成する
5. ユーザーを Admin ロールで登録する
6. JWT を発行し、ダッシュボードに遷移する

**例外系**
- 2e. メールアドレスが既に使用済み → エラー「このメールアドレスは既に登録されています」
- 2e. パスワードが8文字未満 → バリデーションエラー

---

### UC-SYS02: ログインする

| 項目 | 内容 |
|------|------|
| **アクター** | 未認証ユーザー |
| **目的** | システムにログインする |
| **事前条件** | アカウントが存在する |
| **事後条件** | JWT が発行され、認証済み状態になる |

**正常系**
1. ユーザーがログイン画面を開く
2. メールアドレスとパスワードを入力する
3. ログインボタンを押す
4. 認証成功 → JWT を発行し、ダッシュボードに遷移する

**例外系**
- 4e. 認証失敗 → エラー「メールアドレスまたはパスワードが正しくありません」（ユーザー存在を推測させない）

---

### UC-SYS03: ログアウトする

| **事前条件** | ログイン済み |

---

### UC-SYS05: パスワードをリセットする

| 項目 | 内容 |
|------|------|
| **アクター** | 未認証ユーザー |
| **目的** | パスワードを忘れた場合に、新しいパスワードを設定する |
| **事前条件** | アカウントが存在する |
| **事後条件** | パスワードが更新される |

**正常系**
1. ユーザーがログイン画面で「パスワードを忘れた方」を選択する
2. メールアドレスを入力する
3. システムがリセットトークン（1時間有効）を生成し、メールで送信する
4. ユーザーがメール内のリンクからパスワードリセット画面を開く
5. 新しいパスワードを入力する（8文字以上）
6. パスワードが更新され、ログイン画面に遷移する

**例外系**
- 3e. 未登録のメールアドレス → 成功と同じレスポンスを返す（ユーザー存在漏洩防止）
- 4e. トークン期限切れ → エラー「リンクの有効期限が切れています。再度リセットを申請してください」
- 5e. パスワードが8文字未満 → バリデーションエラー

---

### UC-SYS04: ダッシュボードを確認する

| 項目 | 内容 |
|------|------|
| **アクター** | 全ロール |
| **目的** | 自分に関連する経費の概況を把握する |
| **事前条件** | ログイン済み |
| **事後条件** | なし（参照のみ） |

**正常系（Member）**
1. ログイン後にダッシュボードが表示される
2. 以下の情報が表示される:
   - 下書きレポート数
   - 提出中（承認待ち）レポート数
   - 却下されたレポート数
   - 直近のレポート一覧

**正常系（Approver）**
1. 上記に加えて:
   - 承認待ちレポート数（要対応件数）
