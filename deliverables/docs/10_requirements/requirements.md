# 要件定義書

## 1. 本書の目的と位置づけ

本書は、マルチテナント型 経費精算SaaS の機能要件・非機能要件を定義する。
事前資料（`preliminary/`）で整理した業務理解・業務ルールを基に、「何を作るか」を明確にする。

## 2. 機能要件 — コア機能（MVP）

### 2.1 認証・ユーザー管理

**アクター**: 全ロール

#### 機能一覧

| ID | 機能 | 説明 |
|----|------|------|
| AUTH-F01 | サインアップ | メールアドレス + パスワードでアカウント作成。同時にテナント（企業）を新規作成し、作成者を Admin として登録 |
| AUTH-F02 | ログイン | メールアドレス + パスワードで認証。JWT（アクセストークン + リフレッシュトークン）を発行 |
| AUTH-F03 | トークンリフレッシュ | リフレッシュトークンを用いてアクセストークンを再発行 |
| AUTH-F04 | ログアウト | リフレッシュトークンを無効化 |
| AUTH-F05 | 認証情報取得 | 現在ログイン中のユーザー情報（名前・メール・ロール・テナント）を取得 |
| AUTH-F06 | パスワードリセット | メールアドレスでリセットトークンを発行し、新パスワードを設定可能にする |

#### 入力/出力

| 機能 | 入力 | 出力 |
|------|------|------|
| サインアップ | メール、パスワード、ユーザー名、会社名 | ユーザー情報、JWT |
| ログイン | メール、パスワード | JWT（access_token, refresh_token） |
| トークンリフレッシュ | refresh_token | 新しい access_token |
| ログアウト | refresh_token | 成功/失敗 |
| パスワードリセット要求 | メールアドレス | 成功メッセージ（ユーザー存在有無に関わらず同一応答） |
| パスワードリセット実行 | リセットトークン、新パスワード | 成功/失敗 |

#### ビジネスルール

| ルールID | ルール |
|---------|--------|
| SEC-001 | 認証方式はメール + パスワード |
| SEC-002 | パスワードハッシュは Argon2id |
| SEC-003 | アクセストークン有効期間: 15分、リフレッシュトークン有効期間: 7日 |
| SEC-004 | JWT 署名アルゴリズム: RS256（非対称鍵） |
| SEC-005 | ログアウト時にリフレッシュトークンを無効化 |
| SEC-006 | パスワードリセット: リセットトークン（1時間有効）を発行。トークンは1回使用で無効化 |
| SEC-010 | パスワード最小長: 8文字以上 |
| SEC-011 | 認証失敗時のレスポンスでユーザーの存在を推測させない |

#### バリデーション

| フィールド | 制約 |
|-----------|------|
| メールアドレス | 有効なメール形式、システム全体で一意 |
| パスワード | 8文字以上 |
| ユーザー名 | 1〜100文字 |
| 会社名 | 1〜200文字 |

### 2.2 RBAC（ロールベースアクセス制御）

**アクター**: システム全体に適用

#### 機能一覧

| ID | 機能 | 説明 |
|----|------|------|
| RBAC-F01 | ロール検証ミドルウェア | 全APIリクエストでロールを検証し、権限不足なら 403 を返却 |
| RBAC-F02 | リソース所有者チェック | 申請系操作は所有者のみ可能（Member / Approver / Admin 共通） |

#### ロール定義

| ロール | 説明 | 付与タイミング |
|--------|------|--------------|
| Admin | テナント管理者。全データ閲覧・メンバー管理 | テナント作成時に自動付与 / Admin が手動付与 |
| Approver | 承認者。経費レポートの承認・却下。自分の経費レポートの作成・提出も可能 | Admin が付与 |
| Member | 一般社員。自分の経費レポートの作成・提出・編集 | Admin が付与（デフォルトロール） |
| Accounting | 経理担当。経費閲覧・支払完了記録 | Admin が付与 |

#### ビジネスルール

| ルールID | ルール |
|---------|--------|
| RBC-001 | 全APIでミドルウェアによるロール検証を実施 |
| RBC-002 | ユーザーは1テナントに対して1つのロールを持つ |
| RBC-003 | リソースアクセスは「ロール」と「所有権」の両方で制御 |
| RBC-004 | 同一テナント内の権限不足時は 403 Forbidden を返却 |

> 詳細な権限マトリクスは `rbac.md` を参照。

### 2.3 テナント分離

**アクター**: システム全体に適用（ユーザーが直接操作する機能ではない）

#### 機能一覧

| ID | 機能 | 説明 |
|----|------|------|
| TNT-F01 | tenant_id 自動付与 | データ作成時にログインユーザーの tenant_id を自動設定 |
| TNT-F02 | tenant_id フィルタ | 全クエリで tenant_id による絞り込みを自動適用 |
| TNT-F03 | RLS ポリシー | PostgreSQL の Row Level Security でDB層のテナント分離を保証 |

#### ビジネスルール

| ルールID | ルール |
|---------|--------|
| TNT-001 | 全テーブルに tenant_id カラムが必須 |
| TNT-002 | 全クエリに WHERE tenant_id = ? が必須 |
| TNT-003 | tenant_id の付与・検証はリポジトリ層で強制 |
| TNT-004 | PostgreSQL RLS でアプリ層の保証を二重化 |
| TNT-005 | テナント間のデータ参照は一切不可 |
| TNT-006 | テナント境界越えアクセスは 404 Not Found を返却（存在漏洩防止） |

### 2.4 経費レポート CRUD

**アクター**: Member/Approver/Admin（作成・編集・削除・提出は自分のレポートのみ）、Accounting（閲覧のみ）

#### 機能一覧

| ID | 機能 | 説明 |
|----|------|------|
| RPT-F01 | レポート作成 | タイトル・対象期間を指定して経費レポートを新規作成（初期状態: draft） |
| RPT-F02 | レポート一覧取得（自分） | 自分のレポート一覧を取得（フィルタ: ステータス、期間） |
| RPT-F03 | レポート詳細取得 | レポートの詳細情報（明細・添付含む）を取得 |
| RPT-F07 | テナント全レポート一覧取得 | 同テナントの全ユーザーのレポート一覧を取得（Admin: 管理目的、Accounting: 支払管理目的） |
| ADM-F01 | テナント情報取得 | 自テナントの基本情報（会社名等）を取得 |
| RPT-F04 | レポート編集 | draft 状態のレポートのタイトル・対象期間を編集 |
| RPT-F05 | レポート削除 | draft 状態のレポートを削除（論理削除） |
| RPT-F06 | レポート提出 | draft → submitted に状態遷移。明細1件以上が必要 |
| ITM-F01 | 明細追加 | レポートに経費明細（日付・金額・カテゴリ・摘要）を追加 |
| ITM-F02 | 明細編集 | draft 状態のレポート内の明細を編集 |
| ITM-F03 | 明細削除 | draft 状態のレポート内の明細を削除 |

#### 入力/出力

**レポート作成**

| 入力 | 制約 |
|------|------|
| タイトル | 必須。1〜200文字 |
| 対象期間（開始日） | 必須。日付形式 |
| 対象期間（終了日） | 必須。開始日以降の日付 |

| 出力 | 内容 |
|------|------|
| レポート情報 | id, タイトル, 対象期間, ステータス(draft), 合計金額(0), 作成日時 |

**明細追加**

| 入力 | 制約 |
|------|------|
| 日付 | 必須。日付形式 |
| 金額 | 必須。正の整数（円単位） |
| カテゴリ | 必須。6種類から選択 |
| 摘要 | 必須。1〜500文字 |

| 出力 | 内容 |
|------|------|
| 明細情報 | id, 日付, 金額, カテゴリ, 摘要, 作成日時 |

#### カテゴリ一覧

| コード | 日本語 | 英語 |
|--------|--------|------|
| transportation | 交通費 | Transportation |
| accommodation | 宿泊費 | Accommodation |
| food | 飲食費 | Food & Beverage |
| supplies | 消耗品費 | Supplies |
| communication | 通信費 | Communication |
| other | その他 | Other |

#### ビジネスルール

| ルールID | ルール |
|---------|--------|
| RPT-001 | レポートにはタイトルが必須 |
| RPT-002 | レポートには対象期間（開始日・終了日）が必須 |
| RPT-003 | 対象期間の開始日は終了日以前であること |
| RPT-006 | レポートの合計金額は明細の金額合計から自動計算 |
| RPT-011 | レポートの編集は draft 状態でのみ可能 |
| RPT-012 | レポートの編集は所有者（作成者）のみ可能。ロールに関わらず他者のレポートは編集不可 |
| RPT-013 | レポートの削除は draft 状態でのみ可能 |
| RPT-014 | レポートの提出には明細が1件以上必要 |
| ITM-001 | 明細には日付が必須 |
| ITM-002 | 明細には金額が必須（正の整数値） |
| ITM-003 | 明細にはカテゴリが必須 |
| ITM-004 | 明細には摘要が必須 |
| ITM-005 | カテゴリは固定6種類 |
| ITM-010 | 明細の追加・編集・削除はレポートが draft 状態の時のみ |

### 2.5 承認フロー（状態遷移）

**アクター**: Member（提出）、Approver（承認・却下）、Accounting（支払完了）

#### 機能一覧

| ID | 機能 | 説明 |
|----|------|------|
| WFL-F01 | 承認 | Approver が submitted レポートを承認（submitted → approved）。承認コメント任意 |
| WFL-F02 | 却下 | Approver が submitted レポートを却下（submitted → rejected）。却下理由必須 |
| WFL-F03 | 支払完了 | Accounting が approved レポートに支払完了を記録（approved → paid） |
| WFL-F04 | 承認待ち一覧 | Approver が同テナントの submitted レポート一覧を全件取得（MVP は全件承認型） |
| WFL-F05 | 支払待ち一覧 | Accounting が同テナントの approved レポート一覧を取得 |

#### 入力/出力

**承認**

| 入力 | 制約 |
|------|------|
| 承認コメント | 任意。0〜1000文字。承認時の注意事項や申し送り事項を記録 |

**却下**

| 入力 | 制約 |
|------|------|
| 却下理由 | 必須。1〜1000文字 |

#### ビジネスルール

| ルールID | ルール |
|---------|--------|
| WFL-001 | 状態遷移はドメイン層で一元管理 |
| WFL-002 | 許可される遷移のみ実行可能 |
| WFL-003 | 状態遷移を実行できるロールは遷移ごとに定義 |
| WFL-004 | 終端状態（rejected, paid）からの遷移は不可 |
| WFL-012 | 却下時は理由の入力が必須 |
| RPT-015 | 却下後の再申請は新規レポートとして作成 |
| RPT-016 | 再申請レポートは元レポートへの参照を保持 |

#### MVP スコープ判断

| 項目 | 判断 | 理由 |
|------|------|------|
| 提出取消（submitted → draft） | **対象外** | 状態遷移の複雑化を避ける。承認者の却下で運用カバー。詰み対策として Approver 0人テナントでは提出不可 |
| Approver の自己承認 | **禁止** | 内部統制の基本原則。自分が提出したレポートを自分で承認できない |
| 多段階承認 | **対象外** | MVP では1段階承認。複数 Approver の承認ルート設計は複雑度が高い |

> 状態遷移の詳細は `workflow.md` を参照。

| ルールID | ルール |
|---------|--------|

| ATT-010 | ファイルのダウンロードは署名付きURL経由 |
| ATT-011 | 署名付きURL発行前に認可チェック必須 |

| ATT-013 | アップロード時にMIMEタイプを検証 |
| ATT-014 | S3パスにテナントIDを含む（`{tenant_id}/{report_id}/{file_id}`） |

## 3. 機能要件 — 付随機能（Phase 3）

以下の機能は MVP 対象外。Phase 3 以降で実装する。要件の概要のみ記載し、MVP 設計時に拡張性を阻害しない構造を意識する。

### 3.1 招待フロー

| 項目 | 内容 |
|------|------|
| 概要 | Admin がメールアドレス + ロールを指定してテナントへ招待 |
| トークン | 招待トークン（72時間有効）を生成、招待URLをメールで送信 |
| アカウント作成 | 招待URL経由でパスワード設定、指定ロールでテナントに参加 |
| MVP への影響 | users / tenant_memberships テーブル設計時に招待フローの拡張を考慮 |

### 3.2 アプリ内通知

| 項目 | 内容 |
|------|------|
| 概要 | 提出・承認・却下・支払完了のイベントで通知を発行 |
| 通知先 | イベントに応じた対象ユーザー（例: 提出 → Approver、承認 → Member） |
| 既読管理 | 未読/既読のステータス管理 |
| MVP への影響 | 状態遷移のイベント発行ポイントを設計時に意識 |

### 3.3 監査ログ

| 項目 | 内容 |
|------|------|
| 概要 | 操作履歴（誰が・いつ・何を・変更前後の値）を記録 |
| 制約 | INSERT ONLY（更新・削除不可） |
| 閲覧権限 | Admin のみ |
| MVP への影響 | audit_logs テーブルの事前設計、INSERT ONLY 制約の考慮 |

### 3.4 CSV エクスポート

| 項目 | 内容 |
|------|------|
| 概要 | 経費一覧をCSV形式で出力 |
| フィルタ | 期間・ステータス・カテゴリで絞り込み可能 |
| 権限 | Accounting / Admin のみ |
| MVP への影響 | 一覧取得APIのフィルタ設計を共用可能にする |

### 3.5 メンバー管理画面

| 項目 | 内容 |
|------|------|
| 概要 | メンバー一覧表示・ロール変更・メンバー削除 |
| 権限 | Admin のみ |
| MVP への影響 | MVP 対象外。メンバー管理画面・APIは Phase 3 で実装（`02_scope.md` 参照） |

### 3.6 テナント設定画面

| 項目 | 内容 |
|------|------|
| 概要 | 会社情報（会社名等）の編集画面 |
| 権限 | Admin のみ |
| MVP への影響 | テナント基本情報の取得API（ADM-F01）はMVPで提供。編集画面は Phase 3 |

## 4. 非機能要件

### 4.1 パフォーマンス

| 指標 | 目標値 | 備考 |
|------|--------|------|
| API レスポンスタイム（p95） | 500ms 以下 | 一覧取得を含む |
| ファイルアップロード | 5秒以下（5MBファイル） | ネットワーク帯域に依存 |
| 同時接続ユーザー数 | 100ユーザー | MVP の想定規模 |
| ページネーション | カーソルベース、デフォルト20件/ページ | 大量データ時の安定性 |

### 4.2 セキュリティ

#### インフラ・通信セキュリティルール

| ルールID | ルール |
|---------|--------|
| SEC-012 | レート制限: 認証済み 100 req/min/user、未認証 20 req/min/IP |
| SEC-013 | CORS: 許可オリジンを環境変数で明示指定（ワイルドカード禁止） |
| SEC-014 | セキュリティヘッダー: HSTS, X-Content-Type-Options, X-Frame-Options を全レスポンスに付与 |

#### 要件一覧

| 要件 | 内容 | ルールID |
|------|------|---------|
| 認証 | JWT（RS256） | SEC-003, SEC-004 |
| パスワード | Argon2id ハッシュ、8文字以上 | SEC-002, SEC-010 |
| テナント分離 | アプリ層 + RLS の二重保証 | TNT-001〜005 |
| RBAC | 全APIでミドルウェア検証 | RBC-001 |
| レート制限 | 認証済み: 100 req/min/user、未認証: 20 req/min/IP | SEC-012 |
| CORS | 許可オリジンを明示指定 | SEC-013 |
| セキュリティヘッダー | HSTS, X-Content-Type-Options, X-Frame-Options | SEC-014 |
| ファイルアップロード | MIMEタイプ検証、5MB制限 | ATT-013, ATT-003 |

### 4.3 可用性

| 指標 | 目標値 | 備考 |
|------|--------|------|
| 稼働率 | 99.5%（月間ダウンタイム約3.6時間） | ポートフォリオの現実的な目標 |
| ヘルスチェック | `/health` エンドポイント | ロードバランサ / 監視ツールから利用 |
| DB バックアップ | RDS 自動バックアップ（7日間保持） | 手動リストア手順は対象外 |

### 4.4 データ

| 要件 | 内容 | ルールID |
|------|------|---------|
| 論理削除 | deleted_at タイムスタンプ方式 | DAT-002 |
| タイムスタンプ | 全レコードに created_at, updated_at | DAT-004 |
| 監査証跡 | 提出以降のレポートは物理削除不可 | DAT-001 |
| MVP監査証跡 | 完全な監査ログは Phase 3 だが、状態遷移の固定化・提出後編集禁止・再申請の新規レポート化・遷移属性（approved_by 等）の保持により、改ざんしにくいデータ構造を優先する | - |
| 通貨 | 日本円（JPY）のみ。整数値（円単位） | ITM-002 |
| 文字コード | UTF-8 | - |
| タイムゾーン | UTC で保存、表示時に JST 変換 | - |

## 5. やらないこと（スコープ外）

`02_scope.md` で定義済みの内容を再掲し、要件定義の境界を明確にする。

| 項目 | 理由 |
|------|------|
| メール通知（業務イベント） | 提出・承認・却下・支払完了などの業務イベント通知メールは対象外。アプリ内通知（Phase 3）で代替。ただし、認証機能としてのパスワードリセットメール（AUTH-F06）は MVP 対象 |
| 多段階承認 | 承認ルート設計の複雑度が高い |
| カスタムカテゴリ | 将来拡張可能な設計にするが MVP では固定6カテゴリ |
| 多言語対応 (i18n) | UI は日本語固定 |
| モバイルアプリ | Web アプリのみ |
| OCR / 領収書自動読み取り | 手動入力のみ |
| 外部会計ソフト連携 | API 連携（freee, MF等）は対象外 |
| SSO / OAuth | メール + パスワード認証のみ |
| ファイルのウイルスチェック | 対象外 |
