# 経費精算業務 概要書

## 1. 経費精算とは何か

### 定義

経費精算とは、**従業員が業務のために立て替えた費用を、会社に請求し、払い戻しを受ける一連のプロセス**である。

具体例:
- 営業担当が客先訪問のために使ったタクシー代
- 出張時のホテル宿泊費
- チームの懇親会の飲食費
- 業務に必要な文房具や書籍の購入費

従業員が自分のお金で一旦支払い（立替）、後から会社に精算を求める。この「立替→申請→承認→払戻し」の流れが経費精算の基本構造である。

### なぜ「精算」が必要なのか

企業活動では日常的に少額の支出が発生するが、全てを事前に会社が支払うのは現実的ではない。そこで以下の理由から「従業員立替→事後精算」の仕組みが広く採用されている。

1. **即時性**: 現場で発生する費用に即座に対応できる
2. **柔軟性**: 事前申請では対応できない突発的な支出をカバー
3. **管理効率**: 法人カードを全員に持たせるよりコストが低い
4. **証跡管理**: 領収書と申請を紐づけることで、税務・監査上の証跡を確保

### 企業における経費精算の位置づけ

```
┌─────────────────────────────────────────────────┐
│                  企業の会計業務                    │
├─────────┬──────────┬──────────┬─────────────────┤
│  売上管理 │ 仕入管理  │ 給与計算  │   経費精算      │
│         │          │          │ ← 本プロジェクト │
└─────────┴──────────┴──────────┴─────────────────┘
```

経費精算は企業の会計業務の一部であり、以下と密接に連携する:
- **会計システム**: 精算データは仕訳として会計帳簿に記録される
- **給与計算**: 精算額は給与と一緒に振り込まれることが多い
- **税務申告**: 経費は法人税の損金として処理される（適切な証跡が必要）

## 2. 経費精算に関わる法的背景（日本）

経費精算は単なる社内手続きではなく、法令に基づいた証跡管理が求められる。

### 関連法令

| 法令 | 経費精算との関係 |
|------|----------------|
| **法人税法** | 経費（損金）として認められるには、業務関連性と適切な証跡が必要 |
| **消費税法** | 仕入税額控除を受けるには、適格請求書等（インボイス）の保存が必要 |
| **電子帳簿保存法** | 領収書の電子保存に関するルール。タイムスタンプ等の要件あり |
| **所得税法** | 通勤手当や出張旅費の非課税限度額の規定 |

### 実務上のポイント

- **領収書の保存義務**: 法人税法上、原則7年間の保存が必要
- **電子保存の条件**: 電子帳簿保存法により、スキャナ保存にはタイムスタンプや検索要件が必要
- **インボイス制度**: 2023年10月開始。適格請求書発行事業者の登録番号が記載された領収書でないと、仕入税額控除が受けられない

> **本プロジェクトへの影響**: MVP ではインボイス制度対応や電子帳簿保存法の厳密な準拠は対象外だが、領収書の添付・保管機能は備える。将来的な法対応を見据えた設計（タイムスタンプ付与の拡張ポイント等）は意識する。

## 3. 経費精算の歴史的変遷

### 紙ベース時代

```
従業員                   上司                  経理部
  │                      │                      │
  ├─ 紙の精算書に記入 ──→│                      │
  ├─ 領収書を糊付け ────→│                      │
  │                      ├─ 印鑑で承認 ────────→│
  │                      │                      ├─ 内容確認
  │                      │                      ├─ 仕訳入力
  │                      │                      ├─ 銀行振込手配
  │                      │                      │
```

**課題**:
- 紙の紛失リスク
- 承認者不在時の滞留
- 経理の手入力による転記ミス
- 月末に申請が集中し経理部が疲弊
- 領収書の物理保管スペース

### Excel/メール時代

Excelテンプレートに入力し、領収書の写真と一緒にメールで提出する方式。紙よりは効率的だが、以下の課題が残る:
- バージョン管理が困難（「経費精算_v3_最終_修正.xlsx」問題）
- 承認ステータスの追跡が困難
- 集計作業が手動
- 添付ファイルがメールに散在

### SaaS 時代（現在）

クラウド型の経費精算サービスが主流になりつつある。

**代表的なサービス**:

| サービス | 特徴 | 対象企業規模 |
|---------|------|------------|
| freee経費精算 | freee会計との連携。UIがシンプル | 中小企業 |
| マネーフォワード クラウド経費 | MFクラウド会計との連携。OCR機能 | 中小〜中堅 |
| ジョブカン経費精算 | 勤怠管理等との統合。低価格 | 中小企業 |
| SAP Concur | グローバル対応。多段階承認。規定チェック | 大企業 |
| 楽楽精算 | 国内シェア上位。カスタマイズ性が高い | 中堅〜大企業 |

**SaaS化のメリット**:
- いつでもどこでも申請・承認可能（スマホ対応）
- リアルタイムのステータス追跡
- 自動集計・レポート生成
- OCRによる領収書自動読み取り（一部サービス）
- 会計ソフトとの自動連携
- 電子帳簿保存法対応

## 4. 登場人物（ステークホルダー）と関係性

### 全体像

```
┌─────────────────────────────────────────────────────┐
│                     テナント（企業）                    │
│                                                       │
│   ┌──────────┐    承認依頼    ┌──────────┐            │
│   │  Member   │──────────────→│ Approver │            │
│   │ (申請者)  │←──────────────│ (承認者)  │            │
│   └──────────┘  承認/却下     └──────────┘            │
│        │                           │                  │
│        │ 申請                      │ 承認済み通知      │
│        ↓                           ↓                  │
│   ┌──────────────────────────────────────┐            │
│   │           Accounting (経理)           │            │
│   │  承認済み経費の確認・支払処理・集計     │            │
│   └──────────────────────────────────────┘            │
│                                                       │
│   ┌──────────────────────────────────────┐            │
│   │             Admin (管理者)             │            │
│   │  メンバー管理・設定・全体監視           │            │
│   └──────────────────────────────────────┘            │
└─────────────────────────────────────────────────────┘
```

### 各ロールの概要

| ロール | 人物像（例） | 主な責務 |
|--------|------------|---------|
| **Member** | 営業担当、エンジニア、一般社員 | 経費を立替え、領収書を添付して精算を申請する |
| **Approver** | 部長、マネージャー、チームリーダー | 部下の申請内容を確認し、適切なら承認、不備があれば却下する |
| **Accounting** | 経理部員、財務担当 | 承認済みの経費を確認し、支払処理を行い、会計システムに記録する |
| **Admin** | 総務部長、システム管理者、経営者 | テナント（企業）の設定、メンバー管理、全体の運用管理を行う |

### ロール間の情報の流れ

```
Member が経費を立替える
    ↓
Member が経費レポートを作成し、領収書を添付して提出
    ↓
Approver が承認待ち一覧で確認（Phase 3: 通知で受信）
    ↓
Approver が内容を確認
    ├─ 問題なし → 承認 → Accounting が一覧で確認（Phase 3: 通知）
    └─ 不備あり → 却下（理由付き）→ Member がステータスで確認（Phase 3: 通知）
                     ↓
                 Member が内容を修正し再申請
    ↓
Accounting が承認済みレポートを確認
    ↓
Accounting が支払処理を実行（銀行振込等）
    ↓
支払完了を記録 → Member がステータスで確認（Phase 3: 通知）
```

## 5. マルチテナントの意味と必要性

### SaaS におけるマルチテナントとは

1つのアプリケーション（サービス）を**複数の企業（テナント）が共有**して利用する構成のこと。

```
┌──────────────────────────────────────┐
│        経費精算 SaaS アプリケーション    │
│                                        │
│  ┌──────────┐  ┌──────────┐           │
│  │ テナント A │  │ テナント B │  ...     │
│  │ (A社)     │  │ (B社)     │           │
│  │           │  │           │           │
│  │ ユーザーA1 │  │ ユーザーB1 │           │
│  │ ユーザーA2 │  │ ユーザーB2 │           │
│  │ データA    │  │ データB    │           │
│  └──────────┘  └──────────┘           │
│                                        │
│  ※ 同じDBを共有するが、データは厳密に分離  │
└──────────────────────────────────────┘
```

### なぜマルチテナントか

| 方式 | 説明 | メリット | デメリット |
|------|------|---------|----------|
| **シングルテナント** | 企業ごとに個別のアプリ・DBを用意 | 完全なデータ分離 | 運用コスト高、スケールしにくい |
| **マルチテナント** | 1つのアプリ・DBを共有 | 運用コスト低、機能追加が全テナントに反映 | データ分離の設計が重要 |

本プロジェクトでは**マルチテナント（共有DB方式）** を採用し、`tenant_id` による論理分離 + PostgreSQL RLS による二重保証でデータの安全性を担保する。

### テナント分離で防ぐべきリスク

- **情報漏洩**: A社の経費データがB社のユーザーに見える
- **データ改ざん**: 他社のレポートを承認・却下できてしまう
- **集計汚染**: A社のダッシュボードにB社のデータが混入する

> これらは「あってはならないバグ」であり、テナント分離はセキュリティの最重要要件。

## 6. 本プロジェクトが解決する課題

### ターゲット企業像

- 従業員 10〜200名程度の中小企業
- 経費精算が紙 or Excel で行われている、または既存ツールに不満がある
- IT部門が小さく、シンプルなSaaSを求めている

### 解決する課題

| 課題 | 本システムでの解決策 |
|------|-------------------|
| 紙の紛失・管理コスト | 電子申請 + 領収書の画像アップロード |
| 承認者不在による滞留 | Web上でいつでも承認可能 |
| 経理の手動集計 | 自動集計・ステータス管理 |
| 申請状況が見えない | リアルタイムのステータス追跡 |
| 不正な経費申請 | 承認フロー + 証跡管理（将来: 監査ログ） |
| 権限管理の不備 | RBAC による厳密なアクセス制御 |

## 7. 本資料のまとめ

経費精算業務の本質は「**立替金の正当性を組織的に検証し、適切に払い戻すプロセス**」である。

要件定義に進む前に押さえるべきポイント:

1. **業務の目的**: 従業員の立替費用を正確かつ迅速に精算すること
2. **法的要件**: 領収書の保管義務、税務上の証跡確保
3. **ステークホルダー**: 4つのロールが明確な責務分担で協業する
4. **SaaS化の価値**: 紙/Excel からの脱却による効率化と透明性の向上
5. **テナント分離**: マルチテナントSaaSの最重要セキュリティ要件
