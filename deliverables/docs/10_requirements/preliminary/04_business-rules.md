# 業務ルール・制約一覧

## 概要

本資料では経費精算業務に存在する**ビジネスルール**を網羅的に整理する。各ルールは要件定義・ドメイン設計・実装の各段階で参照し、漏れなく実現することを目的とする。

ルールは以下のカテゴリに分類する:
1. 経費レポートに関するルール
2. 経費明細に関するルール
3. 添付ファイルに関するルール
4. 状態遷移に関するルール
5. 権限（RBAC）に関するルール
6. テナント分離に関するルール
7. 認証・セキュリティに関するルール

---

## 1. 経費レポートに関するルール

### データルール

| ID | ルール | 根拠 |
|----|--------|------|
| RPT-001 | レポートにはタイトルが必須 | レポートの識別に必要 |
| RPT-002 | レポートには対象期間（開始日・終了日）が必須 | 月次集計・フィルタに必要 |
| RPT-003 | 対象期間の開始日は終了日以前であること | データ整合性 |
| RPT-004 | レポートは作成者（ユーザー）に紐づく | 所有者の特定に必要 |
| RPT-005 | レポートは必ず1つのテナントに属する | テナント分離 |
| RPT-006 | レポートの合計金額は明細の金額合計から自動計算 | 手入力による不整合防止 |

### 操作ルール

| ID | ルール | 根拠 |
|----|--------|------|
| RPT-010 | レポートの作成は Member / Approver / Admin が可能 | RBAC |
| RPT-011 | レポートの編集は `draft` 状態でのみ可能 | 提出後の改ざん防止 |
| RPT-012 | レポートの編集は所有者（作成者）のみ可能。ロールに関わらず他者のレポートは編集不可 | 所有権の尊重 |
| RPT-013 | レポートの削除は `draft` 状態でのみ可能 | 監査証跡の保持 |
| RPT-014 | レポートの提出には明細が1件以上必要 | 空申請の防止 |
| RPT-015 | 却下されたレポートに対する再申請は新規レポートとして作成 | 監査証跡の保持、変更追跡の容易さ |
| RPT-016 | 再申請レポートは元レポートへの参照（reference）を保持 | 却下→再申請の経緯を追跡可能にする |

## 2. 経費明細に関するルール

### データルール

| ID | ルール | 根拠 |
|----|--------|------|
| ITM-001 | 明細には日付が必須 | いつの支出かを特定 |
| ITM-002 | 明細には金額が必須（正の整数値） | 精算対象の金額。小数は扱わない（日本円） |
| ITM-003 | 明細にはカテゴリが必須 | 経費の分類、会計処理に必要 |
| ITM-004 | 明細には摘要（利用目的・内容説明）が必須 | 承認判断・税務証跡に必要 |
| ITM-005 | カテゴリは固定6種類（交通費/宿泊費/飲食費/消耗品費/通信費/その他） | MVP ではカスタムカテゴリは対象外 |
| ITM-006 | 明細は必ず1つのレポートに属する | 親子関係 |
| ITM-007 | 明細の日付はレポートの対象期間内であることが望ましい | データ整合性（警告レベル。強制はしない） |

### 操作ルール

| ID | ルール | 根拠 |
|----|--------|------|
| ITM-010 | 明細の追加・編集・削除はレポートが `draft` 状態の時のみ | 提出後の改ざん防止 |
| ITM-011 | 明細の操作権限はレポートの操作権限に準ずる | レポートと一体管理 |

### 金額に関する考慮事項

| 考慮点 | 本プロジェクトでの方針 |
|--------|---------------------|
| 通貨 | 日本円（JPY）のみ。多通貨対応は対象外 |
| 小数点 | 扱わない。整数値（円単位） |
| 上限金額 | システム上の上限は設定しない（承認者の判断に委ねる） |
| マイナス金額 | 許可しない（返金等は別の仕組みで処理） |
| 税込/税抜 | 税込金額で入力（消費税の内訳管理は対象外） |

## 3. 添付ファイルに関するルール

### データルール

| ID | ルール | 根拠 |
|----|--------|------|
| ATT-001 | 添付ファイルは経費明細に紐づく | 証跡と支出の対応 |
| ATT-002 | 許可するファイル形式: JPEG, PNG, PDF | 領収書として一般的な形式 |
| ATT-003 | 1ファイルのサイズ上限: 5MB | ストレージコストとアップロード速度のバランス |
| ATT-004 | 1明細に複数ファイル添付可能 | 1つの支出に対して複数の証跡がある場合 |
| ATT-005 | ファイルは S3 に保存、メタデータは DB に保存 | ストレージの分離 |

### セキュリティルール

| ID | ルール | 根拠 |
|----|--------|------|
| ATT-010 | ファイルのダウンロードは署名付きURL経由 | 直接アクセスの防止 |
| ATT-011 | 署名付きURL発行前に認可チェック必須 | 同テナント・権限確認 |
| ATT-012 | 署名付きURLの有効期限は短時間（例: 15分） | セキュリティ |
| ATT-013 | アップロード時にMIMEタイプを検証 | 不正ファイルの防止 |
| ATT-014 | ファイル名にテナントIDを含むパス設計（例: `{tenant_id}/{report_id}/{file_id}`） | テナント分離 |

### 操作ルール

| ID | ルール | 根拠 |
|----|--------|------|
| ATT-020 | 添付の追加・削除はレポートが `draft` 状態の時のみ | 提出後の改ざん防止 |
| ATT-021 | 添付の閲覧はレポート閲覧権限に準ずる | レポートと一体管理 |

## 4. 状態遷移に関するルール

### 基本ルール

| ID | ルール | 根拠 |
|----|--------|------|
| WFL-001 | 状態遷移はドメイン層で一元管理 | 整合性保証、ロジックの散在防止 |
| WFL-002 | 許可される遷移のみ実行可能、それ以外はエラー | 不正な状態遷移の防止 |
| WFL-003 | 状態遷移を実行できるロールは遷移ごとに定義 | RBAC の徹底 |
| WFL-004 | 終端状態（rejected, paid）からの遷移は一切不可 | 確定した結果の保護 |

### 遷移別ルール

| ID | 遷移 | 実行者 | 追加条件 |
|----|------|--------|---------|
| WFL-010 | draft → submitted | 所有者（Member / Approver / Admin） | 明細が1件以上存在すること |
| WFL-011 | submitted → approved | Approver | なし |
| WFL-012 | submitted → rejected | Approver | 却下理由の入力が必須 |
| WFL-013 | approved → paid | Accounting | なし |
| WFL-014 | draft → submitted | 所有者 | 同一テナントに Approver ロールのメンバーが1人以上存在すること |

### 状態遷移の監査属性

レポートには以下の属性を保持し、状態遷移の結果を記録する。監査ログ（Phase 3）とは別に、レポート自体の属性として管理する。

| 属性 | 設定タイミング | 説明 |
|------|--------------|------|
| submitted_at / submitted_by | draft → submitted | 提出日時・提出者 |
| approved_at / approved_by | submitted → approved | 承認日時・承認者 |
| rejected_at / rejected_by | submitted → rejected | 却下日時・却下者 |
| rejection_reason | submitted → rejected | 却下理由（必須） |
| paid_at / paid_by | approved → paid | 支払完了日時・処理者 |

### 各状態で許可される操作

| 状態 | レポート編集 | 明細追加/編集/削除 | 添付追加/削除 | 提出 | 承認/却下 | 支払完了 | 削除 |
|------|------------|-------------------|-------------|------|----------|---------|------|
| draft | ○ | ○ | ○ | ○ | × | × | ○ |
| submitted | × | × | × | × | ○ | × | × |
| approved | × | × | × | × | × | ○ | × |
| rejected | × | × | × | × | × | × | × |
| paid | × | × | × | × | × | × | × |

## 5. 権限（RBAC）に関するルール

### 基本原則

| ID | ルール | 根拠 |
|----|--------|------|
| RBC-001 | 全APIでミドルウェアによるロール検証を実施 | 認可の一貫性 |
| RBC-002 | ユーザーは1テナントに対して1つのロールを持つ | シンプルな権限モデル |
| RBC-003 | リソースへのアクセスは「ロール」と「所有権」の両方で制御 | 最小権限の原則 |
| RBC-004 | 権限不足時は 403 Forbidden を返却 | セキュリティ |

### ロール別操作権限マトリクス

| 操作 | Admin | Approver | Member | Accounting |
|------|-------|----------|--------|------------|
| **レポート作成** | ○（自分のみ） | ○（自分のみ） | ○（自分のみ） | × |
| **レポート閲覧（自分の）** | ○ | ○ | ○ | - |
| **レポート閲覧（全体）** | ○ | ○（承認対象: 同テナントの submitted） | × | ○ |
| **レポート編集** | ○（自分のdraftのみ） | ○（自分のdraftのみ） | ○（自分のdraftのみ） | × |
| **レポート削除** | ○（自分のdraftのみ） | ○（自分のdraftのみ） | ○（自分のdraftのみ） | × |
| **レポート提出** | ○（自分のdraftのみ） | ○（自分のdraftのみ） | ○（自分のdraftのみ） | × |
| **承認** | × | ○（他人のレポートのみ） | × | × |
| **却下** | × | ○（他人のレポートのみ） | × | × |
| **支払完了** | × | × | × | ○ |
| **メンバー招待** | ○ | × | × | × |
| **メンバー管理** | ○ | × | × | × |
| **ロール変更** | ○ | × | × | × |

### 所有権ルール

| ID | ルール | 根拠 |
|----|--------|------|
| RBC-010 | 申請系操作（作成/編集/削除/提出）は、ロールに関わらず所有者のレポートに限定 | 他人の経費を操作できてはならない |
| RBC-011 | Approver は同テナントの submitted レポートを承認/却下可能 | 承認者の責務 |
| RBC-012 | Accounting は同テナントの approved レポートに支払完了を記録可能 | 経理の責務 |
| RBC-013 | Admin は同テナントの全レポートを閲覧可能（編集不可） | 管理者権限 |
| RBC-014 | Admin は自分が作成したレポートに限り、申請者として作成・編集・削除・提出が可能。他者のレポートは閲覧のみ | 所有権の尊重・不正改ざん防止 |
| RBC-015 | MVP では Approver は同一テナント内の submitted 状態のレポートをすべて閲覧・承認・却下できる（全件承認型）。担当者割当は Phase 3 以降の拡張とする | MVP のシンプルさ優先 |
| RBC-016 | Approver は自分が作成したレポートを承認・却下できない（自己承認・自己却下禁止） | 内部統制の基本原則 |

## 6. テナント分離に関するルール

### 基本原則

| ID | ルール | 根拠 |
|----|--------|------|
| TNT-001 | 全テーブルに `tenant_id` カラムが必須 | テナント分離の基盤 |
| TNT-002 | 全クエリに `WHERE tenant_id = ?` が必須 | データ漏洩防止 |
| TNT-003 | `tenant_id` の付与・検証はリポジトリ層で強制 | ハンドラ層での漏れ防止 |
| TNT-004 | PostgreSQL RLS でアプリ層の保証を二重化 | 防御の多層化 |
| TNT-005 | テナント間のデータ参照は一切不可 | 情報漏洩防止の最重要ルール |

### 適用範囲

| リソース | tenant_id 必須 | RLS 適用 |
|----------|--------------|----------|
| ユーザー | ○（tenant_memberships経由） | ○ |
| 経費レポート | ○ | ○ |
| 経費明細 | ○（冗長保持。レポート経由のJOIN不要でRLS適用可能） | ○ |
| 添付ファイル | ○（メタデータに冗長保持）/ S3パスにも含む | ○ |
| 招待（Phase 3） | ○ | ○ |
| 監査ログ（Phase 3） | ○ | ○ |
| 通知（Phase 3） | ○ | ○ |

## 7. 認証・セキュリティに関するルール

### 認証

| ID | ルール | 根拠 |
|----|--------|------|
| SEC-001 | 認証方式: メール + パスワード | MVP の認証方式 |
| SEC-002 | パスワードハッシュ: Argon2id | 安全なハッシュアルゴリズム |
| SEC-003 | JWT 認証: アクセストークン（15分）+ リフレッシュトークン（7日） | セキュリティとUXのバランス |
| SEC-004 | JWT 署名: RS256（非対称鍵） | 鍵管理のセキュリティ |
| SEC-005 | ログアウト時にリフレッシュトークンを無効化 | セッション管理 |
| SEC-006 | パスワードリセット: メールアドレスでリセットトークンを発行し、新パスワードを設定可能 | SaaS 運用上必須 |

### セキュリティ

| ID | ルール | 根拠 |
|----|--------|------|
| SEC-010 | パスワード最小長: 8文字以上 | セキュリティポリシー |
| SEC-011 | 認証失敗時のレスポンスでユーザー存在を推測させない | 列挙攻撃防止 |
| SEC-012 | APIレート制限: 認証済み 100 req/min/user、未認証 20 req/min/IP | DDoS/ブルートフォース対策 |
| SEC-013 | CORS ポリシー: 許可するオリジンを明示的に指定 | XSS対策 |
| SEC-014 | セキュリティヘッダー: HSTS, X-Content-Type-Options, X-Frame-Options | 基本的なセキュリティ対策 |

### 403 / 404 の使い分け

| 状況 | HTTPステータス | 理由 |
|------|--------------|------|
| テナント境界を越えるアクセス | 404 Not Found | リソースの存在自体を秘匿する（存在漏洩防止） |
| 同一テナント内での権限不足 | 403 Forbidden | 認可失敗として明示する |

| ID | ルール | 根拠 |
|----|--------|------|
| DAT-001 | 提出以降のレポートは物理削除不可 | 監査証跡の保持 |

## 9. ルール一覧（クイックリファレンス）

### ルールIDの採番体系

| プレフィックス | カテゴリ |
|--------------|---------|
| RPT | 経費レポート |
| ITM | 経費明細 |
| ATT | 添付ファイル |
| WFL | ワークフロー（状態遷移） |
| RBC | RBAC（権限） |
| TNT | テナント分離 |
| SEC | セキュリティ |
| DAT | データ保全 |

**採番ルール**: 一度採番されたIDを別の意味で再利用してはならない。下流工程で新たなルールが必要な場合は、本資料で新規IDを採番してから下流に反映する。

### MVP で確定した論点

| # | 論点 | 決定事項 | 理由 |
|---|------|---------|------|
| 1 | 提出取消（submitted → draft） | MVP 対象外 | 状態遷移の複雑化を避ける。詰み対策として、Approver が0人のテナントでは提出不可とするバリデーションを設ける（WFL-014） |
| 2 | 明細の日付がレポート対象期間外 | 警告のみ、エラーにしない | ユーザビリティ優先 |
| 3 | Admin によるレポート編集 | 自分のレポートは申請者として操作可能。他者のレポートは閲覧のみ、編集不可 | 所有権の尊重。管理者が他者の申請を改ざんできてはならない |
| 4 | Approver が自分のレポートを自己承認 | 禁止（自己承認不可）（RBC-016） | 内部統制の基本原則 |
| 5 | 1レポートあたりの明細数上限 | 設定しない（MVPでは） | 実用上の問題が起きにくい |
| 6 | 1明細あたりの添付ファイル数上限 | 設定しない（MVPでは） | 実用上の問題が起きにくい |

## 10. 本資料のまとめ

### 要件定義に向けた重要なポイント

1. **ルールの分類**: ルールはデータ制約、操作制約、権限制約、セキュリティ制約に分かれ、それぞれ実装場所が異なる
2. **ルールの厳格度**: 「絶対に守るルール」（テナント分離、RBAC）と「運用でカバーするルール」（締め日管理）を区別する
3. **MVPでの確定事項**: 提出取消や自己承認禁止など、MVP段階で確定した論点を明記した
4. **ルールID**: 各ルールにIDを付番し、要件定義・設計・テスト仕様から参照可能にする

### MVP での監査証跡の位置づけ

MVP では完全な監査ログは提供しないが、以下の設計により改ざんしにくいデータ構造を優先する。

- 状態遷移の固定化（許可された遷移のみ実行可能）
- 提出後のレポート編集禁止
- 再申請時の新規レポート化（元レポートは rejected のまま保持）
- 状態遷移の監査属性（approved_by / paid_by 等）をレポートに保持

完全な操作履歴の記録は Phase 3 の監査ログ機能で補完する。
