# アクター別業務分析

## 概要

経費精算業務には4つのロール（アクター）が関与する。本資料では各アクターの**日常業務**、**システムに対する期待**、**現状の課題（ペインポイント）** を具体的に記述し、要件定義の基礎とする。

---

## 1. Member（申請者 / 一般社員）

### 人物像

- 営業担当、エンジニア、企画職など、経費精算が「本業ではない」人
- 月に数回〜十数回の経費が発生する
- 経費精算は「面倒だがやらないとお金が戻ってこない」作業

### 日常業務における経費精算の流れ

```
【日常】
1. 業務中に経費が発生（タクシー、会食、備品購入など）
2. 領収書を受け取り、保管する（紛失リスクあり）
3. 数日〜数週間分をまとめて精算書を作成
4. 領収書をスキャン/撮影してシステムに登録
5. 経費レポートとして提出
6. 承認結果を待つ
7. 却下された場合、修正して再申請
8. 承認され、給与と一緒に振り込まれるのを待つ
```

### システムに対する期待

| 期待 | 詳細 |
|------|------|
| **すぐ終わる** | 経費精算に時間をかけたくない。入力項目は最小限にしてほしい |
| **迷わない** | 何を入力すればいいか、どのカテゴリを選べばいいか直感的にわかる |
| **状況が見える** | 提出した申請が今どうなっているか（承認待ち？却下？支払済み？）を一目で確認したい |
| **スマホ対応** | 外出先で領収書を撮影してすぐ登録したい（MVP ではレスポンシブ対応） |
| **下書き保存** | 途中まで入力して後で再開したい |

### ペインポイント（現状の課題）

| # | 課題 | 影響 |
|---|------|------|
| 1 | **領収書をなくす** | 精算できず自腹になる。再発行は手間がかかる |
| 2 | **カテゴリに迷う** | 「この支出はどのカテゴリ？」で毎回悩む。間違えると差し戻される |
| 3 | **月末にまとめて作業** | 溜め込んで月末に大量入力。記憶が曖昧で摘要が雑になる |
| 4 | **承認が遅い** | 提出しても承認者が放置。いつ振り込まれるかわからない |
| 5 | **却下理由が不明確** | 「不備あり」としか書かれず、何を直せばいいかわからない |
| 6 | **再申請の手間** | 却下後に一から作り直すのが面倒。修正箇所だけ直したい |

### 頻出する操作パターン

```
【最も多い操作】
- 経費レポート作成 → 明細追加（1〜10件）→ 領収書添付 → 提出

【次に多い操作】
- 過去のレポートのステータス確認
- 却下されたレポートの内容確認 → 新規レポートで再申請

【たまにある操作】
- 下書き保存したレポートの編集再開
- 提出済みレポートの内容確認（明細の振り返り）
```

## 2. Approver（承認者 / マネージャー）

### 人物像

- 部長、マネージャー、チームリーダーなど、部下を持つ管理職
- 承認業務は多数ある業務の一つ（経費以外にも休暇申請、購買申請等がある）
- 短時間で正確に判断したい

### 日常業務における承認の流れ

```
【日常】
1. 通知（メール/アプリ内）で承認依頼を受け取る
2. 承認待ち一覧を確認
3. 各レポートの内容を確認:
   - 経費の妥当性（金額が相場と比べて適切か）
   - カテゴリが正しいか
   - 領収書が添付されているか
   - 摘要（利用目的）が業務に関連しているか
4. 問題なければ承認、不備があれば理由を添えて却下
5. 急ぎの案件（出張精算等）は優先的に処理
```

### システムに対する期待

| 期待 | 詳細 |
|------|------|
| **一覧性** | 承認待ちの件数と内容が一目で把握できる。古い依頼が埋もれない |
| **判断材料が揃っている** | 金額・カテゴリ・摘要・領収書がワンクリックで確認できる |
| **一括処理** | 複数の申請を短時間でさばける（1件ずつ開かずに済む） |
| **却下時の明確な伝達** | 却下理由を具体的に書いて申請者に伝えたい |
| **モバイル対応** | 移動中や会議の合間にスマホで承認したい |

### ペインポイント（現状の課題）

| # | 課題 | 影響 |
|---|------|------|
| 1 | **承認依頼の見落とし** | メールに埋もれて気づかず、申請者の精算が遅れる |
| 2 | **判断基準が属人的** | 「これは認めていいのか？」の判断基準がマネージャーごとに異なる |
| 3 | **領収書の確認が面倒** | 添付ファイルを1枚ずつ開いて確認するのに時間がかかる |
| 4 | **差し戻し後の追跡** | 却下した後、再申請されたかどうかを追いにくい |
| 5 | **月末の承認ラッシュ** | 月末に大量の申請が来て処理が追いつかない |

### 承認時の判断基準（実務上の観点）

Approver が承認/却下を判断する際の典型的な観点:

| 観点 | 確認内容 |
|------|---------|
| **業務関連性** | その支出は業務に必要なものか？私用ではないか？ |
| **金額の妥当性** | 相場と比べて高すぎないか？（タクシー代5万円は要確認） |
| **証跡の有無** | 領収書が添付されているか？金額と一致しているか？ |
| **カテゴリの正確性** | 飲食費を交通費で申請していないか？ |
| **摘要の具体性** | 「打ち合わせ」ではなく「XX社との商談」のように具体的か？ |
| **会社の経費規定** | 上限額を超えていないか？事前承認が必要な支出ではないか？ |

> **本プロジェクトへの影響**: MVP では経費規定のマスタ管理（上限額設定等）は対象外。承認/却下の判断は Approver の裁量に委ねる。ただし、却下理由の入力は必須とする。

## 3. Accounting（経理担当）

### 人物像

- 経理部の担当者。数値の正確性と期限遵守が求められる
- 経費精算は月次の会計処理の一部として実施
- 法令遵守（税務・監査対応）の意識が高い

### 日常業務における経費精算処理の流れ

```
【月次処理】
1. 承認済みの経費レポート一覧を確認
2. 各レポートの内容を精査:
   - 金額の正確性
   - カテゴリ（勘定科目）の適切さ
   - 領収書の添付状況
   - 消費税区分の確認
3. 問題がなければ支払処理を実行
   - 銀行振込データの作成
   - 支払日の設定（給与日に合わせることが多い）
4. 支払完了を記録
5. 会計システムへの仕訳入力（または自動連携）
6. 月次レポートの作成（部門別・カテゴリ別の集計）
```

### システムに対する期待

| 期待 | 詳細 |
|------|------|
| **正確な集計** | 承認済み経費の合計金額をミスなく把握したい |
| **一覧・フィルタ** | ステータス・期間・部門・カテゴリで経費を絞り込みたい |
| **エクスポート** | CSV等で会計システムに取り込めるデータを出力したい |
| **支払管理** | 「承認済み・未払い」の一覧と、支払完了の記録を管理したい |
| **監査対応** | 過去の精算履歴を遡って確認できること |

### ペインポイント（現状の課題）

| # | 課題 | 影響 |
|---|------|------|
| 1 | **手動集計のミス** | Excel集計でのコピペミス、合計金額の不一致 |
| 2 | **カテゴリ不統一** | 申請者によってカテゴリの使い方がバラバラ。仕訳に手間がかかる |
| 3 | **領収書の突合** | 申請金額と領収書の金額が一致しない場合の差し戻しコスト |
| 4 | **月末の集中** | 月末締めに間に合わない申請が翌月に繰り越される |
| 5 | **監査時の証跡探し** | 「3ヶ月前のあの経費の領収書は？」に即座に答えられない |
| 6 | **支払漏れ** | 承認済みなのに支払処理が漏れて従業員からクレーム |

### 経理特有の業務知識

| 概念 | 説明 |
|------|------|
| **勘定科目** | 経費のカテゴリを会計上の科目に対応させる（交通費→旅費交通費、飲食費→交際費/会議費 等） |
| **仕訳** | 経費を借方/貸方に分けて記帳する会計処理 |
| **月次締め** | 月末で経費を締め、翌月初に集計・支払処理を行う |
| **消費税区分** | 課税/非課税/不課税の区分。仕入税額控除に影響する |

> **本プロジェクトへの影響**: MVP では勘定科目マッピングや消費税区分管理は対象外。固定の6カテゴリで運用し、CSVエクスポートは Phase 3 で対応。

## 4. Admin（管理者）

### 人物像

- テナント（企業）の管理責任者。総務部長、小規模企業では経営者自身が兼任することも
- システムの初期設定、メンバー管理、運用ポリシーの設定を担当
- 日常的にシステムを使うというより、セットアップ時と問題発生時に操作する

### 日常業務における管理業務の流れ

```
【初期セットアップ】
1. テナント（企業）情報の登録
2. メンバーの招待（メール + ロール指定）
3. Approver ロールの付与（Phase 3 で担当者割当を拡張予定）
4. （将来）経費カテゴリのカスタマイズ

【日常運用】
5. 新メンバーの追加・退職者の削除
6. ロール変更（Member → Approver への昇格等）
7. 問題発生時の対応（パスワードリセット等）

【定期確認】
8. 全体の利用状況確認
9. 未処理の申請がないか監視
```

### システムに対する期待

| 期待 | 詳細 |
|------|------|
| **簡単な初期設定** | 会社情報を入力し、メンバーを招待するだけで使い始められる |
| **メンバー管理** | メンバーの追加・ロール変更・削除が直感的にできる |
| **全体把握** | 全社の経費状況（未処理件数、合計金額等）を俯瞰できる |
| **セキュリティ** | 退職者のアクセスを確実に無効化できる |
| **監査ログ** | 「誰がいつ何をしたか」を追跡できる（Phase 3） |

### ペインポイント（現状の課題）

| # | 課題 | 影響 |
|---|------|------|
| 1 | **オンボーディングの手間** | 新入社員への精算フローの説明に時間がかかる |
| 2 | **退職者のアクセス管理** | 退職後のアカウント無効化が漏れるリスク |
| 3 | **ロール変更の反映** | 組織変更時に承認者の割り当てを変更する手間 |
| 4 | **全体状況の把握困難** | 「今月の未処理件数は？」に即答できない |
| 5 | **トラブル対応** | 「パスワードを忘れた」「申請が消えた」等の問い合わせ対応 |

## 5. ロール間の相互作用マトリクス

各ロールが他のロールとどのように関わるかを整理する。

| From ＼ To | Member | Approver | Accounting | Admin |
|-----------|--------|----------|------------|-------|
| **Member** | - | レポート提出、却下理由確認 | 支払状況確認 | 問い合わせ |
| **Approver** | 承認/却下通知 | - | （なし） | 問い合わせ |
| **Accounting** | 支払完了通知 | （なし） | - | 問い合わせ |
| **Admin** | 招待、ロール管理 | 招待、ロール管理 | 招待、ロール管理 | - |

## 6. 各ロールの操作頻度と重要度

| 操作 | 実行者 | 頻度 | 重要度 | 備考 |
|------|--------|------|--------|------|
| 経費レポート作成 | Member | 高（週数回） | 高 | 最も使われる機能 |
| 明細追加 | Member | 高 | 高 | レポート作成と一体 |
| 領収書添付 | Member | 高 | 高 | 証跡管理の要 |
| レポート提出 | Member | 中（月数回） | 高 | 状態遷移のトリガー |
| ステータス確認 | Member | 高 | 中 | 読み取り専用 |
| 承認 | Approver | 中 | 最高 | 業務フローの要 |
| 却下 | Approver | 低 | 最高 | 却下理由必須 |
| 支払処理 | Accounting | 低（月数回） | 最高 | 金銭に直結 |
| メンバー招待 | Admin | 低 | 高 | 初期/入退社時 |
| ロール変更 | Admin | 低 | 高 | 組織変更時 |

## 7. 本資料のまとめ

### 要件定義に向けた重要な示唆

1. **Member のUXが最優先**: 最も頻繁にシステムを使うのは Member。入力の手間を最小化する設計が重要
2. **Approver の判断効率**: 承認待ち一覧の一覧性と、判断に必要な情報へのアクセスしやすさ
3. **Accounting の正確性**: 集計ミスは金銭的損失に直結。フィルタ・ソート・エクスポート機能が重要
4. **Admin の操作は低頻度だが影響大**: メンバー管理やロール変更はセキュリティに直結する操作
5. **却下→再申請のフロー**: 却下理由の明確さと再申請の容易さが、Member と Approver 双方の満足度に影響
