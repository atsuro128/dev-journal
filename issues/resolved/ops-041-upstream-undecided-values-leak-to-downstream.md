# 上流の未確定値が下流 Step まで流出し、task-plan で吸収されている

## 発見日
2026-03-28

## カテゴリ
project-management

## 影響度
高

## 発見経緯
user-report

## 関連ステップ
Step 0〜5（上流全体 → 詳細設計）

## 問題

Step 5 の task-plan に「判断ポイント」として記録されていた以下の項目は、本来上流 Step で確定しているべき値だった。

| 値 | task-plan での決定 | 本来確定すべき Step | 現状 |
|---|---|---|---|
| 月別サマリー期間 | 3ヶ月固定 | Step 1 requirements or Step 4 screens.md | `直近 N ヶ月` のまま Step 5 まで到達 |
| 直近レポート一覧件数 | 5件 | Step 4 screens.md | 未記載 |
| 明細の編集方式 | スライドパネル | Step 4 screens.md | 未記載 |
| S3 キー命名規則 | `{tenant_id}/{report_id}/{attachment_id}` | Step 3 architecture.md | 未記載 |
| アップロード方式 | API 経由プロキシ（MVP） | Step 3 architecture.md | ダウンロードの署名付き URL のみ言及、アップロード方式は未確定 |

### 根本原因の推察

1. **上流 Step の完了条件が「具体値の確定」を要求していない**
   - requirements.md の DASH-005 が `直近 N ヶ月` という変数表記を許容したまま完了条件を通過した
   - screens.md がロール別の表示項目を列挙しているが、具体的な表示件数・UI パターンの確定を求めていない
   - architecture.md がエンドポイント一覧で `POST 添付アップロード` を定義しているが、方式（API プロキシ vs 署名付き URL 直接アップロード）の選定を完了条件に含めていない

2. **work-breakdown のレビュー観点が「値の確定」を検証していない**
   - 「要件 ID が存在するか」「矛盾がないか」は検証しているが、「変数（N, TBD 等）が残っていないか」を検証する観点がない
   - 結果として、未確定値が品質ゲートを通過してしまう

3. **「下流で決める」という暗黙の前提が正当化されていない**
   - 一部の値は意図的に下流へ委譲する場合がある（例: DB カラム型は詳細設計で決める）
   - しかし、ビジネス要件レベルの値（表示件数、表示期間）や、アーキテクチャ判断（アップロード方式）が下流委譲される正当な理由がない
   - 「委譲する場合は理由を明記する」というルールが work-breakdown にない

## 影響

- 詳細設計者が上流未確定の判断を代行することになり、設計判断の責務が曖昧になる
- task-plan の「判断ポイント」セクションが本来不要な判断の受け皿になり、task-plan の役割が肥大化した原因の一つ
- テンプレートとして再利用した際、別プロジェクトでも同じ値の流出が発生する

## 提案

本 issue の完了条件は「未確定値を直す」だけではなく、**再発を防ぐ work-breakdown テンプレートの運用設計改善**を含む。

### 1. work-breakdown テンプレートの改善

- **上流 Step の完了条件に「未確定値ゼロ」を追加**: 成果物内に N, TBD, 未定, 要検討 等の変数が残っていないことを完了条件とする
- **下流委譲ルールの明文化**: 意図的に下流 Step へ値の確定を委譲する場合は、成果物内に「§X.X は Step N で確定予定（理由: ...）」と明記する。理由なき委譲を禁止する
- **レビュー観点に「未確定値チェック」を追加**: 全 Step 共通のレビュー観点として「成果物内に未確定の変数・プレースホルダーが残っていないか」を追加する

### 2. 既存成果物の修正

- requirements.md DASH-005: `N ヶ月` → 具体値に確定
- screens.md: ダッシュボードの表示件数・表示期間を確定
- screens.md: 明細の編集方式を確定
- architecture.md: アップロード方式を確定、S3 キー命名規則を追記

### 3. 完了条件

- [x] work-breakdown テンプレートの Step 0〜4 に「未確定値ゼロ」の完了条件が追加されている
- [x] work-breakdown テンプレートに下流委譲ルールが明文化されている
- [x] work-breakdown テンプレートのレビュー観点に未確定値チェックが追加されている
- [x] 既存成果物の未確定値が全て解消されている（requirements.md, screens.md, architecture.md）

## 解決内容

### Part A: テンプレート改善（ai-dev-framework）
- `templates/work-breakdown-template.md` に「未確定値ゼロの原則」セクションを追加（下流委譲ルール含む）
- Step 0〜4 の work-breakdown に完了条件「未確定値ゼロ」を追加
- Step 0〜4 の work-breakdown にレビュー観点「未確定値チェック」を追加

### Part B: 既存成果物修正（dev-journal）
- `requirements.md` DASH-005: `直近 N ヶ月` → `直近 3 ヶ月`
- `screens.md`: 月別サマリー `N ヶ月` → `3 ヶ月`（3箇所）、直近レポート件数 `5 件` 追加（2箇所）、明細編集方式（スライドパネル）追記
- `architecture.md`: アップロード方式（API プロキシ、multipart/form-data）付記、S3 キー命名規則追記
