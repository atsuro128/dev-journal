# architecture.md のディレクトリツリー粒度が不適切

## 発見日
2026-04-16

## カテゴリ
architecture

## 影響度
中

## 発見経緯
user-report — issue 112（DTO 分離）対応時に、architecture.md のツリー更新が毎回必要になる構造的問題をユーザーが指摘。

## 関連ステップ
Step 3（アーキテクチャ設計）/ Step 8（基盤構築）

## ブロッカー
なし

## 問題

`architecture.md` のディレクトリツリー（§3.1 / §4.1）が具体的なファイル名レベルで記載されている。Step 3（アーキテクチャ設計）の時点では機能仕様が固まっておらず、具体的なファイル構成は確定できない。結果として、実装が進むたびにツリーが陳腐化し、繰り返し乖離 issue が発生している（issue 110, 112）。

### 根本原因

設計フェーズごとの「書くべき粒度」が定義されていない。

| Step | 書くべき粒度 | 現状 |
|------|------------|------|
| Step 3（アーキテクチャ） | レイヤー構成・依存ルール・パッケージ責務 | ファイル名まで列挙 |
| Step 8（基盤構築） | 具体的なディレクトリ・ファイル構成 | 正本が存在しない |

### 構造的な問題

- architecture.md がファイル名列挙を持つため、実装変更のたびに追従更新が必要
- Step 8 以降の「コードが正本」という原則が明文化されていない
- 基盤構築時点でのディレクトリ構成スナップショットが成果物として定義されていない

## 影響

- 実装変更のたびに architecture.md の更新 issue が発生する（issue 110, 112 が実例）
- 設計書とコードの乖離が常態化するリスク
- レビュー工程で本質的でないツリー差分が繰り返し指摘される

## 提案

### 1. architecture.md のツリーをパッケージ粒度に抽象化

ファイル名の列挙を削除し、パッケージ（ディレクトリ）単位の責務記述に置き換える。

Before:
```
internal/domain/
├── entity.go
├── report.go
├── dto.go
...
```

After:
```
internal/domain/     # エンティティ・値オブジェクト・リポジトリIF
internal/service/    # ユースケース・DTO・ビジネスロジック
internal/handler/    # HTTP ハンドラ・リクエスト/レスポンス変換
internal/repository/ # データアクセス実装
```

### 2. Step 8 の成果物にディレクトリ構成ドキュメントを追加

基盤構築完了時点のディレクトリ構成スナップショットを Step 8 の成果物として作成する。これが実装レベルの構成の正本となる。

- 配置先: `dev-journal/deliverables/docs/80_foundation/directory_structure.md`
- 内容: `tree` コマンド出力 + 各ディレクトリ・主要ファイルの責務説明
- 更新タイミング: 構造変更を伴う PR マージ時（任意）

### 3. 設計粒度ルールの明文化

各 Step で記述すべき粒度を work-breakdown またはテンプレートに明記する。

## 完了条件

- architecture.md のディレクトリツリーがパッケージ粒度に抽象化されている
- Step 8 の成果物に `directory_structure.md`（実装レベルのディレクトリ構成正本）が追加されている
- architecture.md に「Step 8 以降のファイルレベル構成はコード（または directory_structure.md）を参照」と明記されている
