# アーキテクチャ設計書にスケーラビリティのトレードオフが未記載

## 発見日
2026-04-02

## カテゴリ
architecture

## 影響度
中

## 発見経緯
user-report

## 関連ステップ
Step 3（アーキテクチャ設計）

## 問題

Step 3 でマルチテナント方式として RLS を採用したが、RLS 方式の限界とスケール時の移行パスがアーキテクチャ設計書に記載されていない。トレードオフの記載を追加し、設計判断の根拠を明確にする必要がある。

### 記載が不足している限界

1. **コネクションプール上限**: デフォルト4本（CPU数）。数千ユーザー規模で不足する可能性があるが、拡張方針が未定義
2. **RDS Proxy との相性**: RLS の `set_config` はセッション変数を使用。RDS Proxy のコネクション共有でテナント ID が残留するリスクがある
3. **リードレプリカ非対応**: 全クエリが1つの DB プールに向いており、読み取り分散の設計がない
4. **テナント別 DB 分離時の大規模改修**: 将来スケールのために万が一 DB 分離が必要になった場合、既存データの移行・アプリの接続ルーティング全面改修・RLS の撤去が必要。リリース後に発覚した場合、既存データの移行戦略（ダウンタイム or 二重書き込み）も必要になる

## 影響

- 将来のスケール時に場当たり的な対応になる

## 提案

アーキテクチャ設計書に以下を追記する:

1. **RLS 方式のトレードオフ**: 採用理由（MVP では運用コスト最小化を優先）と限界を明記
2. **スケール時の移行パス**: 段階的な拡張手順を定義
   - Phase 1: コネクションプール数の調整（環境変数のみ、コード変更なし）
   - Phase 2: RDS Proxy / PgBouncer 導入（RLS との相性検証が前提）
   - Phase 3: リードレプリカ導入（Repository 層に読み書き振り分けの改修が必要）
   - Phase 4: テナント別 DB 分離（大規模改修。データ移行戦略の策定が必要）
3. **各段階で必要な改修規模の見積もり**

---

## 解決内容
ADR-0003 に「スケール時の既知の限界」セクションを追加（コネクションプール上限、RDS Proxy 相性、リードレプリカ非対応、テナント別 DB 分離、集計クエリ性能）。architecture.md は ADR-0003 への参照リンクのみに修正。参照表の ADR-0003 説明を「詳細設計」→「判断根拠」に修正。
