## 合意事項（2026-04-15）

### 監査の 2 軸

本 issue は以下 2 つの監査軸を**単一のマトリクスで統合的に実施**する:

1. **ロール別表示制御**（軸 1）: 各画面の UI コントロール × 4 ロールの表示/非表示
2. **レスポンシブ挙動**（軸 2）: 各画面 × 3 viewport（PC / タブレット / スマホ）の表示・振る舞い

理由: 監査の実務では「画面を開いて確認する」動作が共通しており、別々に実施すると二度手間。screens.md の画面マトリクスを一度抽出すれば、列に「ロール表示」「レスポンシブ挙動」を両方入れて一元管理できる。

### 決定事項

#### 決定 1: 監査対象の網羅範囲 = 全画面 × 全ロール × 全 viewport
- 全 `screens/*.md` を対象
- 全 4 ロール（Member / Approver / Accounting / Admin）
- 全 3 viewport（PC 1280px / タブレット 900px 前後 / スマホ 375px）
- md ブレークポイント切り替えの境界値検証を含む

#### 決定 2: 検証の振り分け = 重要度で自動 / smoke を分離
- **クリティカル操作**（副作用あり、誤操作で取り返しがつかない / UX 破綻のリスク大）→ **自動テスト + smoke 両方**
  - 例: 添付削除ボタン、申請承認/却下ボタン、支払確定ボタン、テナント設定変更、md ブレークポイント切り替え
- **情報表示系**（見えても実害小、見えない時のみ問題）→ **smoke のみ**
  - 例: グローバルナビメニュー、画面内リンク、フィルタドロップダウン、テーブル列崩れ、カード化
- **その他**（実装レベルの細部）→ **自動テストのみ**
  - 例: タップ領域サイズ、スクロール挙動、カーソル形状

副作用の定義: サーバー状態を変える操作（CREATE / UPDATE / DELETE）

#### 決定 3: フェーズ分け = リサーチと実装を 2 つの sub-issue に分割

本 issue は「方針合意の親 issue」として残し、以下 2 つの sub-issue を起票して順次対応:

- **sub-issue 1（リサーチ）**: 棚卸し + 既存テスト確認 + ギャップ分析 + マトリクスドキュメント化
  - 成果物: `dev-journal/deliverables/docs/60_test/ui_coverage_matrix.md`（仮称）
  - screens/*.md 全画面の UI コントロール × ロール × viewport を抽出
  - 既存テスト（component test / smoke）でのカバー有無を明示
  - ギャップリストを整理
- **sub-issue 2（実装）**: リサーチ結果を元にテスト追加 + smoke 拡充 + 個別バグ修正
  - 決定 2 の振り分けに従ってテスト実装
  - 監査過程で発見された実装バグは個別 issue として起票・修正

フェーズ分割の理由: リサーチ結果を見てから実装の範囲を再評価できる。大規模な監査なのでスコープが膨らみやすく、段階的に進めないとブロッカー化する。

## 問題の正体

### セキュリティ層と UX 層の境界

| 層 | 守るもの | 現状の網羅手段 |
|---|---|---|
| API 認可 | 攻撃者が直接 API を叩いても弾く | BE 自動テスト（CRS-021〜054 RBAC マトリクス）で網羅予定 |
| UI 表示制御 | 権限のないユーザーにボタン/リンク/メニューを見せない | **未網羅**（smoke_check.md スコープ外、フロント component test の網羅実態不明） |

API 認可が守られていれば「権限のないボタンを押しても 403」となるためセキュリティ的には致命的ではない。しかし UI レベルで「見えてはいけないボタンが見える」状態は:

1. **UX バグ**: ユーザーが押せると思って操作 → 403 トースト → 困惑
2. **情報開示**: ボタンの存在からシステムの内部仕様（誰がどの操作を実行できるか）が推測される
3. **設計書との乖離**: 詳細設計書（`screens/*.md`）には「ロール × ボタン表示」マトリクスが定義されているが、実装がそれに従っていない可能性を検出できない

### smoke_check.md の責務範囲

`smoke_check.md` §1 で明示的にスコープ外:

```
### 含まない観点（自動テスト or UAT で担保）
- 認可ロジック（ロール × リソース） → BE 自動テスト + RBAC マトリクスでカバー
```

ただし「BE 自動テスト + RBAC マトリクスでカバー」とあるが、これは API 層のカバレッジ。UI 表示制御は別軸であり、明示的に責任を持つ場所がない。

### レスポンシブ SMK のカバレッジ不足

現状 SMK-060〜063 の 4 項目しかなく、以下が未検証:

1. **viewport の抜け**
   - タブレット（md 境界付近、900-1200px）が一切ない
   - md ブレークポイント切り替え（常時 Drawer ↔ ハンバーガー）の境界値検証なし
   - iPad 横（1024px）と縦（768px）の挙動差なし

2. **ロールの抜け**
   - Accounting / Admin のモバイル検証ゼロ
   - テナント設定、管理画面（全レポート、月次集計）のモバイル未確認

3. **画面の抜け**
   - パスワードリセット、エラー画面（404, 403）、Approver/Accounting 一覧、Admin 全画面

4. **操作パターンの抜け**
   - 添付ファイルのアップロード / ダウンロード / プレビュー のモバイル挙動
   - ワークフロー操作（却下 / 支払確定）
   - エラートースト / ダイアログのモバイル表示
   - DataGrid / Table の横スクロール

5. **コンポーネント系の抜け**
   - Drawer permanent / temporary 切り替え境界（md ブレークポイント）
   - Modal / Dialog のモバイル表示
   - Toast 位置（スマホ bottom / top）

ざっくり追加必要な SMK 項目は 20〜30 件規模。現状は「やってる感」レベル。
