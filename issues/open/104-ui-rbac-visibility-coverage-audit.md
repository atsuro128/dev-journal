## 問題の正体

### セキュリティ層と UX 層の境界

| 層 | 守るもの | 現状の網羅手段 |
|---|---|---|
| API 認可 | 攻撃者が直接 API を叩いても弾く | BE 自動テスト（CRS-021〜054 RBAC マトリクス）で網羅予定 |
| UI 表示制御 | 権限のないユーザーにボタン/リンク/メニューを見せない | **未網羅**（smoke_check.md スコープ外、フロント component test の網羅実態不明） |

API 認可が守られていれば「権限のないボタンを押しても 403」となるためセキュリティ的には致命的ではない。しかし UI レベルで「見えてはいけないボタンが見える」状態は:

1. **UX バグ**: ユーザーが押せると思って操作 → 403 トースト → 困惑
2. **情報開示**: ボタンの存在からシステムの内部仕様（誰がどの操作を実行できるか）が推測される
3. **設計書との乖離**: 詳細設計書（`screens/*.md`）には「ロール × ボタン表示」マトリクスが定義されているが、実装がそれに従っていない可能性を検出できない

### smoke_check.md の責務範囲

`smoke_check.md` §1 で明示的にスコープ外:

```
### 含まない観点（自動テスト or UAT で担保）
- 認可ロジック（ロール × リソース） → BE 自動テスト + RBAC マトリクスでカバー
```

ただし「BE 自動テスト + RBAC マトリクスでカバー」とあるが、これは API 層のカバレッジ。UI 表示制御は別軸であり、明示的に責任を持つ場所がない。

### ロール別 SMK 件数の偏り

| ロール | SMK 件数 |
|---|---|
| Member | 44 |
| Approver | 7 |
| Accounting | 3 |
| Admin | 2 |

Approver / Accounting / Admin は導線の入口確認（SMK-001〜009）程度しかカバーされていない。

### 設計書での定義例

`dev-journal/deliverables/docs/50_detail_design/screens/report-detail.md` §（添付ファイルロール別表示）:

```
| 添付アップロード | 表示 | 非表示 | 非表示 |
| 添付削除ボタン   | 表示 | 非表示 | 非表示 |
| 添付ダウンロード | 表示 | 表示  | 表示  |
```

このようなマトリクスが各 `screens/*.md` に存在するはずだが、これを検証する仕組みが smoke / 自動テスト / 手動チェックのどこにあるかが不明瞭。

## 対応スコープ（合意確認）

実装着手前に下記 3 点をユーザーと合意する。

### 論点 1: 監査対象の網羅範囲

- 案 A: **全画面 × 全ロール** で設計書のマトリクスを抽出 → 実装と照合（網羅）
- 案 B: **権限差が出る画面のみ**（テナント設定、承認待ち、支払待ち、添付操作等）に絞る
- 案 C: **添付ファイル領域だけ**（今回の発見起点に限定、最小スコープ）

→ 推奨は **案 A**。他の漏れも一網打尽にする目的なら網羅すべき。

### 論点 2: 検証の振り分け

設計書マトリクスを誰が検証するか:

- 案 A: 全部 smoke_check.md に追加（手動目視）
- 案 B: 全部フロント component test に追加（自動）
- 案 C: 重要度で振り分け
  - **クリティカル（誤操作で副作用がある操作）**: フロント自動テスト + smoke 両方
  - **情報表示系（ナビ、リンク）**: smoke のみ
  - **その他**: フロント自動テストのみ

→ 推奨は **案 C**（自動と手動の役割分担を明確にする）。

### 論点 3: 監査の進め方

- ステップ 1: **棚卸し** — `screens/*.md` 全画面のロール × ボタンマトリクスを抽出 → CSV/markdown テーブルで可視化
- ステップ 2: **既存テスト確認** — フロント component test / smoke_check.md でカバーされている項目を marking
- ステップ 3: **ギャップ分析** — 未カバー項目をリストアップ
- ステップ 4: **テスト追加 or smoke 拡充** — ギャップを論点 2 の方針で埋める
- ステップ 5: **実装との突き合わせ** — 検証実行 → 乖離を実装バグとして個別 issue 起票

ステップ 1〜3 を 1 つのサブ issue（リサーチ）、ステップ 4〜5 を別のサブ issue（実装）に分けることも検討。
