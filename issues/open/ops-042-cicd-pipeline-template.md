## 問題

CI/CD パイプラインの構成判断（ステージ構成、トリガー条件、ブランチ保護ルール、デプロイ方式）は、プロジェクトごとに毎回決め直す必要がない。ブランチ運用（ops-039）や hooks 運用（ops-037）と同様に、一度決めればテンプレート化できる運用判断である。

現状、Step 8 work-breakdown の CI/CD カテゴリに判断ポイントとして残されているが、ai-dev-framework の運用テンプレートとしてまとまっていない。

## 影響

- Step 8 の CI/CD タスクで毎回同じ判断を繰り返すことになる
- プロジェクト間で CI/CD 構成がばらつく
- 初回プロジェクトでの判断が次回以降に引き継がれない

## 提案

以下の判断を ai-dev-framework の運用テンプレートとして定義する:

- パイプラインのステージ構成（lint → test → build）
- トリガー条件（PR 作成時、main マージ時、手動実行）
- ブランチ保護ルール（main 直接プッシュ禁止、CI 通過必須）
- マージ前チェック（テスト通過、ビルド成功、lint エラーゼロの必須化）
- デプロイ方式（自動 vs 手動承認）

ops-037（hooks）、ops-039（ブランチ運用）と合わせて、開発運用テンプレート群として整備する。

### 完了条件

- [ ] CI/CD 構成テンプレートが ai-dev-framework 内に作成されている
- [ ] Step 8 work-breakdown の CI/CD カテゴリから、テンプレート参照の導線がある

---

## 解決内容

CI/CD 構成判断はフレームワークの例外資料ではなく、プロジェクトの正本（architecture.md）に記載すべきと判断。以下の対応を実施。

1. **architecture.md に §8 CI/CD パイプライン構成を追記** — PR パイプライン（lint → test → build）、main マージパイプライン、ブランチ保護ルール、テスト実行環境を定義
2. **work-breakdown テンプレートに CI/CD 構成判断を設計フェーズの必須項目として追加** — 実装フェーズへの判断持ち越しを防止
3. **8-8 チケットの入力を architecture.md §8 に変更** — 判断ポイントを削除し、設計通りに実装するだけの状態にした
4. **ai-dev-framework/operations/cicd-pipeline.md は作成せず** — 例外資料を作らない方針
