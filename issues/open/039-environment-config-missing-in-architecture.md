# architecture.md に環境構成の定義が欠落

## 発見日
2026-03-31

## カテゴリ
architecture

## 影響度
高

## 発見経緯
user-report

## 関連ステップ
Step 3（アーキテクチャ設計）、Step 11（システムテスト・UAT）

## 問題

architecture.md に環境の種類（開発 / ステージング / 本番）と用途・使い分けの定義がない。
実務を想定したポートフォリオとして、環境構成は設計段階で定義されているべき内容。

## 影響

- Step 11 のデプロイ環境構築タスク（11-A）で、どの環境を構築すべきか判断できない
- deploy.yml のデプロイ先が未定義のまま
- テストをどの環境で実施するかの判断基準がない
- Step 11 の 11-A（デプロイ環境構築）のデプロイ先がステージングか本番か未確定。実務ではステージングにデプロイ → システムテスト → UAT → 本番デプロイの流れが標準であり、環境構成の定義なしには Step 11 のタスク構成・順序が確定しない

## 提案

1. architecture.md に環境構成セクションを追加し、以下を定義する:
   - 環境の種類と用途（開発 / ステージング / 本番）
   - 各環境の構成差分（インスタンスサイズ、DB 構成等）
   - デプロイパイプラインと環境の対応関係
2. テンプレート（`ai-dev-framework/templates/docs-v2/30_arch/architecture.md`）にも環境構成セクションを追加し、今後の再利用に備える
3. 環境構成の追加により Step 3 の work-breakdown（`step3-architecture.md`）の成果物・完了条件に影響がないか確認し、必要なら修正する
4. 環境構成の定義結果に基づき、Step 11 の work-breakdown（`step11-system-test.md`）を見直す:
   - 11-A のデプロイ先をステージング/本番のどちらにするか確定
   - ステージング → 本番の昇格フローが必要ならタスク追加を検討
