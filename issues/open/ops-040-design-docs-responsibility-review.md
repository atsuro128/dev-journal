# 設計資料の責務定義・再レビュー・再編

## 発見日
2026-03-26

## カテゴリ
project-management

## 影響度
中

## 発見経緯
user-report

## 関連ステップ
Step 1〜6（全設計成果物）

## 問題
成果物ツリー構造のみを確認した外部レビューで、以下 4 点の懸念が指摘された。

1. **文書間の責務重複** — 同じ情報が複数文書に散在している可能性
2. **非機能の散在** — 非機能要求が requirements / architecture / security 等に分散
3. **トレーサビリティ不足** — 要件→設計→テストの追跡線が不明確
4. **運用系の弱さ** — SaaS 運用に必要な監視・障害対応・リリース方針の欠落

ただしこの指摘はファイル名・配置構造のみに基づいており、中身を読んだ上での判定ではない。
妥当性の検証を含めて段階的に対応する。

## 影響
- ガイドを汎用開発テンプレートとして再利用する際に、文書の責務・境界が曖昧だとレビュー観点が定まらない
- 下流工程（Step 7 以降）で設計資料を参照する際の効率に影響する可能性

## 提案

以下の 5 フェーズで段階的に対応する。先に文書構造を壊さず、責務と境界を明確にしてから内容改善・再編に進む。

**基本方針**:
- 見た目の整理やファイル数削減を目的にしない
- 各文書が何の正本なのかを明確にし、次工程の判断を迷わせないことを重視
- 上流は考えやすく、下流は実装しやすい粒度を目指す

**判断基準**:
- 次工程の担当者が、重要判断を勝手に補完しなくて済む
- 正本が明確である
- 文書間の境界が明確である
- レビューしやすい / 将来の変更に追従しやすい

---

### Phase 1: 現状分析と責務定義

**目的**: 各文書が何を書く文書か、何を書かない文書か、他文書との境界はどこかを明確にする。

**実施内容**: 既存の各文書に対して以下を整理する。
- 文書名 / 主目的 / 扱うべき内容 / 扱わない内容
- 依存先・参照先の文書
- 重複しやすい相手文書
- 正本として持つべき情報

**禁止事項**:
- 大規模な文書統合を行わない
- 文書名変更を先行して行わない
- 内容を削除して整えたつもりにならない
- 責務が固まっていない段階でファイル移動しない

**チェックリスト**:
- [ ] 各文書の責務定義表を作成
- [ ] 重複・散在・不足の初期分析
- [ ] 外部指摘 4 点の妥当性判定

**出力物**: 現状の問題点一覧 / 各文書の責務定義表 / 重複・散在・不足の初期分析結果

---

### Phase 2: 内容改善

**目的**: 文書構造は大きく崩さず、4 つの主要問題を改善する。

#### 2-1. 文書間の責務重複の改善
- どの文書が何の正本かを明確にする
- 重複がある場合は「正本 / 要約 / 参照」の区別を明示する
- どの記述が移設対象か、どの重複が意図的な要約として許容されるかを判断する

#### 2-2. 非機能の散在の改善
非機能要求が architecture / security / monitoring / requirements 等に散っている場合:
- 非機能要求の正本配置先を決定する
- 各設計文書で扱うべき詳細と参照関係を整理する
- 確認観点: 性能 / 可用性 / セキュリティ / 監査性 / 可観測性 / 保守性 / 拡張性 / バックアップ・復旧 / テナント分離保証 / ログ保持・運用方針

#### 2-3. トレーサビリティ不足の改善
以下の追跡線を整理する（必要に応じて traceability.md を新設）:
- 要件 → ユースケース
- 要件・業務ルール → ドメイン
- 要件・ドメイン → アーキテクチャ判断
- ユースケース・ドメイン → 画面 / API / 認可 / DB
- 要件・設計 → テスト

#### 2-4. 運用系の弱さの改善
SaaS として最低限必要な運用観点を補う:
- 監視対象 / アラート条件 / ログ・監査ログ方針
- 障害時の一次切り分け / バックアップ・リストア
- リリース・ロールバック / 環境差分 / シークレット管理

**禁止事項**:
- 構造再編を主目的にしない
- いきなりファイルを大量統合しない
- 記述を減らすこと自体を成果にしない
- トレーサビリティを頭の中だけで済ませない

**チェックリスト**:
- [ ] 責務重複の正本/参照整理
- [ ] 非機能要求の配置整理
- [ ] トレーサビリティの補強
- [ ] 運用観点の補強

**出力物**: 4 観点ごとの改善結果 / 修正対象文書一覧 / 正本・参照の整理結果

---

### Phase 3: 文書構造の再編・統合

**目的**: 内容が整理された後、上流は粗めに、下流は実装・検証しやすい粒度に文書構造を再設計する。

**上流文書の方針**（要件〜基本設計）:
- 同一人物が行き来しながら読む前提で、統合寄りにする
- 強く依存する文書は統合候補とする
- 分割理由が明確でないファイルは統合候補とする

**下流文書の方針**（詳細設計・テスト）:
- 画面ごと / 機能ごと / API ごと / 横断関心ごとで整理する
- 画面 1 つを実装するのに資料を過剰に行き来しないようにする

**再編ルール**:
- 統合理由・分割維持理由を明示する
- 統合元と統合先を記録する
- 正本がどこへ移ったかを記録する
- 旧構成との差分が追えるようにする（refactor-log を作成）

**禁止事項**:
- 統合して見た目を減らしただけで満足しない
- 正本を曖昧にしたまま文書を減らさない
- 上流と下流を同じ分割思想で扱わない

**チェックリスト**:
- [ ] 上流文書の統合検討
- [ ] 下流文書の粒度最適化
- [ ] 変更履歴の記録（refactor-log）

**出力物**: 新しい文書構成案 / 各文書の責務一覧 / 統合・分割・改名の変更履歴

---

### Phase 4: 各章の必須内容レビュー

**目的**: 再編後の各章について、その章に入っているべき内容が揃っているか確認する。

**レビュー観点**:
- その章の責務に沿った内容になっているか
- 他章の内容を重複して持ちすぎていないか
- 必須項目に抜け漏れがないか
- 下流工程で必要な判断材料になっているか
- 参照関係が自然か

**章ごとの確認項目**:

| 章 | 確認項目 |
|----|---------|
| 要件定義 | 背景 / 目的 / スコープ / アクター / ユースケース / 業務ルール / 業務フロー / RBAC / 機能要求 / 非機能要求 |
| ドメイン設計 | 主要概念 / エンティティ・値オブジェクト / 関係 / 不変条件 / 状態遷移 / 業務ルールとの対応 |
| アーキテクチャ設計 | 全体構成 / レイヤ責務 / 認証・認可 / テナント分離 / 外部連携 / 可観測性 / 技術選定理由 / ADR との関係 |
| 基本設計 | 画面一覧 / 各画面の目的 / 全体 UI フロー / 主要遷移 / 権限による差分 / 詳細設計への橋渡し |
| 詳細設計 | 画面・API 仕様 / 認可 / DB との対応 / バリデーション / ファイル制御 / 例外時挙動 / 監視・ログ・監査観点 |
| テスト設計 | テスト方針 / 正常系・異常系 / テナント分離 / 認証・認可 / ワークフロー / 横断関心 / 要件・設計との対応 |

**チェックリスト**:
- [ ] 章ごとの責務充足確認
- [ ] 抜け漏れ一覧と追加修正

**出力物**: 章ごとのレビュー結果 / 抜け漏れ一覧 / 追加修正提案

---

### Phase 5: ユースケース横断レビュー（可能なら実施）

**目的**: 章単位レビューだけでは見落としやすい、ユースケース単位の整合を確認する。

**レビュー観点**: 主要ユースケースごとに以下が揃っているか確認する。
- 要件定義に存在するか / 業務ルールがあるか / ドメインで説明できるか
- アーキテクチャ上の実現方針があるか
- 画面仕様があるか / API 仕様があるか / 認可があるか
- DB 設計に反映されているか / テストがあるか / 運用・監査観点があるか

**チェックリスト**:
- [ ] 主要ユースケースごとの設計整合確認

**出力物**: ユースケースごとの整合レビュー結果 / 不足箇所一覧

---

## 備考
- Phase 1 で妥当性が低いと判断された指摘は、理由を記録して対応スキップとする
- Step 7 の実装作業とは独立して進行可能
- 最終目的はガイドの汎用テンプレート化に向けた責務明確化とレビュー観点の絞り込み
