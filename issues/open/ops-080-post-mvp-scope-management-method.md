## 問題

現状の ai-dev-framework には Step 0〜11（MVP 完成まで）の work-breakdown は整備されているが、**Step 11-F UAT 完了後（= MVP リリース後）のワークフローが定義されていない**。

結果として、以下の構造的問題が発生している:

1. **MVP スコープ外と判断した品質保証項目が消失するリスク** — 現状では「MVP スコープ外」と判断した項目を記録する場所・読み返すトリガーがなく、実質的に忘れ去られる
2. **運用・改善フェーズのタスク管理方式が未定** — 各 issue を `open/` に溜めるだけでは優先度・読むタイミングが不明
3. **読ませるトリガー不在** — セッション開始時に MVP スコープ外事項を確認する仕組みがない

### 本 issue が起票された契機

issue 073（work-breakdown の保証種別 5 分類 → 8 分類統一）の対応中に、以下の観点が「traceability.md で『カバー済み』扱いだが実ケースが不足している」状態として顕在化した:

- CORS / セキュリティヘッダ
- タイムゾーン（UTC 保存 / JST 表示）
- /health エンドポイント
- 真の競合シナリオ（楽観ロック 409 レース再現）
- 二重送信防止

これらを「MVP スコープ外」と判断したが、**では具体的にどこで管理するか？** という問いに答える仕組みが存在しなかった。

暫定的な対応として、5 観点の詳細を `issue 081` に一括記録している（章立てで §1〜§5）。本 issue (ops-080) で管理方式を決定した後、issue 081 を適切な管理先へ移動または統合する。

## 影響

- MVP リリース後、「何を次に対応すべきか」を把握する仕組みが存在しない
- Step 11-F UAT 完了時に、指揮役・ユーザー双方が post-MVP の TODO を見失うリスク
- 今後同様の「スコープ外」判断が発生するたびに、記録先・追跡方式の議論を繰り返すコスト
- 既存の issue 管理フロー（`open/` → `pending-review/` → `resolved/`）だけでは MVP スコープ外事項を自然に扱えない

## 提案

### 作業内容

本 issue の対応では、以下を段階的に行う。

#### 1. 選択肢の整理

考えられる管理方式を整理し、比較検討する。候補:

- **(X) Step 12 以降を work-breakdown に新設** — 「MVP リリース後の運用・改善」フェーズを定義
- **(Y) `dev-journal/issues/post-mvp/` サブフォルダ新設** — 既存の issue ライフサイクルと並列
- **(Z) `dev-journal/post-mvp-backlog.md` 単一ファイル** — 最軽量のチェックリスト
- **(W) 別の案** — ユーザー・指揮役で議論

#### 2. 比較観点

- 既存の work-breakdown / issue ワークフローとの整合性
- 起票・更新・読み返しのコスト
- 読ませるトリガー（セッション開始時 / UAT 完了時 / 新ルールファイル / 他）
- 個別追跡可能性（issue レベルの粒度か、ファイル内セクションの粒度か）
- 要件遡及が必要な項目（要件 ID 不在の観点）の扱い方

#### 3. 決定と実装

選択した方式に応じて以下を実装:

- フォルダ・ファイル構造の新設
- テンプレートの作成（必要なら）
- `CLAUDE.md` / `.claude/rules/` への参照追加
- `/session-start` スキルへの統合（必要なら）
- `progress.md` への「Post-MVP バックログ」セクション追加（必要なら）
- Step 11-F 完了条件への追記（必要なら）

#### 4. 既存 issue の移動・統合

- issue 081（MVP スコープ外のテスト観点一括管理、§1〜§5）を決定された管理方式に沿って移動または分割
- issue 073 の `traceability.md` 備考欄で「issue 081 §N で追跡。管理方式は ops-080 で検討予定」と記載した参照を、決定後の実パスに更新

### スコープ

**本 issue に含めること**:
- 管理方式の検討・決定
- 管理方式の実装（フォルダ構造・テンプレート・トリガー）
- 既存 issue 081 の移動・統合
- issue 073 の traceability.md 備考欄の再更新

**本 issue に含めないこと**:
- 各テスト観点の実装（issue 081 の §1〜§5 で追跡）
- 要件遡及（issue 081 §4 §5 の課題）
- Step 12 以降の work-breakdown の具体的な内容策定（方式として X を採用した場合の別タスク）

### 対応タイミング

**本セッションでは起票のみ**。対応は別セッションで改めて時間を取り、選択肢の比較検討を十分に行った上で決定する。
