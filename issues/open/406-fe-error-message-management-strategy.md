# FE エラーメッセージの管理方式が未定義

## 発見日
2026-04-06

中

## 発見経緯

なし

## 問題

フロントエンドのエラーメッセージ管理方式が設計成果物で定義されていない。具体的には:

1. エラーメッセージ文字列をどこに保持するか（ハードコード / 定数ファイル / i18n ライブラリ）
2. API エラーコード（INVALID_CREDENTIALS, RATE_LIMIT_EXCEEDED 等）からユーザー向けメッセージへの変換ロジックをどこに置くか
3. SEC-011 のようにセキュリティ要件でフロント側でメッセージを上書きするケースの扱い

state-management.md §6 でエラーの分類と処理方針（Snackbar / Alert / リダイレクト）は定義済みだが、メッセージそのものの管理方式が抜けている。

## 影響

- Step 10（機能実装）でエラーメッセージの扱いが実装者依存になる
- 画面ごとにバラバラな管理方式になるリスク
- SEC-011 の統一エラーメッセージが正しく適用されない可能性

## 提案

state-management.md §6 にサブセクション「エラーメッセージ管理方針」を追加し、以下を定義する:

1. メッセージの保持場所（例: `constants/errorMessages.ts` に集約）
2. API エラーコード → 表示メッセージの変換マッピング
3. セキュリティ要件によるメッセージ上書きルール
