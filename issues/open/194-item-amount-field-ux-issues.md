## 問題

### A. 全角文字入力（IME 未確定）でフォーカス移動すると残置

**発生条件**:
1. 明細追加ダイアログの「金額」フィールドにフォーカス
2. 全角入力（例: 「あいう」「１２３」）を IME で入力
3. **IME で確定せずに**、別フィールド（日付・カテゴリ・摘要）にフォーカス移動
4. 金額フィールドに全角文字が残ったまま表示される

**期待挙動**:
- 通常の文字入力 → フォーカスアウト時には自動削除される（数値専用フィールドの数値以外除去ロジックが動く）
- IME 未確定状態でフォーカス移動した場合も同様に削除されるべき
- もしくは、IME 確定を強制してから除去ロジックを走らせる

**実害**:
- バリデーションエラーで止まるため業務継続には支障なし
- ただし、ユーザーから見ると「フォーカス移動の前後で挙動が変わる」のは違和感

### B. 大金額入力をそもそもエラー扱いとするか（要相談）+ エラー文言の不自然さ

**論点 B-1: 金額上限の必要性（要相談）**

- 現状: 極端に大きい数値はエラーになる（バリデーションで弾かれる）
- **そもそも、経費精算 SaaS で金額に上限を設ける必要があるか？** を要件レベルで再検討
  - 「経費精算で 1,000 万円の領収書なんてあるか？」という業務観点
  - 「不正入力を弾く」セキュリティ観点 vs 「正当な高額経費を受け付ける」業務観点
- 設計成果物（`policies.md` / `usecases.md` / 要件 ITM-001〜005）に明確な金額上限規定が無い場合、ここで決める必要がある
- 仕様候補:
  - 案 1: 上限なし（金額は ITM-001 で「正の整数」のみ、上限なし）
  - 案 2: 業務的に妥当な上限（例: 1 億円 = ¥100,000,000）
  - 案 3: DB 型の物理上限のみ（INTEGER の場合 ¥2,147,483,647、BIGINT なら実質無制限）
- DB 型と FE バリデーションを整合させる必要あり（`db_schema.md` の `expense_items.amount` 型を確認）

**論点 B-2: エラーメッセージの不自然さ**


**現象**:
- 金額フィールドに極端に大きい数値（例: 桁数オーバー）を入力
- エラーメッセージ: **「円単位の整数で入力してください」**

**問題**:
- このメッセージは「小数点不可」「数値以外不可」を意図したものだが、「極端に大きい数値」に対するメッセージとしては**意味が伝わらない**
- ユーザーは「整数で入れているのに何が悪いのか」と混乱する
- 適切なメッセージ例:
  - 「金額が上限を超えています（上限: ¥X,XXX,XXX）」
  - 「金額の桁数が大きすぎます」
  - 「金額は X 桁以内で入力してください」

**設計（`screens/report-detail.md` 等）の確認**:
- 金額の最大値が要件で定義されている場合、その値を明示するメッセージにすべき
- もし最大値が定義されていないなら、まず要件として「金額上限」を決める（仕様ギャップ）

### 共通原因（推定）

- 数値フィールドの実装 (`frontend/src/...`) が、「整数バリデーション」と「範囲バリデーション」を区別していない可能性
- IME 未確定状態の処理は React の `onChange` / `onBlur` イベントタイミングと IME ライフサイクル（compositionstart / compositionend）の組み合わせが必要

## 影響

- バリデーションエラーで業務継続可能、ブロッカーではない
- ただし、UAT で明確に「気になる」と指摘されたため UX 品質として改善余地あり
- post-MVP の入力フォーム品質改善の一環として対応推奨

## 提案

### A の対応案
- IME `compositionend` イベントでも除去ロジックを走らせる
- もしくは `onBlur` で常に最終的な数値除去を実施

### B の対応案
- 金額の上限値を要件として定義（例: ¥9,999,999,999）
- バリデーション分岐:
  - 整数でない → 「整数で入力してください」
  - 上限超過 → 「金額が上限（¥X,XXX,XXX,XXX）を超えています」
  - 下限未満（マイナス）→ 「正の数値を入力してください」

## 着手タイミング

**post-MVP**。MVP リリース後に UX 観点の改善として対応する。
