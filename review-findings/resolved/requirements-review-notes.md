# 要件定義資料 査読結果（修正指示一覧）

## 1. 目的

本書は、以下8ファイルの査読結果をもとに、**実装前に修正しておきたい点**を整理したものである。

- `01_business-overview.md`
- `02_actor-analysis.md`
- `03_business-flow.md`
- `04_business-rules.md`
- `rbac.md`
- `requirements.md`
- `usecases.md`
- `workflow.md`

本資料では、以下の観点で指摘を整理する。

- **整合性**: ファイル間で記述が矛盾していないか
- **明確性**: 読み手によって解釈が分かれないか
- **実装容易性**: API・DB・テストに落とし込みやすいか
- **MVP境界**: MVP / Phase 3 の境界が曖昧になっていないか

## 2. 結論

全体としては、業務理解・業務ルール・RBAC・状態遷移・ユースケースの流れが整理されており、要件定義資料として十分に高品質である。  
特に、**マルチテナント + `tenant_id` + RLS の二重保証**、および **状態遷移をドメイン層で一元管理する方針** は強い。  

一方で、実装前に優先的に修正したいポイントは以下の3点である。

1. **申請系操作の実行者表記が一部で不一致**
2. **Approver の閲覧範囲が「全件」か「担当分のみ」か曖昧**
3. **ダッシュボードがユースケースには存在するが、要件一覧に未昇格**

## 3. 優先度A（実装前に修正推奨）

## A-1. `draft → submitted` の実行者表記を統一する

### 指摘
`04_business-rules.md` では、`draft → submitted` の実行者が **Member（所有者）** になっている。  
しかし他の資料では、Approver/Admin も **自分の経費申請者として申請可能** という前提で統一されている。

### 問題点
このままだと、以下の解釈が分かれる。

- API実装者: Member だけ提出可能と解釈
- フロント実装者: Approver/Admin にも提出ボタンを表示
- テスト担当: どちらが正なのか判断不能

### 影響ファイル
- `04_business-rules.md`
- `03_business-flow.md`
- 必要に応じて `workflow.md` の文言確認

### 修正方針
「提出可能なのは **所有者** であり、所有者のロールは `Member / Approver / Admin` を含む」と統一する。

### 修正文例
#### `04_business-rules.md`
変更前:

```md
| WFL-010 | draft → submitted | Member（所有者） | 明細が1件以上存在すること |
```

変更後:

```md
| WFL-010 | draft → submitted | 所有者（Member / Approver / Admin） | 明細が1件以上存在すること |
```

#### `03_business-flow.md`
`Member（所有者のみ）` という表現があれば、以下へ寄せる。

```md
所有者（Member / Approver / Admin）
```

### 補足
`requirements.md` と `rbac.md` と `usecases.md` はすでに「Approver/Admin も自分の経費申請は可能」という方向でほぼ整合しているため、**ルール資料側を合わせる**のが自然。

---

## A-2. Admin の権限説明を明文化する

### 指摘
`rbac.md` では、Admin は「全データ閲覧権限を持つが、他人のレポートの編集・承認は不可」と書かれている。  
一方で権限マトリクスでは、Admin のレポート作成・編集・削除・提出が許可されており、一見すると矛盾して見える。

### 問題点
厳密には矛盾ではないが、**Admin が“管理者としてできること”** と **Admin が“自分の申請者としてできること”** が分離されていないため、読み手が迷う。

### 影響ファイル
- `rbac.md`
- `04_business-rules.md`
- `requirements.md`（必要なら補足追加）

### 修正方針
以下の原則を明記する。

- **Admin は自分が作成したレポートに限り、申請者として操作可能**
- **他者レポートに対しては閲覧のみ**
- **承認・却下は不可**

### 追記推奨文
`rbac.md` のロール説明または特殊ルールに以下を追加。

```md
- Admin は自分が作成したレポートに限り、申請者として作成・編集・削除・提出が可能である。
- Admin は他者のレポートについては閲覧のみ可能であり、編集・承認・却下はできない。
```

### 修正メリット
この一文があるだけで、権限マトリクスの `○ / ※` と説明文がすべて自然につながる。

## A-3. Approver の閲覧範囲を確定する

### 指摘
資料全体を見ると、Approver の承認待ち一覧は「同テナントの submitted レポート一覧」と読める箇所が多い。  
一方、一部説明では「担当分のみ」と解釈できる表現がある。

### 問題点
ここが未確定だと、バックエンド設計が分岐する。

- **全件承認型**: `submitted` 全件が見える
- **担当者割当型**: `approver_id` や承認ルート設計が必要

MVPで後者をやると設計が重くなる。

### 推奨判断
MVPでは **全件承認型** に固定することを推奨する。

### 明文化推奨文
以下のように統一する。

```md
MVP では Approver は、同一テナント内の submitted 状態のレポートをすべて閲覧・承認・却下できる。
担当者割当や承認経路の個別設定は Phase 3 以降の拡張とする。
```

### 影響ファイル
- `rbac.md`
- `04_business-rules.md`
- `usecases.md`
- `requirements.md`
- 必要なら `02_actor-analysis.md`

### 備考
自己承認禁止ルールはそのまま維持でよい。  
「全件見える」と「自分のものを承認できる」は別論点であり、**自己承認禁止** があるため内部統制上の最低ラインは保てる。

## A-4. ダッシュボードを要件一覧へ昇格する

### 指摘
`usecases.md` には `UC-SYS04: ダッシュボードを確認する` がMVPとして定義されている。  
しかし `requirements.md` のコア機能一覧には、対応する機能IDが見当たらない。

### 問題点
ユースケースにあるのに要件一覧にないと、以下が起きやすい。

- API設計漏れ
- 画面一覧漏れ
- テスト観点漏れ
- 工数見積もり漏れ

### 影響ファイル
- `requirements.md`

### 追加推奨
`### 2.x ダッシュボード` を追加し、少なくとも以下を定義する。

#### 追加例
```md
### 2.x ダッシュボード

**アクター**: 全ロール

#### 機能一覧

| ID | 機能 | 説明 |
|----|------|------|
| DASH-F01 | ダッシュボード取得 | ロールに応じた件数・一覧情報を取得する |

#### ビジネスルール

| ルールID | ルール |
|---------|--------|
| DASH-001 | Member には自分の draft / submitted / rejected 件数を表示 |
| DASH-002 | Approver には承認待ち件数を追加表示 |
| DASH-003 | Accounting には支払待ち件数を追加表示 |
| DASH-004 | Admin にはテナント全体の件数とメンバー数を表示 |
```

### 補足
最初はAPI 1本でもよい。  
ただし、**MVP機能なら要件定義側に存在するべき**。

## 4. 優先度B（着手前に整えると強い）

## B-1. 再申請時のコピー範囲を明記する

### 指摘
再申請について、`workflow.md` では「元レポート参照を保持」「新規レポート作成」「添付はコピーしない」などの具体性がある一方、`usecases.md` では「元レポートの内容をコピー」としか書かれていない。

### 問題点
「内容」がどこまでを指すか曖昧。

- タイトル
- 対象期間
- 明細
- 添付
- 却下理由

実装者ごとに判断が分かれる。

### 推奨追記
`usecases.md` の UC-M09 に以下を追加する。

```md
- 再申請時は、タイトル・対象期間・明細をコピー対象とする。
- 添付ファイルはコピーせず、必要に応じて再アップロードする。
- 新規レポートには元レポートIDを reference として保持する。
```

### 理由
添付までコピーすると、証跡の再利用ポリシーがあいまいになる。  
MVPでは「添付は引き継がない」の方が説明しやすく安全。

## B-2. 通知が Phase 3 であることを業務フロー側にも明記する

### 指摘
業務フロー上では通知が自然に組み込まれている一方、通知機能自体は Phase 3 扱いになっている。

### 問題点
MVPだけ読む人が「通知あり」と誤解する可能性がある。

### 推奨修正
通知が登場する箇所に以下のいずれかを付記する。

#### 例1
```md
※ 通知機能は Phase 3。MVP では一覧画面・ステータス表示で代替する。
```

#### 例2
```md
Approver に通知が届く（Phase 3。MVPでは承認待ち一覧で確認）
```

### 影響ファイル
- `01_business-overview.md`
- `03_business-flow.md`

## B-3. メンバー管理がMVP対象外であることに伴う「初期セットアップ方法」を補足する

### 指摘
MVPでは招待・メンバー管理が未実装である一方、複数ロール前提の業務フローになっている。

### 問題点
デモやレビュー時に、以下の疑問が出る。

- Approver / Accounting はどう用意するのか
- 誰がロールを付与するのか
- MVPで複数ユーザーをどう再現するのか

### 推奨対応
別紙 `demo-setup.md` を作る。

#### 記載例
- 初期Admin作成方法
- seed データで作成するユーザー
- 各ユーザーのロール
- 各ユーザーの所属テナント
- ログイン用テストアカウント一覧

### 効果
これは実装仕様そのものではないが、**ポートフォリオ運用上かなり強い**。

## 5. 優先度C（品質向上）

## C-1. 権限マトリクスの凡例をより読みやすくする

### 指摘
`rbac.md` の `○ / × / ※ / △` はよく整理されているが、初見では `※` と `△` の差がやや分かりにくい。

### 改善案
表の直後に短い説明を追記する。

```md
- ※ = 所有者であることが条件
- △ = 関連業務対象であることが条件（例: 承認対象、支払対象）
```

### 効果
レビューアや面接官に伝わりやすくなる。

---

## C-2. 「404 と 403 の使い分け」を補足しておく

### 指摘
テナント境界越えは `404 Not Found`、同一テナント内の権限不足は `403 Forbidden` という整理は良い。  
ただし、この設計意図は要件上かなり重要。

### 推奨追記
`requirements.md` または `04_business-rules.md` に補足として以下を追加。

```md
- テナント境界を越えるアクセスは、リソースの存在自体を秘匿するため 404 Not Found を返却する。
- 同一テナント内での権限不足は、認可失敗として 403 Forbidden を返却する。
```

### 効果
セキュリティ設計の意図が明確になり、説明力が上がる。

---

## C-3. MVPでの監査証跡の位置づけを明文化する

### 指摘
監査ログ自体は Phase 3 だが、MVPでも「提出後編集不可」「却下後は再申請を新規レポートとして扱う」など、監査性を意識した設計になっている。

### 推奨追記
以下の趣旨をどこかに入れる。

```md
MVP では完全な監査ログは提供しないが、状態遷移の固定化・提出後編集禁止・再申請時の新規レポート化により、改ざんしにくいデータ構造を優先する。
完全な操作履歴の記録は Phase 3 の監査ログ機能で補完する。
```

### 効果
「監査ログ未実装なのに監査性を語ってよいのか」という疑問への答えになる。
