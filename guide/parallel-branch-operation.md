# 並列実装のブランチ運用ガイド

## 目的

Step 7 以降の並列実装で、ファイル競合・レビュー漏れ・マージ順の混乱を防ぐ。
本ガイドは、このプロジェクトにおけるブランチ運用の**確定版ガイド**とする。

## 適用範囲

- Step 7: 基盤構築
- Step 8: テストコード実装
- Step 9: 機能実装
- Step 10: システムテスト・UAT
- Step 5 以降で、複数担当が同時に同一リポジトリを編集する場合

## 基本原則

- 並列タスクは、**隔離された作業環境**で実行する
- `1タスク = 1短命ブランチ` を原則とする
- 統合は `main` への直接コミットではなく、**PR 経由**で行う
- 共有ファイルや共通基盤の変更は、task plan で事前に扱いを決める
- マージ順は依存グラフと task plan に従う
- マージ可否の最終判断はユーザー承認を必須とする

## このプロジェクトでの確定事項

### 1. 作業環境の分離

- 並列タスクは、作業ディレクトリを共有しない
- 分離方法は `git worktree` を第一候補とする
- 使用ツールが別の隔離方式を提供する場合でも、以下を満たせば可とする
  - タスクごとに独立した変更差分を保持できる
  - 他タスクの未コミット変更を巻き込まない
  - PR 単位でレビュー・マージできる

### 2. ブランチ単位

- ブランチはタスク単位で作成する
- Step / Task ID が追跡できる命名を必須とする
- 1本のブランチに複数タスクを混在させない
- 長期間生きる開発ブランチ（例: `develop`）は本プロジェクトでは必須としない

推奨例:

- `step7-7b-foundation`
- `step8-8a-auth-tests`
- `step9-9b-reports`

### 3. main への統合

- `main` への直接 push は禁止
- 変更は PR を作成して統合する
- PR は task plan の担当境界と一致していること
- 1 PR に unrelated change を混在させない

### 4. レビューとの接続

- 実装担当は、PR 作成時に以下を明記する
  - 対応タスク ID
  - 変更対象ファイル
  - 共通部品への影響
  - 後続タスクが取り込むべき変更点
  - 未解決の前提や制約
- 内部 reviewer は、PR 差分またはブランチ差分に対してレビューする
- 品質ゲート判定は、project workflow の `PASS / FIX / ESCALATE / PASS with NOTE` に従う
- Codex レビューの実施タイミングは、対象 Step の workflow / task plan に従う

## task plan に必ず入れる項目

Step 7 以降の task plan では、少なくとも以下を明記すること。

- タスクごとの担当境界
- 共有ファイルの初回作成者と追記ルール
- マージ順
- 依存先取り込み後の再確認条件
- 共通基盤変更を独断で行ってよい範囲 / 禁止範囲
- 競合発生時の一次対応責任者

これらが未定義のまま、並列タスクを開始してはならない。

## 共有ファイルの扱い

次のようなファイルは、並列競合の高リスク対象として task plan で先に扱う。

- OpenAPI 生成物
- 共通 middleware
- 共通エラー処理
- API クライアント基盤
- 共通 fixture / helper
- CI 設定
- ルーティングや共通レイアウトなど複数機能が触る UI 基盤

扱い方針は以下のいずれかを選ぶ。

- 初回作成担当を固定し、他担当は追記のみ可とする
- 別タスクとして先行実装する
- 並列を避けて直列化する
- Lead 判断が入るまで変更禁止とする

## PR 作成からマージまで

1. タスク担当が隔離環境で実装する
2. ローカル検証を行う
3. ブランチを push し PR を作成する
4. reviewer が差分を確認し、品質ゲート判定を行う
5. 指摘があれば同一ブランチで修正する
6. レビュー通過後、ユーザー承認を得てマージする
7. マージ後、ブランチと不要な隔離環境を削除する

## 競合時のルール

- 先に完了した依存元タスクを先にマージする
- 後続タスクは、依存元を取り込んだうえで再確認を行う
- 共有ファイルの競合は、task plan で定義した一次対応責任者が解消する
- 設計判断が必要な競合は、Lead からユーザーへエスカレートする
- 無断で相手ブランチの意図を上書きしない

## Step 7 で確認すべきこと

Step 7 の作業計画レビューでは、少なくとも以下を確認する。

- Step 8 / Step 9 の並列単位に対して、ブランチ粒度が妥当か
- 共有ファイルの調整方針が task plan に落ちているか
- PR ベースでレビュー・統合できる最小成果物単位になっているか
- CI とブランチ保護の前提が矛盾していないか
- Dev Container / ローカル環境の複数インスタンス運用と整合しているか

## このガイドで意図的に固定しないこと

以下は別ガイドまたは各 Step の task plan で決定する。

- CI の具体的なステージ定義
- GitHub Actions の YAML 詳細
- Dev Container のポート番号や DB 分離方式
- PR テンプレート本文
- 使用ツール固有のパラメータ名
