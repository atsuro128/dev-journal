# Step 1: 要件定義（業務理解 → ユースケース化） — 作業分解

## 概要

経費精算の"業務"を理解し、作るものを言語化する。

## 成果物（出力）

| 成果物 | パス |
|--------|------|
| 機能要件・非機能要件 | `deliverables/docs/10_requirements/requirements.md` |
| ユースケース一覧 | `deliverables/docs/10_requirements/usecases.md` |
| 状態遷移 | `deliverables/docs/10_requirements/workflow.md` |
| ロール/権限要件 | `deliverables/docs/10_requirements/rbac.md` |

## 成果物の詳細

### requirements.md

- コア機能：認証・RBAC・経費申請・承認フロー・添付ファイル
- 付随機能：招待フロー・アプリ内通知・監査ログ・CSVエクスポート
- 非機能要件：レスポンスタイム・可用性・レート制限・ファイルサイズ制限

### usecases.md

- 申請者・承認者・Accounting・Adminそれぞれの操作シナリオを網羅
- 招待フロー・通知受信・監査ログ閲覧も含める

### workflow.md

- 申請〜承認の状態遷移（Mermaid推奨）

### rbac.md

- ロール/権限の要件

## 最低限のMVP業務フロー例

- 申請 → 承認 → 差戻し → 再申請 → 承認 → 完了

## 【root-project 整備】

- `prompts/requirement.md`：要件整理に使う指示文テンプレートを整備（ユースケース抽出・状態遷移整理などの型）
- `rules/security-policy.md`：セキュリティ要件の基礎方針（認証方式・レート制限・ファイルサイズ制限の方針）を記載

## 完了条件

- 「誰が」「何を」「どの順で」するか説明できる
- 例外（却下/取消など）を"やる/やらない"で決めている
- 招待フロー・通知・監査ログ・CSVエクスポートの扱いが明記されている

## レビュー観点

成果物内の「品質チェック」は作成者のセルフチェック。以下は reviewer が検証する観点。

### 1. 上流整合性（Step 0 との整合）
- MVP スコープ（`02_scope.md`）に記載された全機能が `requirements.md` の要件として定義されているか
- `02_scope.md` でスコープ外とした機能が `requirements.md` に混入していないか
- Phase 3 機能が Phase 3 として明示されており、MVP と混同されていないか

### 2. 内部整合性（Step 1 成果物間）
- RBAC ルールが `requirements.md`・`rbac.md`・`usecases.md`・`workflow.md`・`preliminary/04_business-rules.md` の間で矛盾していないか
- 各ロールの権限範囲（特に Admin・Approver・Accounting の「自分のレポート」と「他者のレポート」の操作権限）が全文書で同一の解釈になっているか
- 状態遷移の実行者表現が全文書で統一されているか（「Member のみ」vs「所有者（全ロール）」の区別）
- `workflow.md` の遷移条件と `rbac.md` の権限マトリクスが整合しているか（自己承認禁止・自己処理禁止を含む）
- ルールIDの定義元と参照先が一致しているか（存在しないIDへの参照がないか、同一IDが異なる意味で使われていないか）

### 3. 完全性
- 全ロール（Admin, Approver, Member, Accounting）のユースケースが `usecases.md` に定義されているか
- 各ユースケースに正常系・代替系・例外系が定義されているか
- 認証系ユースケース（サインアップ・ログイン・ログアウト・パスワードリセット）が網羅されているか
- ダッシュボードが要件として定義されているか（要件ID付き）
- 非機能要件（パフォーマンス・セキュリティ・可用性・データ）が数値目標付きで定義されているか
- 403/404 の使い分け（テナント境界越え=404、同一テナント内の権限不足=403）が明記されているか

### 4. 意思決定の明確性
- 「判断が必要」「推奨方針」のまま残っている未決事項がないか（全て「決定事項」に確定しているか）
- MVP で「やる/やらない」の判断が、理由付きで確定しているか（提出取消、担当者割当など）

### 5. スコープ準拠
- `requirements.md` の機能一覧と `02_scope.md` の MVP 定義が双方向で整合しているか（一方に存在し他方に存在しない機能がないか）

### 6. 用語準拠
- 全成果物の操作用語が `glossary.md` に従っているか（「差戻し」ではなく「却下」、「申請」ではなく「提出」など）

### 7. 下流作業可能性
- Step 2（ドメイン設計）の作業者が、エンティティの属性・不変条件・状態遷移ルールを一意に読み取れるか
- ユースケース一覧サマリと本文定義のアクター記述が一致しているか
